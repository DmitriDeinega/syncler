The implementation of V3 #14 (Live Channels) is a solid foundation, but there are several critical "wiring" gaps and one major security/privacy issue regarding push scoping that must be addressed before proceeding to V3 #16.

### 1. Per-Concern Verdicts

1.  **WS subprotocol parsing (OK):** The parsing logic correctly handles commas and whitespace. Intermediaries reordering headers will not break it.
2.  **Connect token store in-process (OK):** CPython's GIL makes the `dict.pop` and `_TOKEN_STORE[tok] = entry` operations atomic within the `asyncio` loop since there are no `await` points during the mutation. Acceptable for V0.1.
3.  **Revocation publish failure tolerance (OK):** Fail-open is acceptable for V0.1. Since the hub is in-process, a failure here likely implies a systemic process failure anyway.
4.  **Rate-limit semantics (NIT):** Per-socket is technically simpler, but it allows a device to bypass the 16KB/s "per plugin" cap by opening multiple sockets. **Recommendation:** Add a connection cap (e.g., max 4 sockets per plugin/device) to `plugin_socket`.
5.  **Heartbeat rolls on any frame (OK):** Lenient activity tracking is standard and acceptable.
6.  **Webhook signing fail-open (NIT):** If a signing seed is configured but invalid, the server should ideally refuse to forward rather than sending unsigned requests.
7.  **WS endpoint pairing re-check (OK):** Deferring the polling fallback is fine for V0.1 since the in-process bus is reliable.
8.  **deviceJwtProvider placeholder (OK):** Throwing `no_session` is sufficiently "loud" as it results in a `Result.failure` at the SDK level.
9.  **BridgeLiveChannelHandle.close() state (NIT):** Subsequent `send()` calls after `close()` will still trigger an AIDL round-trip only to receive a `channel_not_open` error from the host. **Recommendation:** Add a local `isClosed` flag to `BridgeLiveChannelHandle`.
10. **WS frame size enforcement (NIT):** Adding a 64KB check on the Android side (in `LiveChannelClient`) would provide faster feedback and save bandwidth.
11. **Replay buffer on ordered lane (OK):** 256 is a reasonable V0.1 cap for best-effort delivery.
12. **Live inbound URL scheme check (FIX):** The `PluginPublishRequest` validator is missing the scheme check for `live_inbound_url`. **Recommendation:** Use the existing `_is_allowed_action_endpoint_scheme` helper at publish time.
13. **Step 8 deferral (OK):** Acceptable for mid-track review, provided e2e tests land before V3 #17.

### 2. Major Findings (Beyond the numbered list)

#### 🚨 CRITICAL: Push Scoping & Privacy Leak
The current `push_live` endpoint and `plugin_socket` subscription use a global topic: `plugin:{plugin_row_id}`.
- **The Issue:** A sender pushing an update for **User A** will have that update delivered to the WebSockets of **User B**, **User C**, etc., if they use the same plugin. 
- **The Impact:** While the E2EE recipient envelopes ensure User B cannot *decrypt* the payload, User B's device still receives the frame. This wastes bandwidth and leaks metadata (User B knows a live update occurred for User A and can see the unencrypted `user_id` and `expires_at` fields in the V2 envelope).
- **FIX:** The topic must be scoped to the user: `user:{user_id}:plugin:{plugin_row_id}`. The `push_live` endpoint must extract `user_id` from the envelope (or require it in the request body) and publish only to that user's topic.

#### ⚠️ FIX: Webhook Forwarding is Not Wired
While `server/app/live/webhook.py` is implemented, it is **not called** from the WebSocket handler.
- **The Issue:** In `server/app/routers/live.py`, `_handle_inbound_frame` simply `ack`s message frames and returns. It never invokes `forward_to_sender`.
- **The Impact:** Device → Sender communication is currently non-functional (silent black hole).
- **FIX:** Update `_handle_inbound_frame` to lookup the plugin's `live_inbound_url` and invoke the forwarder before sending the `ack`.

#### ⚠️ FIX: Rate Limit Consumption
The server-side `_SocketState.consume` is called in `pump_inbound` but the return value (the violation flag) is only checked **after** processing the frame. If a 10MB frame arrives, the server will process it (or try to) before closing.
- **Recommendation:** Move the size check and rate-limit check to the very top of the loop, and ensure `MAX_FRAME_BYTES` is strictly enforced before any JSON parsing.
