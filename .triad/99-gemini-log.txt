**RATING: YELLOW**

The spec is extremely well-thought-out and the cryptographic model is solid. The CAS constraints and downgrade protections are exactly what we need. However, there are a few critical logical gaps regarding how the AAD interacts with the CAS counters, and a contradiction in the text regarding salts. 

Here is my review and answers to the open questions:

### Answers to Open Questions

1. **AAD shape for user-state + pairing-state:** 
   The fields are exactly right. Binding these variables prevents replay, re-binding, and downgrade attacks. **HOWEVER, there is a critical footgun here:** Since the AAD includes `state_version`, and the server *bumps* the `state_version` on write, what integer does the client put in the AAD during encryption? 
   The client MUST predict the future version (i.e., `target_state_version = state_version_observed + 1`) and use *that* in the AAD. The spec must explicitly state: 
   *(a)* The client encrypts using `state_version_observed + 1` in the AAD.
   *(b)* The server increments `state_version` by exactly 1 on success (not an opaque ID or timestamp).
   If the client uses `N` and the server saves as `N+1`, the blob is permanently bricked.

2. **`key_generation_observed` in the rotation request:** 
   **YES.** Add it to the top level of the request. This prevents concurrent rotations from stepping on each other (e.g. rotating 1->2 vs 1->3). 
   *Crucial addition:* ALL other state-mutating endpoints (e.g., standard CAS state push, pairing creation, pairing state updates) MUST now also require `key_generation_observed` in their payloads and 409 if it doesn't match the server. Otherwise, a pre-rotation client could successfully push a state update encrypted under the OLD master key *after* the rotation has committed.

3. **Recent-auth window for `current_password_proof`:** 
   **Force re-type.** The client typically needs the plaintext password at this exact moment anyway to derive the `wrap_key` and unwrap the old master key (unless you are caching the `wrap_key` in memory indefinitely, which is a bad practice). Forcing a re-type ensures a session hijacker cannot escalate to full account takeover or lock the legitimate user out.

4. **What about cards' encrypted_payload?** 
   **Confirmed.** Since pairing keys are established at bootstrap and not rotated by this protocol, cards remain untouched and decryptable. 

5. **Backup / recovery banner.** 
   **YES.** Make it a strict spec-level requirement (MUST). "Rotate key" and "Change password" both result in unrecoverable data loss if the new password is forgotten. The UI must explicitly acknowledge this.

6. **Mixed-client behavior:** 
   Relying on AEAD decryption failures provides terrible UX (the app just looks mysteriously broken or stuck syncing forever). The server should proactively refuse pre-Phase-8 clients if the user's `key_generation > 1`. If you have a client version header, return `426 Upgrade Required`. If not, return a 400 with a specific error code like `account_upgraded_requires_newer_client` so the older app can at least log it or show a fallback message.

### Fixes required in the Spec Text

- **Contradiction in 10.1 (`password_rewrap`):** The text says "Derive a new `master_key_wrap_key` from the new password + the **existing** `auth_salt`." But the wire format in 10.4 correctly includes a `new_auth_salt`. You should generate a NEW salt on password change. Update 10.1 to say "new password + **new** `auth_salt`".
- **10.4 / 10.5 Pairing Race Condition:** You correctly note in 10.4 that "Missing pairings → 400". Ensure that 10.5 explicitly states this check happens *inside* the `SELECT ... FOR UPDATE` lock on the user row. If a new pairing is created concurrently just before the rotation commits, the rotation must abort (409) rather than leaving an orphaned pairing encrypted with the old master key. 
- **10.8 Test vectors:** For the new AAD shape locking, ensure the test vectors explicitly document that the `old_us_aad` uses `state_version: N` and the `new_us_aad` uses `state_version: N+1` (or whatever the target is), to clarify the prediction mechanic discussed in Q1. 

Once the AAD prediction and `key_generation_observed` across all mutating endpoints are locked in, this is an easy GREEN.
