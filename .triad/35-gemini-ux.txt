Here is an independent, opinionated UX direction for Syncler's next iteration.

# The Vision: Syncler as a Managed Micro-App Host

The current UX is a feed of raw E2EE messages. The next iteration must shift from "a secure inbox of dynamic cards" to a **managed host of micro-apps**. We are not building an email client; we are building a zero-trust, multi-device dashboard where senders can push actionable UI to the user's pocket. 

To achieve this, we have to establish a hard boundary between **what the host owns** (hierarchy, navigation, metadata search, global state sync) and **what the plugin owns** (content rendering, detailed interaction, and local storage state). The host must not try to parse content, and the plugin must not be allowed to escape its sandbox to dominate the user's screen space.

### 1. Card Presentation: The Preview-Detail Split
**Stance: Max-height limit with a strictly enforced Host-controlled Detail View.**

Plugins should not own their height unconditionally. Unbounded auto-grow destroys scroll performance and allows malicious/buggy plugins to hijack the screen. Conversely, a fixed 280dp height makes rich plugins unusable.

The solution is a **two-tier rendering model**:
1. **Preview Mode (Inline Card):** The plugin renders in a constrained context. The host enforces a max-height (e.g., `320dp`). If the plugin's content overflows, it clips, but the host overlays a standard "Tap to Expand" affordance over the bottom edge.
2. **Detail Mode (Full Screen/Modal):** When the user taps the card, the host launches a new full-screen `AndroidView(WebView)` for that specific card. The plugin is passed a runtime flag indicating it is now in `detail` mode and has the entire vertical viewport.

This gives plugins the space they need for rich interactions (e.g., charting for a trading bot) while keeping the primary feed scannable. 

### 2. Read/Unread State: The Silent Signal
**Stance: Client-local authority, synced via the M7 encrypted CAS blob.**

Because the server is E2EE content-blind, it cannot reliably infer "read" state based on content semantics. A card transitions to "read" when:
- It is explicitly interacted with (an action button is pressed).
- It is opened in the Detail View.
- It dwells fully visible in the viewport for $> 3$ seconds.

This state is stored locally in SQLite and patched into the M7 encrypted user-state blob. When the user opens their tablet, it pulls the CAS blob, decrypts it, and marks the corresponding `message_id`s as read locally. Visually, unread cards have a subtle host-rendered accent (e.g., a left-border color bar). Plugins are *unaware* of their read state; this is strictly host chrome.

### 3. Organization: The Grouped Inbox
**Stance: Flat list by date for the Inbox, with a dedicated Plugin Dashboard for structured access.**

A pure folder-by-sender approach hides time-sensitive alerts, but a flat list becomes a chaotic feed. We need both:
- **The Inbox (Timeline):** Cards are ordered by `sentAt`. However, consecutive cards from the *same plugin* within a short time window (e.g., 2 hours) are visually collapsed into a "stack" by the host. Only the latest card is fully rendered in Preview Mode; the rest are represented by a host UI footer: "3 more from [Sender]".
- **The Dashboard (Per-Plugin View):** The app structure needs a Bottom Nav. The second tab is "Plugins". This shows a directory of paired plugins. Tapping a plugin opens a host-managed screen showing *only* cards for that plugin, bypassing the Inbox noise.

### 4. Search: The Federated Model
**Stance: Host does metadata search; Plugin does content search (Opt-in).**

The host cannot do full-text search because the payloads are E2EE and only decrypted just-in-time. Relying entirely on a plugin-implemented global search is too complex for UI handoffs. 
- **Global Search:** The host provides a search bar at the top of the Inbox. This searches *only* host metadata: Sender Name, Plugin Name, and Date.
- **Plugin-Level Search:** If the user navigates to a specific plugin's Dashboard, and the plugin manifest declares `Capability.SEARCH`, the host provides a search bar and forwards the query to the plugin via a new bridge hook `__syncler_internal_dispatch('onSearch', query)`. The plugin then renders its own results using its local `platform.storage`.

### 5. Plugin Role Expansion: The Dedicated Surface
**Stance: Plugins get one extra surface—the "App View"—but no chrome control.**

V1 plugins own the card render. The next logical expansion is allowing a plugin to own a full-screen view independent of a specific message payload. If a plugin declares `Capability.APP_VIEW`, the host adds a persistent entry for it in the "Plugins" tab. When tapped, the host loads the plugin's JS bundle with a `null` payload. The plugin uses this space to render settings, global stats, or history, pulling data from its `platform.storage`. We draw a hard line: plugins do *not* get to manipulate host chrome (no changing the TopAppBar, no adding custom FABs).

### 6. Lifecycle: Local Expiry & Archive
**Stance: Auto-expire by default, explicit Archive for permanence.**

The server caps retention at 30 days. The client should mirror this to prevent unbounded SQLite bloat. 
- **Dismiss vs Archive:** Swiping a card in the Inbox "Dismisses" it (moves it out of the view, marks it read, keeps it in DB until 30 days, sends `DISMISS` back to sender). If the user wants to keep a receipt forever, the host provides an "Archive" host-action. Archived cards bypass the 30-day GC.
- **Plugin-Driven Expiry:** A plugin can use `platform.message.dismissBehavior` to explicitly retract a message earlier based on user interaction.

### 7. App Chrome: Standardizing the Navigation
**Stance: Standard Material 3 Bottom Nav.**
- **Inbox:** The timeline feed (with unread badges).
- **Plugins:** The directory of paired senders and their dedicated full-screen views.
- **Settings:** Device management, pair new sender, log out.
No FAB. A "Compose" affordance doesn't make sense globally because the user doesn't initiate messages to arbitrary plugins; they *respond* to E2EE pushes.

### 8. Edge Cases
- **Fresh-pair "What do I do now":** Upon pairing a new sender, the host injects a synthetic, host-generated card: "Paired with [Sender]. Waiting for their first push..." This prevents empty-state confusion.
- **Plugin fetch failed / render threw:** The current `PayloadFallbackView` is good, but it should be framed as a graceful error state on the card: "Plugin failed to load. [View Raw Data] [Retry]".
- **Revoked bundles / Invalid signatures:** If a bundle hash changes maliciously, the host refuses to load it. The card displays a security warning: "This message cannot be verified. The sender's plugin signature is invalid." This is a hard E-stop.

## The Single Most Important Call
**Standardizing the Preview/Detail lifecycle transition.**

If you ship only one change next, remove the fixed `280dp` height and introduce the "Tap to Expand into Detail View" pattern. This single structural change forces you to define the `viewMode` in the plugin contract, solves the clipping problem for rich apps, and prevents simple notifications from taking up too much vertical space. It is the absolute bedrock of a usable micro-app feed.

## Things to Defer
1. **Plugin-Federated Global Search:** Do not attempt to let plugins hook into the *global* Inbox search bar yet. It requires complex timeout handling, ranking merging, and UI rendering handoffs across multiple isolated WebViews simultaneously. 
2. **Cross-Plugin Communication:** Do not let plugins talk to each other. The sandbox must remain completely isolated.
3. **Complex Inbox Folders:** Defer building UI for user-defined folders. Rely on the flat Inbox timeline + the secondary "Plugins" tab for now.

## Diverse Sender Needs
- **The Lottery Sender:** Needs a prominent Preview card showing "You won!" and a button. Doesn't need a Dashboard. Wants the card to auto-dismiss once the prize is claimed (benefits heavily from `dismissBehavior`).
- **The Trading Bot:** Needs a high-frequency feed. Benefits heavily from Inbox stacking (collapsing 5 price alerts into 1). Desperately needs the **App View** capability to render a full-screen portfolio chart using data cached in `platform.storage`.
- **The AI Agent (Human-in-the-loop):** Needs the Detail View. The prompt might be a long chat history context followed by an "Approve/Reject" action. The 280dp constraint makes reading chat history impossible; the Detail View allows a full-screen, scrollable log.

## SDK & Contract Changes Required

1. **Manifest Additions:**
   - `preferredPreviewHeight?: number;` (Hint to host in dp, capped by host).
   - `declaredCapabilities`: Expand the enum to include `APP_VIEW` and `SEARCH`.

2. **Platform Bridge Changes:**
   - **Environment Injection:** The host needs to inject context into the plugin on load so the plugin knows what to render. 
     `window.__syncler_env = { viewMode: 'preview' | 'detail' | 'app_view' };`
   - **Navigation Hook:** A way for a plugin in 'preview' mode to request expansion programmatically (e.g. if the user taps a specific button *inside* the webview instead of the host card background).
     `platform.ui.requestDetailView(): Promise<void>;`
   - **Lifecycle Hooks:**
     `__syncler_internal_dispatch('onSearch', [query: string])`

## Architectural Blockers
- **The WebView Leak:** The comment `// the bridge's CoroutineScope outlives the WebView` in `InboxScreen.kt` indicates a leak architecture. Moving to a Preview/Detail model means destroying and recreating WebViews constantly as the user scrolls the `LazyColumn` and expands cards. The WebView pooling and bridge cleanup must be bulletproof, or the app will rapidly crash with OOM errors.
- **State Loss on Re-render:** When a user taps a card to go from Preview to Detail, the host will likely destroy the Preview WebView and create a new Detail WebView. The plugin will lose all its DOM state. The SDK must heavily document that plugins must persist UI state in `platform.storage` or hydrate quickly purely from the `payloadJson`.
