The V3 #16 design for `card.patch` is a critical optimization for live-card performance. My review focuses on preserving the V2 E2EE contract while ensuring robust ordering on the ephemeral live channel.

### Summary Verdict

| Concern | Verdict | Rationale |
| :--- | :--- | :--- |
| **1. Wire-shape** | **DESIGN** | Use a dedicated `/v1/cards/patch` endpoint. This allows independent rate-limiting (patches are frequent/small) and cleaner schema separation from the main upsert path. |
| **2. Sequence** | **OK** | Option (c) `base_seq` + `patch_seq` is the gold standard for ephemeral patching. It allows the device to detect if it's patching the wrong "generation" of a card. |
| **3. Routing** | **FIX** | Patches **must** be persisted server-side (in a `card_patches` table) to fulfill the "authoritative inbox" promise. A device that misses a live patch but doesn't see a new upsert must be able to catch up via the inbox. |
| **4. State** | **DESIGN** | State lives in `InboxRepository` (DB-backed). Patches should update the persisted `payloadJson`. `TemplateCard` remains stateless, recomposing when its input string changes. |
| **5. Ops** | **OK** | `replace` only is sufficient for V0.1 (scores, typing, presence). |
| **6. Ergonomics** | **FIX** | The encrypted payload should be an **array** of patches (e.g., `[{path, value}, ...]`). This allows atomic updates to multiple fields (e.g., updating both "Home" and "Away" scores) in one envelope. |
| **7. Validation** | **OK** | Field paths **must** remain encrypted. Metadata leakage (which field changed) violates the E2EE spirit. The device renderer's existing `null` safety is the correct gate. |
| **8. Coexistence** | **OK** | Monotone-reject logic is sound. |
| **9. Flicker** | **OK** | Premature optimization. Compose's structural diffing makes full-card re-renders of simple layouts (4-8 nodes) extremely efficient. |
| **10. Retention** | **FIX** | Server-side patches should have a 48h TTL and be **purged immediately** when a new `cards.upsert` arrives for the same `card_id`. Consolidation (folding) is impossible for the server. |
| **11. Scope** | **OK** | 13-step implementation structure is appropriate. |

### Technical Rationale & Refinements

1.  **The "Authoritative Inbox" Gap:** If patches are live-channel-only (ephemeral), a device that is offline during a high-velocity score stream will see an old score until the *next* full upsert. If the sender assumes patches are durable, the inbox becomes stale. By persisting patches in a sidecar table, the server can include them in the `GET /v1/messages/inbox` response (as an optional `patches` field on the card item), allowing the client to sync to the "true" state.

2.  **Privacy Invariant:** The `card_id` and `base_seq` are acceptable as public metadata in the wire frame (as they are already public in the `upsert` path for CAS). However, the `field_path` is sensitive; knowing a plugin is updating `$.secret_field` vs `$.subtitle` is a metadata leak. Keep it inside the HPKE-sealed envelope.

3.  **SDK Ergonomics:** Don't force the sender to know JSONPath syntax if possible. The SDK should provide a typed helper: `client.patch_card(card_id, subtitle="New Value")`. The SDK handles the mapping to `$.subtitle`.

4.  **Sequence Logic:**
    *   If `patch.base_seq < device.card_seq`: Reject (stale generation).
    *   If `patch.base_seq == device.card_seq` AND `patch.patch_seq <= device.last_patch_seq`: Reject (replay/late arrival).
    *   If `patch.base_seq > device.card_seq`: Device is missing an upsert. Buffer the patch and trigger an inbox pull (or just ignore and wait for the pull).

### Go-List for Implementation Phase

1.  **Server:** Implement `/v1/cards/patch` with its own rate limits (`card_patch` bucket).
2.  **Server:** Add `card_patches` table (PK: `card_id`, `patch_seq`). Purge on new upsert.
3.  **Android:** Update `InboxRepository` to handle the `card.patch` event. Add a `applyPatch(payload: String, path: String, value: String): String` JSON utility.
4.  **Android:** Ensure `InboxRepository` updates the local DB so the UI recomposes automatically via the existing `items` Flow.
5.  **SDK:** Add multi-patch support (`patches: List<PatchOp>`).

**Verdict: PROCEED TO IMPLEMENTATION with the DESIGN/FIX items above integrated.**
