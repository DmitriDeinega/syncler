### Server

*   **FCM Decoupling:** The migration to `_active_devices()` in `store_message` is clean. It correctly separates the storage channel (inbox) from the push channel (FCM), allowing non-FCM devices to function via polling as intended.
*   **Inbox Projection:** Including `plugin_identifier` in the inbox response via a batch query is N+1 safe.

### sdk-plugin (TypeScript)
*   **Bundle Format:** Flipping to IIFE is correct. Since the Android host injects the bundle into a standard `<script>` tag, ESM `export` statements would have caused a syntax error.
*   **Render Hook:** The addition of the `render` dispatch hook correctly completes the bridge contract for card rendering.

### Android
*   **Critical Risk: Resource Leak in CardBridge:** `CardBridge` initializes a `CoroutineScope` with a `SupervisorJob`. In `InboxScreen.kt`, `CardBridge` is instantiated within the `AndroidView.factory` for each card in the `LazyColumn`. Since `destroy()` is never called, every card that is scrolled or recomposed will leak its `SupervisorJob` and its `Dispatchers.IO` threads.
    *   *Recommendation:* Use the `onRelease` block in `AndroidView` (or a `DisposableEffect`) to call `bridge.destroy()`.
*   **Critical Security Gap: Bundle Signature Verification:** `InboxRepository.fetchPluginOrNull` downloads and executes the JS bundle without verifying its SHA-256 hash or Ed25519 signature. A compromised CDN or MITM (in non-debug builds) could substitute the bundle and gain JS execution context.
    *   *Validation:* While flagged as "V1 work," this is a high-severity gap for production. The `latest` API already provides the `signature` and `bundle_hash`; the repository should use `PluginSignatureVerifier` to validate these before execution.
*   **Crypto (AAD Reconstruction):** The reassembly of AAD in `InboxRepository.decrypt` is spec-compliant. The explicit handling of `null` vs `""` for `min_plugin_version` ensures the byte-shape matches what the Python SDK signs.
*   **Samsung Keystore Fix:** The defensive cleanup in `Signing.kt` for stale/substituted P-256 aliases is a robust solution to the OEM Ed25519 provider quirk.
*   **HTTP Relaxation:** The `BuildConfig.DEBUG` gates in `NetworkBridge`, `CardBridge`, and `PluginLoader` are correctly applied and will not leak into release builds.
