Verdict: **YELLOW**

1. Same-UID security wording is too strong. The design should not say `:plugin` “blocks memory-corruption / type-confusion exploits from leaving the sandbox” or that a UID jail “wouldn’t help.” Same UID gives crash/OOM/scheduler isolation, but not app-data isolation after native compromise. If staying same UID, state that explicitly as an accepted V1.5 trade-off. Different UID is still viable with `ParcelFileDescriptor`, `ContentProvider`, or copied staged input, so “breaks file sharing” is not a hard blocker.

2. AIDL identity is under-specified. `IPluginHostCallback.bridgeCall(...)`, `onWebViewError(...)`, `onPluginReady()`, and `onPluginCrashed(...)` do not carry `sandboxToken` or `pluginId`. This is only safe if Phase 10a mandates one callback Binder object per token and no callback reuse. Cleaner contract: include `sandboxToken` in every sandbox -> host callback to prevent token confusion in the single shared `:plugin` process.

3. `oneway` ordering needs tightening. The snippet does not mark methods `oneway`, while the prose says hook/result delivery is `oneway`. Binder preserves ordering for transactions from one client thread to one remote Binder object, but concurrent callers can interleave. If hook order matters per plugin, design should require a per-token serial queue in `PluginSandboxService` after Binder receipt.

4. `PluginLoadRequest.manifestJson` should not be the authority for host-side grants. Parser divergence is real. Prefer passing a parsed/normalized Parcelable containing at least `pluginId`, `declaredCapabilities`, `declaredEndpoints`, `dismissBehavior`, `bundleHashHex`, and renderer/platform fields. Raw manifest can be included for diagnostics/runtime metadata, but the host’s normalized view should be the capability source.

5. Bundle storage is inconsistent with current code/design. Existing `AndroidEncryptedBundleStore` writes encrypted bundles under `noBackupFilesDir`, while the doc says stage readable bundle path under `getCacheDir()`. Phase 10a should decide: plaintext staged file, encrypted file plus decrypt-in-sandbox, or host passes a `ParcelFileDescriptor`/bytes. I would avoid `cacheDir` for canonical staged bundles; use `noBackupFilesDir` or an FD and handle missing-file reload explicitly.

6. Lifecycle has a callback registration race. `loadPlugin()` returns `sandboxToken`, then `registerHostCallback()` happens afterward, but `onPluginReady`/`onWebViewError` can occur during async WebView load. The design needs a rule: either register callback before load, pass callback in `loadPlugin`, or guarantee no lifecycle callback fires until after registration.

7. Migration ordering should move e2e earlier. Current step 5 strips in-process WebView before step 6 validates the replacement. Add a temporary dual path/test harness before removal, or make “strip” last after e2e proves bind/load/dispatch/bridge/unload/death recovery.

Missing Phase 10a items:

- Define loaded-plugin state machine: `loading`, `ready`, `errored`, `unloaded`, `process_dead`, and what each AIDL method returns/throws in each state.
- Define max IPC payload sizes and large-result behavior. Binder has transaction limits; `payloadJson`, `argsJson`, and `resultJson` need size caps or file/stream fallback.
- Define callback/result cleanup on unload/process death so pending bridge promises do not hang until timeout.
- Define whether `PluginSandboxService` unbinds/stops when no plugins are loaded. Singleton binding is acceptable, but document the deliberate resident-process behavior and whether `unload all` calls `unbindService`.

Existing callback-into-WebView paths look compatible in principle. Current `NotificationBridge` does not yet implement click callbacks, and `MessageBridge` is host-only response/dismiss plumbing, so `dispatchHook` can cover future host -> plugin events. The contract just needs ordering, token identity, and lifecycle semantics locked before 10b.
