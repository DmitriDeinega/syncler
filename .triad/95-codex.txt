**Per-Area Verdicts**

Schema: **GREEN**  
Composite PK on `(sender_id, nonce)` is the right durable replay primitive. Add the ORM model too, because tests use `Base.metadata.create_all()`, and include the 12-byte check in both Alembic and model metadata if possible.

Atomic record-or-reject: **YELLOW**  
`INSERT ... ON CONFLICT DO NOTHING` is correct, but be careful with transaction boundaries. In this codebase `store_message()` commits internally, so a nonce insert done before `store_message()` will be committed together with the message if `store_message()` succeeds. If `store_message()` raises before commit, make sure the uncommitted nonce insert is rolled back before the session is reused, especially in tests that reuse `db_session`.

TTL cleanup: **YELLOW**  
On-write cleanup is acceptable for V1.5, but I would implement it conservatively: best-effort, after successful insert, and not on every request. Idle deployments retaining rows is not a security bug. Existing `server/app/jobs/retention.py` already exists, so also adding nonce pruning there would be low-friction and avoids making cleanup purely traffic-dependent.

Cards scope: **YELLOW**  
Scope card upsert in if the table is being added anyway. But the bigger card replay gap is actually `POST /cards/delete`: its signed envelope has no nonce, no expiry, and no sequence, so a captured delete can be replayed indefinitely to delete a current card with the same `(sender_id, user_id, card_key)`. Upsert replay is partly sequence-protected; delete is not.

NonceRegistry removal: **GREEN**  
Delete the in-memory registry. Keeping it as L1 cache adds rollback/cache-consistency footguns for little value.

**Open Questions**

1. TTL cleanup: choose **A plus existing retention job integration** if allowed. Pure A is fine for V1.5, but add nonce cleanup to `jobs/retention.py` too since that surface already exists.

2. Cards.py: **scope in for upsert**, but do not consider cards “handled” unless delete replay is either fixed now or explicitly tracked as a separate security item.

3. TTL value: use `MAX_RETENTION` / max accepted envelope lifetime, not a non-existent `MESSAGE_EXPIRY_SECONDS`. Current messages use `MAX_RETENTION = 30 days`; live cards use `MAX_TTL = 48h`. If one shared `(sender_id, nonce)` namespace covers all envelope types, prune after the maximum accepted lifetime, preferably with a small cushion.

4. L1 cache: **no**.

5. Multi-worker semantics: `ON CONFLICT DO NOTHING` on a real unique/PK index is the right Postgres primitive. Use `sqlalchemy.dialects.postgresql.insert`, and `rowcount > 0` should be fine with asyncpg. Main gotcha is not isolation level; it is transaction lifecycle and making sure a failed downstream operation does not accidentally burn or strand a nonce in a reused session.

**Anything Missing**

Add migration downgrade, model class, and migration parity test coverage. `test_migration.py` compares Alembic-created tables with `Base.metadata.tables`, so forgetting the ORM model will fail or, worse, leave test DBs without the table.

Consider a FK from `nonce_replay.sender_id` to `senders.id`, likely `ondelete="CASCADE"`, unless sender rows are intentionally never deleted.

Test true concurrency with separate sessions/connections. Concurrent inserts through the same `AsyncSession` will not prove cross-worker safety.

Security footgun: if nonce recording is moved before all semantic validation, an attacker with a valid signature over an otherwise invalid envelope may burn that nonce. That is not necessarily exploitable with random one-shot nonces, but it changes current behavior. Prefer rollback-on-failure or put record+store under one coherent transaction boundary.
