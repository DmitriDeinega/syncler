=== REACTION ===
The pivot to a plugin-first, zero-trust platform correctly simplifies the core product's scope, but it transfers that complexity to the marketplace and plugin SDK, making them the most critical components. The estimated 88-140 hour scope significantly exceeds the 80-hour target, primarily due to the marketplace's unforeseen scale. Deferring the full marketplace UI in V1 in favor of a simpler URL-based installation seems like a necessary tradeoff to meet the initial budget.

=== BATCH 3 QUESTIONS ===
1.  **Marketplace Scope:** For V1, should we build a full marketplace (submission portal, review UI, public browsing), or a "developer-only" V1 where users install approved plugins via a direct URL, deferring the full marketplace to V1.5? The latter could save 30-40 hours.
2.  **Android Technology Stack:** For the Android host app, should we use a simple native WebView, or a framework like Capacitor/React Native? A framework would provide more robust access to device APIs if plugins need more than just network calls in the future.
3.  **Plugin Sandbox Capabilities:** What specific permissions will a V1 plugin have? Can it only make network requests to its single declared backend, or should it also have access to persistent local storage, the camera, or other device APIs?
4.  **Pairing Ceremony:** What is the exact, step-by-step user experience for pairing a new plugin? For instance: user scans a QR on a website, which contains a URL with a one-time token that the app uses to perform the initial E2E key exchange with the sender's server.
5.  **Plugin Updates:** How should a user receive a plugin update (e.g., `LotteryPlugin` v1.1)? Should updates be installed automatically in the background, or must the user manually approve each update through a prompt?
6.  **Server-Side Scope:** Is a Python library sufficient for the V1 Server SDK, or do we anticipate senders needing a JavaScript/Node.js version immediately? Likewise, should the server backend be a self-hostable package or built for a specific cloud provider like AWS?
7.  **Notification Content:** Is a static notification message (e.g., "You have a new item from [Plugin Name]") sufficient for V1? Or is it a firm requirement that the plugin's code must dynamically generate the notification text, which requires enabling background code execution?
