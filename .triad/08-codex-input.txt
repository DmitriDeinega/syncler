## 0. Executive summary

V1 is a multi-device, end-to-end encrypted, plugin-host platform. External programs push structured messages to a user's Android phone(s); each sender ships its own JavaScript plugin that defines its own data, UI, and actions; the platform is content-blind. Pairing brokered through the server. User account = email + password under the Bitwarden zero-knowledge model.

**V1 stack:** Python (FastAPI) server + Kotlin (Jetpack Compose) Android app + JS plugin SDK. Supabase + Fly.io free tier for V1, AWS RDS+Lambda transition pre-ship.

**Estimated build:** ~140 hours of platform work. Overshoots the 80h fantasy by 60h. Cut order in ֲ§8 brings it to ~100h if forced.

**Hour-80 demo (per user Q4):** trading bot + lottery plugin both functioning end-to-end, lottery plugin written from a 6-page spec doc handed to a separate Claude Code instance (the dogfood DX test). Multi-device sync demoed across two Android phones.

## 1. Architecture

### Actors
- **User** ג€” identified by UUID, owns 1+ devices.
- **Device** ג€” Android phone running the Syncler app; carries decrypted master key in memory after login.
- **Sender** ג€” any external program (trading bot, lottery app); holds a sender keypair.
- **Plugin** ג€” JS module distributed by the sender, sandboxed in WebView, runs on each user device.
- **Server** ג€” FastAPI + Postgres; stores opaque ciphertexts and metadata; FCM push fan-out.

### Trust model
- Zero-knowledge for content. Server never sees password, master key, or message bodies.
- Server sees metadata: account exists, plugin X installed by user U, sender S paired with user U, message exists with envelope `(sender, recipient, timestamp, min_plugin_version)`.
- Each plugin install requires explicit user permission grant **per device**.

### Data flow (sender ג†’ user device)
1. Sender's backend calls `client.send_to(user_uuid, encrypted_payload)`.
2. Server records envelope; fans out FCM to all of user's devices.
3. Each device wakes; routes to correct plugin's WebView; plugin's `onMessage()` runs.
4. Plugin decrypts payload (per-sender pairing key derived from master key + sender_id).
5. Plugin calls `platform.showNotification({title, body})`.
6. User taps notification ג†’ opens card view ג†’ plugin's `render(data)` paints UI.
7. User taps action ג†’ plugin POSTs to declared endpoint at sender's API.

## 2. Key hierarchy


**Why the indirection:** rotating password (user changes it) re-wraps the master key with new wrap key. Master key itself stays. All encrypted data remains decryptable. Direct passwordג†’master_key would force re-encryption of everything on every password change.

## 5. Plugin SDK (JavaScript) ג€” example

```javascript
import { BasePlugin, DismissBehavior } from '@syncler/plugin-sdk';

export class LotteryPlugin extends BasePlugin {
  static manifest = {
    id: 'com.lottery.app',
    name: 'Lottery',
    version: '1.0.0',
    declaredCapabilities: ['network'],
    declaredEndpoints: ['https://lottery.app/api/*'],
    dismissBehavior: DismissBehavior.DISMISS_ALL,
  };

  async onMessage(payload) {
    const { numbers, draw_date } = payload;
    return platform.showNotification({
      title: 'Lottery numbers ready',
      body: `${numbers.length} sets for ${draw_date}`,
    });
  }

  render(data) { /* HTML/CSS/JS inside isolated WebView */ }

  async onAction(actionName, payload) {
    if (actionName === 'played') {
      await platform.network.fetch('https://lottery.app/api/played', {
        method: 'POST',
        body: JSON.stringify({ card_id: payload.card_id, numbers: payload.numbers }),
      });
    }
  }
}
```

## 9. Risks (ordered by severity)

1. **FCM data messages + Foreground Service for encrypted dynamic notifications.** Android battery-optimization restrictions on newer Android versions (12+) may delay or block background execution. Mitigation: extensive testing on Pixel + Samsung devices early; fallback to static-text notifications if blocked.
2. **WebView per-plugin isolation.** Android WebView is shared; isolating storage, cookies, and JS contexts per plugin requires careful handling. Risk: side-channel attacks if isolation incomplete. Mitigation: separate WebView instance per plugin, distinct user-agent and storage paths, disable JS-to-host bypass.
3. **Key hierarchy correctness.** Crypto bugs are silent. Mitigation: use Tink (Google) on Android and `cryptography` (Python); write tests against Wycheproof test vectors; no custom crypto.
4. **Multi-device key sync race.** New device login before previous device finished writing user state. Mitigation: explicit pull-on-login with version vector on encrypted state blob.
5. **Spec doc DX test.** 6-page cap may be too tight or too loose. Mitigation: write the doc, integration-test with the lottery's Claude Code, iterate on the doc until integration succeeds. The doc length is itself the unit of measurement.
6. **Plugin update propagation.** Q23 said prompt-to-update per device. If user updates Phone 1 but not Phone 2, new-format messages on Phone 2 hit Gap 3 placeholder. Accepted UX cost.

=== ATTACK SUMMARY ===
The plan's architecture and cryptographic foundations are sound, correctly identifying the core patterns for a zero-knowledge system. However, the plan is critically weak in its assessment of the effort and complexity required for Android implementation, specifically for plugin sandboxing (M4) and reliable background processing (M5). Its proposed hour estimates are dangerously optimistic, and the budget cut-order sacrifices core product functionality instead of peripheral features.

=== SPECIFIC ATTACKS ===
1. **M5 Hour Estimate (18h) is Grossly Underestimated.**
   evidence: M5 requires building a `ForegroundService` triggered by FCM messages. Modern Android (12+) heavily restricts background execution. While high-priority FCMs are a documented exception, they are rate-limited and subject to aggressive, inconsistent battery management by OEMs (Samsung, OnePlus, etc.). Debugging these OEM-specific issues is a known, massive time sink. 18 hours is insufficient for building and hardening this for even two major device manufacturers.
   severity: high
   proposal: Re-estimate M5 to 30-40 hours. Add a specific sub-task for "Compatibility testing and hardening on non-Pixel devices."

2. **M4 Hour Estimate (28h) is Unrealistic.**
   evidence: M4 bundles the JS SDK definition, the native JS bridge implementation, documentation, and the entire plugin sandboxing system. As noted in the plan, robust sandboxing is a high-risk task. A realistic estimate for designing, building, and testing a secure `WebView` host environment alone exceeds 28 hours.
   severity: high
   proposal: Re-estimate M4 to at least 45 hours. Split M4 into two distinct milestones: `M4a (JS SDK & Bridge)` and `M4b (Secure WebView Host Implementation)`.

3. **WebView Isolation Strategy is Insufficient and Insecure.**
   evidence: The plan's mitigation is a "separate WebView instance per plugin." By default, all `WebView` instances in an Android app run in the same process and can share the same underlying data directories. A compromised rendering engine in one plugin could lead to data leakage or host-app compromise. True isolation requires using separate OS processes for `WebView` instances via the `android:process` manifest attribute, which the plan omits.
   severity: critical
   proposal: The plan must be updated to mandate that each plugin's `WebView` runs in a dedicated, sandboxed process. Acknowledge this significantly increases the complexity and cost of M4.

4. **AES-GCM Nonce Management is Critically Absent.**
   evidence: The plan specifies AES-GCM but says nothing about how nonces (IVs) are generated or managed. Reusing a nonce with the same key in AES-GCM completely destroys its confidentiality guarantees. This is a catastrophic cryptographic implementation error waiting to happen.
   severity: critical
   proposal: Mandate a nonce strategy. The sender must generate a unique 96-bit random nonce for every message and prepend it to the ciphertext. The client must expect this format.

5. **Cut-Order Prioritizes Hours Over Value.**
   evidence: Cut #4 proposes saving 8 hours by changing dynamic, plugin-rendered notifications to "static-text notifications." This completely guts the core value proposition of the plugin model, which is to let senders control the content and presentation. This cut turns the platform from a rich content host into a generic, low-value push forwarder, invalidating the complexity of the entire plugin system.
   severity: high
   proposal: Remove Cut #4 from the list. If cuts are needed, they should target non-essential `platform` capabilities in the JS bridge (e.g., defer `platform.camera` or `platform.location` to V1.5) or other peripheral features, not the primary notification entry point.

6. **Plugin Signature Verification is Hand-Waved.**
   evidence: The plan calls for a "signed single .js bundle" but omits the verification mechanism. Android's `WebView` cannot natively verify a custom signature on a JS file. The native app must download the bundle, perform cryptographic verification against the sender's public key, and only then load it into the `WebView` from memory or a secure local file. This critical security step is unstated and uncosted.
   severity: high
   proposal: Add an explicit "Plugin Signature Verification" step to the data flow and a corresponding task to M4, increasing its hour estimate.

7. **Argon2id Parameters are Dangerously Undefined.**
   evidence: The plan specifies `params=high-cost` for Argon2id. This is meaningless without specific numbers. The security of all user data at rest depends on concrete, well-chosen parameters for memory cost (m), iterations (t), and parallelism (p). Leaving this undefined is negligent.
   severity: high
   proposal: Specify the exact Argon2id parameters to be used. Start with current OWASP recommendations (e.g., Argon2id with m=19 MiB, t=2, p=1) and record them in the plan.

8. **Supabase/Fly.io Choice Creates Uncosted Migration Risk.**
   evidence: The plan justifies using Supabase/Fly.io for V1 to "reduce plumbing," with a plan to migrate to AWS pre-ship. This migration (of data, user accounts, storage, etc.) is a non-trivial engineering task that is not estimated in the 140-hour budget. The time spent learning two cloud ecosystems and executing the migration could be avoided by starting with the AWS free tier from day one, as the user originally suggested.
   severity: medium
   proposal: Choose one path. Either commit to Supabase/Fly.io and justify it as the long-term solution, or start with AWS services to eliminate the future migration cost and risk.

9. **Key Derivation Lacks Domain Separation.**
   evidence: `HKDF(master_key, sender_id)` is used to create pairing keys. This is functional but not robust. Best practice dictates using the "info" parameter in HKDF to create domain separation, preventing the same key material from being used for different purposes accidentally in the future.
   severity: medium
   proposal: Change the derivation to `HKDF(master_key, info="syncler-v1-pairing-key:" + sender_id)`. This provides context and prevents future derivation collisions.

10. **The "Dogfood DX Test" is Theatrical.**
    evidence: The test's success or failure is attributed to "spec quality," making it unfalsifiable. A test that cannot fail cleanly is not a test; it's a demonstration. It doesn't produce a load-bearing signal about the quality of the SDK, only about the combination of the spec, the SDK, and the agent building from it.
    severity: low
    proposal: Reframe the test's goal from a binary pass/fail to *generating a list of specific developer experience issues*. The deliverable is not a working plugin, but a report of what was confusing, difficult, or poorly documented in the SDK.

=== MISSED ENTIRELY ===
- **Plugin/Sender Revocation:** The plan has no mechanism to handle a compromised sender key or a malicious plugin. There needs to be a way for a user to revoke a pairing or for the server to block a known-bad plugin.
- **State Sync Conflict Resolution:** The plan mentions "version vector" for state sync but provides no detail on conflict resolution. If a user installs plugin A on one device and plugin B on another while both are offline, what is the state when they both sync? Last-Write-Wins is lossy; a proper merge strategy is needed.
- **Data Retention and Pruning:** The plan does not define how long messages and delivery receipts are stored. This has privacy and cost implications that need to be specified.
- **API Rate Limiting:** The server API is vulnerable to various abuse patterns (login brute-forcing, message spamming). The plan should include rate limiting as a basic security measure.
- **Plugin Update Flow:** The plan uses `min_plugin_version` to block old plugins but doesn't define the user experience for prompting and managing updates. Is it blocking? Can it be deferred?

=== STAYS IN PLAN ===
- **Core Key Hierarchy:** The `password -> wrap_key -> master_key` indirection is the correct model for allowing password changes without re-encrypting the user's entire dataset.
- **FastAPI Server Choice:** For a focused, API-driven backend, FastAPI is an excellent choice that prioritizes performance and developer velocity (via auto-docs).
- **Python Server SDK:** Aligning the Server SDK with the backend language is logical and caters to a large developer audience.
- **The "Gap 2" Punt to Plugins:** Forcing plugin authors to handle action idempotency is a reasonable simplification for V1. The platform's core responsibility is message delivery, not action semantics.
- **Per-Device Enrollment:** Despite potential UX friction, requiring explicit enrollment for each device is the correct security posture and a cornerstone of the trust model.
