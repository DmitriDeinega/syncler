 640: ### Android user flow
 641: 
 642: When the device fetches the preview and sees all four bootstrap fields, the pairing screen uses the automated path:
 643: 
 644: 1. User scans the QR.
 645: 2. App fetches preview; verifies `bootstrap_key_signature` against the sender's Ed25519 pub key.
 646: 3. User confirms the fingerprint (unchanged).
 647: 4. App calls `/complete` ג€” Syncler-side pairing is finalized.
 648: 5. App builds the bootstrap envelope and POSTs to `sender_broker_url`.
 649: 6. Broker decrypts, writes to its storage, returns 201.
 650: 7. `Client.wait_for_pairing` in your sender's send loop picks up the new pairing tuple.
 651: 
 652: If step 5 fails (broker down, network glitch), the app **shows a fallback banner** with the same `user_id` + `pairing_key_hex` block as the V1 manual flow. The device-side pairing is already real (step 4 finalized it); only the sender catch-up needs the manual paste.
 653: 
 654: ### Failure modes
 655: 
 656: | What | App behavior |
 657: |---|---|
 658: | `bootstrap_key_signature` doesn't verify | **Hard refusal.** Pairing NOT finalized. Treated as substitution-attack indicator ג€” user does not get a fallback. |
 659: | Preview has only some bootstrap fields | **Hard refusal**, same reason. Sender MUST register all four atomically. |
 660: | Broker returns 401 (decrypt failed) | Fallback banner (manual paste). |
 661: | Broker returns 409 (replay with different values) | Fallback banner ג€” the pairing key from your sender's `wait_for_pairing` will not match this device's, so the user must paste the device's values. |
 662: | Broker returns 5xx or times out | Retried up to 3 attempts (250ms / 750ms backoff). If still failing, fallback banner. |
 663: | Sender's `wait_for_pairing` times out | Sender's responsibility ג€” typically loop again with a fresh QR. The device-side pairing is still real; user can paste manually. |
 664: 
 665: ### Security boundaries
 666: 
 667: Three guards make this safe:
 668: 
 669: 1. **Bootstrap key signature.** The sender's X25519 bootstrap public key is signed by its long-term Ed25519 signing key (which the user confirms via fingerprint). The Syncler server cannot substitute its own X25519 key ג€” the signature would fail to verify against the Ed25519 pub key the user just confirmed.
 670: 
 671: 2. **AAD-binding on `sender_broker_url`.** Even if the Syncler server could change which URL the device POSTs to, the broker reconstructs AAD using its OWN configured `sender_broker_url`. An envelope crafted for a different URL fails AEAD tag verification.
 672: 
 673: 3. **CAS replay guard at the broker.** A captured envelope replayed to the same broker with the same plaintext is idempotent (200). Replayed with different values gets a 409.
 674: 
 675: ### Production hardening
 676: 
 677: - Use Redis or Postgres for `BrokerStorage`. The shipped `InMemoryBrokerStorage` is single-process only ג€” fine for the trading-bot example where the Client and broker share one Python process, but **insufficient for multi-process deployments**. `Client.create_pairing_qr(sender_broker_url=...)` calls `storage.reserve(pairing_id)`; if the broker app runs in a different uvicorn worker, that call's effect won't be visible to the broker's `is_reserved()` check and every pairing will 404. Use a shared backing store that implements the `BrokerStorage` Protocol atomically (one `reserve` call visible to all workers).
 678: - The `rate_limiter` hook on `make_app(...)` is **mandatory** in production. The pending-pairing registry blocks envelopes for un-issued `pairing_id`s, but once an attacker knows a reserved ID they can still spam decrypt attempts on it ג€” the rate limiter is your CPU defense. Implement per-IP and per-`pairing_id` limits.
 679: - Terminate TLS at your reverse proxy; the broker is HTTP-only inside the trust boundary.
 680: - See `docs/crypto-spec.md ֲ§9.3` for the single-fixed-`sender_broker_url` design (per-pairing URLs are a V2 add).
