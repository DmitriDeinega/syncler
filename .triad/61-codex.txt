The tree is not just failing because of `mutedSenders`. That is the first visible compile failure, but static review found additional Android compile breaks and a server/client API contract mismatch.
**Reviewer A: Server**
Green:

- Backwards compatibility is mostly handled by defaulting `renderer` to `"script"` and conditionally omitting template fields from script publish envelopes.
- Phase 2 dismiss filtering carry-over is implemented in [messages.py](D:/Projects/syncler/server/app/services/messages.py:195).
Yellow:

Red:

- Phase 3b delete endpoint diverges from plan and is ambiguous across users.

Prescription:

3. Keep 0006, renderer/template model/schema/router/service flow, SDK publishing support, and dismiss filtering.

**Reviewer C: Storage / Settings**
- `MuteStore` effective mute formula matches the Phase 3c plan: synced mutes plus local mutes minus local unmutes.
- Inbox filtering already consumes `mutedSenderIds`.
- Build root cause: [MuteStore.kt](D:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/MuteStore.kt:57) references `EncryptedUserState.mutedSenders`, but [EncryptedUserState.kt](D:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/EncryptedUserState.kt:13) has no field, JSON serialization, parser, or merger support.
- The plan’s `Set<String>` plus union semantics cannot reliably support “unmute everywhere” under stale/offline merge. Union resurrects removed values.
1. Minimum compile unblock: add `mutedSenders: Set<String> = emptySet()` to `EncryptedUserState`, serialize/parse `muted_senders`, and include it in `StateMerger`.
2. If Phase 3c lands after V4 as a separate commit, bump to V5. If it lands in the same schema wave as V4, keeping V4 is defensible per the plan.
3. Decide semantics before real Phase 3c landing: either accept monotonic union mutes, or use LWW/tombstone metadata for reliable unmute.
4. Add storage tests before UI polish.
