# Consultation 97 — Phase 7 fix-up Review

I have audited the fix-ups against the Codex 96 feedback. The implementation is robust and follows the requested architectural patterns.

### 1. Codex 96 RED: Non-unique public keys in tests
**Status: GREEN**
`test_nonce_replay.py` now uses a module-level `_sender_pubkey_counter` in `_make_sender()` to generate unique `public_key` values for each sender created within a test run. This correctly avoids `uq_senders_public_key` violations when `test_isolates_senders` (or any multi-sender test) is executed.

### 2. Codex 96 YELLOW: Missing CheckConstraint in ORM
**Status: GREEN**
`NonceReplay.__table_args__` in `server/app/models.py` now includes the `CheckConstraint` for `octet_length(nonce) = 12`. `CheckConstraint` was also added to the top-level `sqlalchemy` imports. This ensures metadata parity between Alembic migrations and `Base.metadata.create_all()`.

### 3. Codex 96 YELLOW: On-write cleanup transaction safety
**Status: GREEN**
In `server/app/services/nonce_replay.py`, the opportunistic `_cleanup_expired` call is now wrapped in `async with db.begin_nested()`. This creates a `SAVEPOINT`, ensuring that a statement-level failure during the `DELETE` (which would otherwise abort the transaction in Postgres) is rolled back locally, leaving the outer transaction intact for the primary `store_message` / `upsert_live_card` operation.

### 4. Codex 96 YELLOW: Advisory-lock contention test
**Status: GREEN**
A new test case `test_cleanup_advisory_lock_contention_across_sessions` was added to `test_nonce_replay.py`. It successfully simulates cross-session contention by manually holding the lock in one session and asserting that `cleanup_expired_with_lock` returns `-1` in another, providing empirical verification of the locking logic.

### 5. Unused Imports
**Status: GREEN**
`NonceReplay` and `NONCE_REPLAY_TTL` have been removed from `server/app/jobs/retention.py`, leaving only the necessary `cleanup_expired_with_lock` import.
