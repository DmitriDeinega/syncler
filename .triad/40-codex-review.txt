**Findings**
- **High - Bundle-by-hash does not yet fix history drift.** Android still resolves every message through `/latest`: [InboxRepository.kt](D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/InboxRepository.kt:92), [InboxRepository.kt](D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/InboxRepository.kt:133), [SynclerApi.kt](D:/Projects/syncler/android/core/network/src/main/kotlin/app/syncler/core/network/SynclerApi.kt:64). After restart, or if the device first sees msg1 after v2 is published, msg1 gets v2’s hash and bundle. The inbox item already carries the historical `plugin_id` row: [schemas.py](D:/Projects/syncler/server/app/schemas.py:174), but the client ignores it for bundle resolution. This should hold Phase 5 if the V1 claim is “render history survives plugin updates.”

- **High - Revocation classification is recorded but not discoverable by devices.** `/latest` filters revoked rows out: [services/plugins.py](D:/Projects/syncler/server/app/services/plugins.py:118), then returns 404 without reason: [routers/plugins.py](D:/Projects/syncler/server/app/routers/plugins.py:130). `PluginLatestResponse` has no revoked/reason fields: [schemas.py](D:/Projects/syncler/server/app/schemas.py:355). So the new enum cannot drive silent/security/unavailable UX yet. This also should hold Phase 5 if “classified revocation UX” is part of Phase 4, not just DB storage.

- **Medium - Re-revoke allows downgrades, not just promotion.** The service comment says upgrade `superseded` to `compromised`, but the code replaces any existing reason with any new one: [services/plugins.py](D:/Projects/syncler/server/app/services/plugins.py:136), [services/plugins.py](D:/Projects/syncler/server/app/services/plugins.py:148). A later signed `superseded` can erase `compromised`. I’d make the reason immutable, or monotonic with `compromised` terminal.

- **Low - The “records reason” test does not assert reason persistence.** `test_revoke_records_reason_when_provided` imports `Plugin`/`select` but never queries the row or checks `revocation_reason`: [test_plugins.py](D:/Projects/syncler/server/tests/test_plugins.py:205), [test_plugins.py](D:/Projects/syncler/server/tests/test_plugins.py:246). It would pass even if `revoke_plugin_row(..., reason=...)` ignored the reason.

**Answers**
1. V1 in-memory-only is not a safe correctness cut for the stated goal. Disk persistence is one solution, but the smaller correctness requirement is exact historical-row resolution. If old bundle URLs can expire, then disk persistence or a server-side URL refresh by plugin row becomes required.

2. Don’t defer the field if Phase 4 is meant to ship history correctness, but simply wiring detail to `bundleByHash` is not enough. Current `InboxItem` already stores `bundleJs`; the real bug is assigning the wrong hash/bundle during refresh by asking `/latest`.

3. Stale CDN bytes are caught if `/latest` returns v2 hash and the CDN serves v1 bytes. The problem is not hash verification; it is choosing `/latest` as the expected hash for an old message.

4. No shape collision found. `reason: null` normalizes to omitted on the server, so a signer that signs a literal `"reason":null` 3-field body will fail verification, not collide.

5. I would not allow arbitrary reason mutation. Use immutable reason or monotonic promotion.

6. Surface it now. A small contract change is warranted if the enum is meant to exist for device UX.

7. Null-as-legacy is fine if all clients centralize “null means unspecified/compromised-class.” Server coercion is cleaner, but not the main blocker.

8. Add tests for DB persistence, signature binding of `reason` cannot be stripped/changed, null/omitted legacy behavior, downgrade prevention if adopted, and the device-facing revoked reason contract once exposed.

Net: I would hold Phase 5 until the exact-bundle render path and revocation reason propagation are fixed or the Phase 4 claims are explicitly narrowed.
