### Review Verdicts

1.  **Matching-by-kind in ActivityCoordinator — OK**
    The process-global concurrency gate (`activeActivityResultCallId`) ensures that `pending` never contains more than one call of any given `ActivityResultKind`. While AndroidX doesn't return the `call_id`, your "match-by-kind" approach is safe under this invariant. Embedding `call_id` in Intent extras is unreliable as many contracts (like `PickVisualMedia`) don't return arbitrary extras.

2.  **Continuation type smuggling — FIX**
    **Bug:** In `CapabilityActivityCoordinator.kt` (line 258), you cast `cont` (a `CancellableContinuation`) to `CompletableDeferred<Any?>?`. This will throw a `ClassCastException` at runtime.
    **Recommendation:** Type the `continuation` field in `PendingCall` as `CancellableContinuation<Any?>` or `Any`. In `completeMatching`, simply cast it to `Continuation<Any?>` (which `CancellableContinuation` implements) to call `.resume()`.

3.  **CapabilityGrantPrompter cooperative wait — NIT/FIX**
    The `while (_pendingPrompt.value != null) delay(50)` spin-wait works for v0.1, but using a `kotlinx.coroutines.sync.Mutex` is more idiomatic and robust. Since `Mutex` is standard in your environment, switching to `mutex.withLock` for the grant request is a low-effort, high-signal improvement.

4.  **`isReturnDefaultValues = true` in unit tests — FIX/DESIGN**
    Relying on stubbed default values from `android.os.SystemClock` masks the fact that your expiry logic isn't actually being tested (as `elapsedRealtime()` always returns 0).
    **Recommendation:** Plumb a simple `Clock` interface (e.g., `interface Clock { fun elapsedRealtime(): Long }`) and inject it into `CapabilityHandleStore`. This allows you to write real expiry tests by "advancing" a mock clock.

5.  **Grant-row TTL — OK**
    Indefinite grants are standard for v0.1. As long as the "Settings" revocation path is eventually implemented, forcing re-consent after N days is a product decision, not a technical blocker.

6.  **Cross-plugin handle visibility — OK**
    The `CapabilityHandleStore.read` lookup is correctly scoped to the `pluginRowId`. The high entropy of the UUID handle (`cap-...`) combined with the explicit ownership check prevents leaks. 
    **NIT:** A defensive redaction of `callId` or `handle` from `Timber` logs in the coordinator/bridge is good practice but not a Phase 12 blocker.

7.  **Settings UI deferral — NIT/FIX**
    Deferring Step 16 (Revoke UI) is acceptable for a "mid-track" dev posture, but it is a **FIX** before Phase 12 closeout. Without it, privacy is one-way (consent is permanent until uninstall), which violates the "revoke at any time" promise in your `CapabilityGrantDialog`.

8.  **Camera FileProvider manifest entry — FIX**
    This is a functional blocker. `CameraBridge` will throw `io_error` on every capture attempt until the `<provider>` is added to the Android manifest. This should be addressed alongside the Step 16 follow-up.

9.  **One-active-op gate scope — OK**
    A process-global gate is defensive and avoids complex Intent stack issues on mobile. Per spec, this is the intended behavior for v0.1.

10. **`platform.fileBytes` chunk reassembly — OK**
    The 256 KB cap is reasonable. 16 MB is roughly 64 round-trips; given this happens after a user-initiated "rare" event (picking/capturing), the latency is negligible for v0.1.

### Additional Findings (Missing from your list)

11. **`PluginBridge` location capability check bug — FIX**
    In `PluginBridge.kt`, `requiredCapability` returns `"location"` for all `platform.location.*` methods. However, the manifest declares `"location.fine"` or `"location.coarse"`. The check `capability !in plugin.grantedCapabilities` will **always fail** because `"location"` is not in the set.
    **Fix:** Update `PluginBridge.dispatch` to check if `plugin.grantedCapabilities` contains either sub-capability when the required capability is `"location"`.

12. **Native SDK (`BridgePluginContext`) JSON overflow — FIX**
    In `BridgePluginContext.kt`, `pickInt` is used to decode `expiresAtMs`. Millisecond timestamps (like `1.7e12`) exceed `Int.MAX_VALUE` (`2.1e9`). `toIntOrNull()` will return `null`, causing `expiresAtMs` to default to `0L` in native plugins. 
    **Fix:** Add/use a `pickLong` helper for timestamp fields.

13. **Native SDK Gallery support — NIT**
    The `decodeGalleryResult` in `BridgePluginContext` only supports single-item returns due to the hand-rolled JSON parser's limitation. If multiple items are picked, only the first is returned. Acceptable for v0.1 native plugins, but should be documented.

14. **GalleryBridge `maxItems` inconsistency — NIT**
    In `GalleryBridge.kt`, you coerce `maxItems` to 10, but the `ActivityResultLauncher` in the coordinator is registered with a fixed `GALLERY_MAX_ITEMS`. This means if a plugin asks for 2 items, the user can still select 10. This is an AndroidX `PickMultipleVisualMedia` contract limitation; your current "coerce to 10" is a sensible v0.1 compromise.
