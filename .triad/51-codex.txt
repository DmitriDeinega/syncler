**Findings**

1. **RED: revoked/device-bound tokens can still mint a fresh active device.**  
   [server/app/routers/devices.py:18](/D:/Projects/syncler/server/app/routers/devices.py:18) uses `current_user` for `/v1/auth/devices/enroll`, and [server/app/auth.py:110](/D:/Projects/syncler/server/app/auth.py:110) ignores `did` and `revoked_at`. A revoked device-bound JWT can call enroll, get a new `session_token`, and regain access. This defeats the Phase 0 revocation guarantee. Fix with a bootstrap-only auth dependency that rejects `did` tokens, or validate `did` tokens for revocation before allowing enroll. Also make bootstrap tokens short-lived and scoped; currently they use the same 24h expiry at [server/app/auth.py:73](/D:/Projects/syncler/server/app/auth.py:73).

2. **RED: sensitive user routes still use user-only auth.**  
   Pairing still uses `current_user` for complete/list/revoke at [server/app/routers/pairing.py:155](/D:/Projects/syncler/server/app/routers/pairing.py:155), [server/app/routers/pairing.py:206](/D:/Projects/syncler/server/app/routers/pairing.py:206), and [server/app/routers/pairing.py:218](/D:/Projects/syncler/server/app/routers/pairing.py:218). Account deletion also uses `current_user` at [server/app/routers/auth.py:96](/D:/Projects/syncler/server/app/routers/auth.py:96). A revoked device can still mutate pairings or delete the account. Senders/plugins are okay as public/sender-signed surfaces; `/plugins/by-id` matches the plan’s “only when user-bound paths are added” caveat.

3. **RED: Android Hilt likely fails to compile.**  
   [UserStateRepository.kt:56](/D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/UserStateRepository.kt:56) moved `Clock` into the `@Inject` constructor with a default value. Dagger/Hilt still treats that as an injectable dependency, and there is no `Clock` binding. Restore a private `Clock.systemUTC()` production field plus a test-only secondary constructor, or add an explicit `@Provides Clock`.

4. **YELLOW: login publishes the bootstrap token as an unlocked session before enrollment finishes.**  
   [AuthRepository.kt:79](/D:/Projects/syncler/android/core/auth/src/main/kotlin/app/syncler/core/auth/AuthRepository.kt:79) calls `session.authenticate(response.sessionToken, masterKey)` before [AuthRepository.kt:80](/D:/Projects/syncler/android/core/auth/src/main/kotlin/app/syncler/core/auth/AuthRepository.kt:80) enrolls and swaps at [Session.kt:66](/D:/Projects/syncler/android/core/auth/src/main/kotlin/app/syncler/core/auth/Session.kt:66). Any observer that reacts to `isUnlocked=true` can start sensitive requests with the bootstrap token and get `device_required`. Avoid exposing unlocked state until the device-bound token is installed.

5. **YELLOW: CAS retry throw is catchable, but the control flow retries twice.**  
   The retry failure throws at [UserStateRepository.kt:318](/D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/UserStateRepository.kt:318). That is catchable by `push()`’s outer `runCatching`, but first it is caught by `attemptPush()` at [UserStateRepository.kt:283](/D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/UserStateRepository.kt:283), which calls `handleConflictAndRetry()` again at [UserStateRepository.kt:284](/D:/Projects/syncler/android/feature/inbox/src/main/kotlin/app/syncler/feature/inbox/UserStateRepository.kt:284). The dirty flag survives, but “retry once” is now effectively “retry twice.” Narrow the catch or return after the explicit 409 handling.

6. **YELLOW: unchanged tests are inconsistent with the new auth contract.**  
   `server/tests/test_state.py` still sends the bootstrap token to `/v1/state` at [test_state.py:38](/D:/Projects/syncler/server/tests/test_state.py:38). `test_auth.py` and the integration auth/device test similarly use the pre-enroll token after enrollment at [test_auth.py:214](/D:/Projects/syncler/server/tests/test_auth.py:214) and [test_integration_auth_devices.py:45](/D:/Projects/syncler/server/tests/test_integration_auth_devices.py:45). These should use the enroll response’s `session_token`.
