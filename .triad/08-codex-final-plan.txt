# Syncler V1 - Build Plan (Final, arbitrated by Codex)

Working name: **Syncler**. Final naming is deferred to V1.5.

V1 is a multi-device, end-to-end encrypted, plugin-host platform. External programs push structured messages to a user’s Android phone or phones. Each sender ships a signed JavaScript plugin defining its data, UI, notifications, and actions. The platform brokers identity, delivery, pairing, revocation, update checks, and encrypted state sync while remaining content-blind.

## 0. Executive Summary

**V1 stack:** Python FastAPI server, PostgreSQL, AWS free-tier-compatible deployment path, Kotlin Jetpack Compose Android app, Android WebView plugin host, JavaScript plugin SDK, Python sender SDK.

**Final estimate:** 216 engineering hours.

The earlier 140h estimate was too optimistic. The corrected estimate accounts for Android background execution, secure plugin hosting, signature verification, concrete crypto, rate limiting, retention, revocation, conflict resolution, and plugin update UX.

**V1 demo target:** trading bot and lottery plugin both function end-to-end. The lottery plugin is still used as the developer-experience dogfood test, but the deliverable is a DX issue report plus integration result, not a theatrical pass/fail claim.

## 1. Architecture

### Actors

- **User:** identified by UUID, owns one or more enrolled devices.
- **Device:** Android phone running Syncler; holds decrypted master key in memory after login.
- **Sender:** external program or backend; owns sender keypair.
- **Plugin:** signed JavaScript bundle distributed by sender; runs inside Syncler’s plugin host.
- **Server:** FastAPI + Postgres; stores opaque ciphertext, metadata, manifests, delivery state, revocation state, and FCM fan-out records.

### Trust Model

The server never sees password, master key, pairing keys, message plaintext, plugin private data, or decrypted user state.

The server does see metadata: account existence, device count, plugin IDs installed, sender pairings, message envelope fields, delivery timestamps, version requirements, revocation flags, and retention expiry.

Each plugin install requires explicit user permission grant per device.

### Data Flow

1. Sender SDK encrypts payload using the per-user pairing key.
2. Sender signs the message envelope.
3. Sender calls `POST /v1/messages/send`.
4. Server validates sender auth, rate limits, pairing status, plugin status, and revocation status.
5. Server stores encrypted body and delivery metadata.
6. Server sends FCM data message to enrolled devices.
7. Android app fetches message envelope and ciphertext.
8. App verifies envelope signature and plugin availability.
9. App routes payload to the verified plugin bundle in the plugin host.
10. Plugin decrypts payload through platform crypto APIs or receives plaintext only after native validation, depending on final implementation boundary.
11. Plugin returns notification descriptor.
12. Native app displays notification.
13. User opens card view; plugin renders message UI.
14. User taps action; plugin calls allowed endpoint through `platform.network.fetch`.
15. Sender SDK can wait for or poll action response.

## 2. Key Hierarchy

### Password and Account Keys

```
password
  -> Argon2id(password, salt=user_uuid || auth_salt, m=19 MiB, t=2, p=1)
      -> 64 bytes output
          -> auth_key = first 32 bytes
          -> master_key_wrap_key = second 32 bytes
```

Server stores:

```
auth_key_hash = SHA-256(auth_key)
encrypted_master_key
argon2_params_version
auth_salt
```

The password is never sent to the server.

### Master Key

```
master_key = 256-bit random value generated client-side at signup
encrypted_master_key = AES-256-GCM(master_key_wrap_key, master_key)
```

Password change re-derives the wrap key and re-wraps `master_key`. It does not re-encrypt all user data.

### HKDF Domain Separation

All derived keys use HKDF-SHA256 with explicit `info`.

```
pairing_key = HKDF(
  ikm = master_key,
  salt = sender_id_bytes,
  info = "syncler-v1-pairing-key:" || sender_id,
  length = 32
)
```

Additional future keys must use distinct `info` strings.

### AES-GCM Nonce Strategy

All AES-GCM encryption uses 96-bit nonces.

For message payloads:

```
nonce = 12 cryptographically random bytes generated by sender per message
ciphertext = AES-256-GCM(pairing_key, nonce, plaintext, aad)
wire_format = nonce || ciphertext || tag
```

AAD includes:

```
message_id
sender_id
user_id
plugin_id
min_plugin_version
created_at
schema_version
```

Nonce reuse with the same key is a release-blocking bug. Tests must verify nonce length, wire format, and AAD failure behavior.

### Key Separation

Separate keys are used for:

- authentication verifier
- master key wrapping
- pairing encryption
- plugin bundle signing
- sender API authentication
- message envelope signatures

No signing key is reused for encryption.

## 5. Plugin SDK

### Bundle Format

V1 uses a signed bundle plus manifest.

Required manifest fields:

```javascript
{
  id,
  name,
  version,
  senderId,
  bundleHash,
  signature,
  declaredCapabilities,
  declaredEndpoints,
  dismissBehavior,
  minPlatformVersion
}
```

Signature verification:

```
signature = Ed25519_sign(sender_private_key, canonical_manifest_without_signature || bundleHash)
```

Android verifies signature using sender public key from trusted pairing/server metadata before loading the plugin.

### Example

```javascript
import { BasePlugin, DismissBehavior } from '@syncler/plugin-sdk';

export class LotteryPlugin extends BasePlugin {
  static manifest = {
    id: 'com.lottery.app',
    name: 'Lottery',
    version: '1.0.0',
    senderId: 'com.lottery.sender',
    declaredCapabilities: ['network'],
    declaredEndpoints: ['https://lottery.app/api/*'],
    dismissBehavior: DismissBehavior.DISMISS_ALL,
  };

  async onMessage(payload) {
    const { numbers, draw_date } = payload;
    return platform.showNotification({
      title: 'Lottery numbers ready',
      body: `${numbers.length} sets for ${draw_date}`,
      importance: 'default',
    });
  }

  render(data) {}

  async onAction(actionName, payload) {
    if (actionName === 'played') {
      await platform.network.fetch('https://lottery.app/api/played', {
        method: 'POST',
        body: JSON.stringify({
          card_id: payload.card_id,
          numbers: payload.numbers,
        }),
      });
    }
  }
}
```

## 8. Cut Order If Budget Breaks

A credible secure V1 is not a 100h build. The old cut order protected hours instead of product value and is replaced.

Value-preserving cuts, in order:

1. Defer camera, gallery, file picker, and location bridge APIs to V1.5. Saves 10-12h.
2. Reduce lottery dogfood from external-agent attempt to internal integration plus DX notes. Saves 4h.
3. Ship only one demo integration, preferably lottery because it tests plugin DX better. Saves 6-8h.
4. Limit OEM hardening to Pixel + Samsung only. Saves 6-8h but increases delivery risk.
5. Defer `wait_for_action` long-poll ergonomics in Python SDK; keep action POST storage. Saves 4h.
6. Defer advanced settings screens for revocation/update management, but keep backend capability and minimal UI. Saves 6h.

Cuts that are not allowed for V1 without redefining the product:

- No static-only notification fallback as the planned V1 path.
- No unsigned plugin loading.
- No hand-waved nonce/key derivation.
- No single-device-only V1 if multi-device remains a stated demo requirement.
- No server-blindness compromise.
- No removal of revocation primitives.

## 9. Risks

1. **Android background delivery:** FCM and foreground service behavior varies by Android version and OEM. Mitigation: build M5 early, test on Pixel and Samsung, keep polling fallback for opened app.
2. **WebView plugin security:** WebView isolation is not equivalent to OS-level app sandboxing. Mitigation: verified bundles, minimized bridge, process/storage isolation, strict allowlists, timeouts, crash containment.
3. **Crypto implementation mistakes:** Silent failure risk is high. Mitigation: Tink/standard libraries, explicit nonce format, AAD tests, HKDF labels, code review.
4. **State sync conflicts:** Offline devices can mutate encrypted state independently. Mitigation: use mergeable state by entity ID for plugins, pairings, dismissals, and device records. Avoid lossy last-write-wins except for scalar preferences.
5. **Plugin update UX:** `min_plugin_version` can block old devices. Mitigation: explicit update prompt, defer option when safe, blocking screen when message requires newer plugin.
6. **Revocation propagation:** Devices may be offline when sender/plugin revocation occurs. Mitigation: check revocation on inbox fetch, plugin launch, and message action.
7. **AWS deployment scope creep:** AWS can consume build time. Mitigation: keep infrastructure minimal and scriptable; no multi-region or enterprise deployment in V1.
