Consultation 148 — V3 #17 Redis Backplane Review

The transition to a Redis backplane is architecturally sound and preserves the privacy/contract invariants established in V3 #14 and #16. The "fail-closed" posture and loud-warning configuration for production are correctly implemented in `lifespan` and `redis_client`.

Below is the verdict for each of your 12 concerns, followed by additional findings.

### Verdicts on Specific Concerns

1.  **Ordered-lane `_resolve_since_cursor` scan: NIT.**
    The current `XRANGE stream - +` is $O(N)$. For v0.1 with $N=1024$ it is acceptable, but using `XREVRANGE stream + -` with an early-exit upon finding the cursor is significantly better, as most since-cursor lookups will target recent messages.
2.  **`pubsub.aclose()` in finally blocks: OK.**
    `redis-py 5.2.1`'s `redis.asyncio.client.PubSub` correctly implements `aclose()`.
3.  **`RedisEventBus` subscriber-count race: OK.**
    The `async with self._lock` in both `subscribe` and `unsubscribe` correctly serializes task lifecycle management. If a new subscriber arrives while a cancellation is pending or just completed, the lock ensures the registry state is consistent before deciding to spawn a new reader task.
4.  **`publish_*` return value semantic drift: DESIGN.**
    You accurately identified that `delivered` now counts worker processes (Redis subscribers) rather than downstream devices. Since the spec explicitly notes this as "best-effort telemetry" and given the distributed nature of the backplane, this is an acceptable trade-off for v0.1.
5.  **Ordered-lane cursor uniqueness: OK/NIT.**
    Acceptable for v0.1. Moving to `XREVRANGE` (Concern 1) would also make the "most recent" duplicate the winner, which is a more intuitive fallback than the current "oldest" winner.
6.  **`prefixed(...)` string concat: OK/NIT.**
    The current thin concat is fine for v0.1. Users must ensure `LIVE_REDIS_KEY_PREFIX` includes a separator if they want one (e.g., `myapp:`), which is common in Redis namespacing.
7.  **`_next_event_id` module-scope `asyncio.Lock`: FIX.**
    Initializing a `Lock` at module scope is risky. While it's not bound to a loop until first use, if your test suite runs different tests in different event loops (common with `pytest-asyncio`), the lock will bind to the first loop and then raise `RuntimeError: Lock is bound to a different event loop` when hit by a subsequent test.
    *   **Recommendation:** Initialize the lock lazily inside `_next_event_id` or bind it to the `EventBus` instance.
8.  **Lazy `InProcessEventBus` construction: OK.**
    Verified that no existing tests or code rely on module-import-time construction of `_bus`.
9.  **`Settings.live_backplane` Literal: OK.**
    Pydantic's loud failure on invalid case/value for Literals is exactly what "fail-closed" requires.
10. **`docker-compose.yml` port mismatch: FIX.**
    Docker-compose exposes `6380:6379`, but `Settings.redis_url` defaults to `localhost:6379`. This breaks the "out-of-the-box" experience for developers running Redis via the provided compose file.
    *   **Recommendation:** Align the default `REDIS_URL` in `config.py` to `localhost:6380` or change the host port in `docker-compose.yml` to `6379`.
11. **Mixed-backend rolling deploy: OK.**
    Acceptable for v0.1 as documented in `live-backplane.md`.
12. **Server pytest PG-block: FLAG.**
    The lack of integration testing for multi-worker scenarios (cross-worker token redemption, fan-out) remains a high risk until the PG dev-box is restored.

### Additional Findings

*   **Redis Version Compatibility:** `RedisTokenStore.redeem` implements a robust fallback for Redis <6.2 using Lua for the atomic `GET` + `DEL` operation. This correctly supports older environments despite the dev-pin being at Redis 7.
*   **Ordered-lane `last_id` initialization:** In `_reader_loop_ordered`, if `since_cursor` is provided but not found, it correctly falls back to `last_id = "$"` (live only). However, if it *is* found and replayed, `last_id` is updated correctly to the last replayed ID, ensuring no gaps during the transition to `XREAD`.
*   **Reader Loop Health:** Both `RedisBroadcastHub` and `RedisEventBus` reader loops catch `Exception` and log with `logger.exception`, but they don't currently implement a retry mechanism if the reader dies (e.g., due to a transient Redis disconnect not caught by the pool).
    *   **Recommendation:** Consider a simple exponential backoff for reader task restarts in v0.2.

**Verdict: Greenlight for V4 subject to fixing Concern 7 (Lock) and Concern 10 (Port mismatch).** Concern 1 (XREVRANGE) is a strongly recommended optimization but not a blocker.
