**Findings**

- **High: concurrent update CAS can falsely return success for the loser.**  
  [server/app/services/state.py](d:/Projects/syncler/server/app/services/state.py:77) runs `UPDATE ... WHERE state_version = expected`, but never checks `rowcount` or uses `RETURNING`. If two clients read version `1`, both compute `new_version = 2`; one update wins, the other updates zero rows, commits, then [re-reads](d:/Projects/syncler/server/app/services/state.py:88) version `2` and returns success. With SQLAlchemy’s identity map this re-read may be even less reliable. Fix by treating `rowcount == 0` / no `RETURNING` row as conflict, then rollback/expire and fetch current state for the 409 body.

- **High: initial create race returns an unhandled DB error instead of CAS conflict.**  
  In the no-row path, both first writers with `expected_state_version = 0` can reach [db.add + commit](d:/Projects/syncler/server/app/services/state.py:63). The primary key prevents double insert, so they will not both succeed, but the loser gets an unhandled `IntegrityError`/500 rather than a 409 with current state. Catch `IntegrityError`, rollback, read current, raise `StateConflictError`.

- **Medium: `StateMerger` is not actually three-way, and user-scoped deletes/conflicts can lose data.**  
  [StateMerger.merge](d:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/StateMerger.kt:18) only has `local` and `remote`, no base/ancestor. For [userScopedStorage](d:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/StateMerger.kt:83), remote overwrites same keys and absent keys are treated as “unchanged”, not deletions. Since `StorageBridge` supports delete at [StorageBridge.kt](d:/Projects/syncler/android/feature/plugin-host/src/main/kotlin/app/syncler/android/pluginhost/capabilities/StorageBridge.kt:44), a local delete will be resurrected if remote still has the key, and a local edit to the same key is dropped on conflict. Remote-wins is acceptable only if this storage is cache-like/disposable; for durable user plugin state, add per-key metadata/tombstones or exclude this class from V1 sync.

- **Medium: 409 DTO does not match the actual FastAPI error envelope.**  
  Server returns `{"detail": {"current_state_version": ..., "current_encrypted_blob": ...}}` from [routers/state.py](d:/Projects/syncler/server/app/routers/state.py:69). FastAPI does not stringify the dict; it nests it under `detail`. Android defines [StateConflictBodyDto](d:/Projects/syncler/android/core/network/src/main/kotlin/app/syncler/core/network/SynclerApi.kt:83) as if the body were top-level, so parsing `errorBody()` directly into that DTO will fail. Add a `StateConflictResponseDto(detail: StateConflictBodyDto)` or change the server to return a direct JSON response for 409.

- **Medium: schema-v0/missing-field migration is only partially safe.**  
  Top-level missing arrays/maps are safe via `optJSONArray`/`optJSONObject`, but missing `schema_version` defaults to `SCHEMA_V1` at [EncryptedUserState.kt](d:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/EncryptedUserState.kt:46), so older blobs are silently treated as v1 instead of routed through a v0 migration. Nested records still use required `getString` calls, for example [installed plugin fields](d:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/EncryptedUserState.kt:83) and [modified_at](d:/Projects/syncler/android/core/storage/src/main/kotlin/app/syncler/core/storage/EncryptedUserState.kt:126), so older partial entries can still throw.

**Verification Notes**

1. **CAS race:** update path is not safe; two concurrent PUTs can both receive 200 for the same target version.
2. **`userScopedStorage` remote-wins:** too lossy for durable state, especially same-key edits and deletes.
3. **Missing fields:** top-level missing collections parse safely; true schema-v0 handling and nested missing fields are not safe enough.
4. **Base64:** no mismatch found. Server uses standard padded base64; Android helpers also produce standard/no-wrap strings.
5. **PUT empty blob:** rejecting length `0` is reasonable. Empty GET blob is the “no row/version 0” sentinel; persisted encrypted state should not be empty.
6. **409 parseability:** server body is valid JSON but nested under `detail`; Android needs an envelope DTO or server should return a direct body.
