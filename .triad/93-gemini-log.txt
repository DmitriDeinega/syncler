The Phase 6 plan is well-reasoned and addresses the remaining V1.5 DX loose ends effectively. Moving the broker from "accept any UUID" to a mandatory registry-backed model significantly improves the security posture of the automated pairing flow.

### 1. Item Review

*   **Item A (Pending-pairing registry): GREEN.** The shift to a mandatory reservation check in the broker handler correctly closes the "V1.5 deviation" and aligns the implementation with the intended security model. The proposed protocol changes are minimal and idiomatic.
*   **Item B (Trading-bot migration): GREEN.** Migrating the example bot is the necessary "dogfooding" step to prove the automated flow's UX. The plan to use an in-process broker thread for the bot keeps the example simple while demonstrating the end-to-end flow.

### 2. Answers to Open Questions

1.  **Item A — `complete` policy when not reserved:** **Raise `UnknownPairingIdError`.** This is the most robust approach. The storage layer should enforce its invariants. A handler that calls `complete` without checking `is_reserved` (or failing to handle a race condition) should be alerted to the violation rather than silently failing.
2.  **Item A — HTTP status for unknown `pairing_id`:** **404 with opaque body.** This is the correct REST representation for a non-existent pairing slot. Since `pairing_id` has high entropy (122 bits), a 404 does not leak useful state to a random prober, and an opaque body prevents distinguishing between "ID not reserved" and other internal rejections for an attacker who already has a valid ID.
3.  **Item B — In-process vs out-of-process broker:** **In-process via thread.** For a "smallest realistic example," requiring users to manage multiple processes (uvicorn + bot loop) adds friction. A background thread sharing `InMemoryBrokerStorage` is perfectly sufficient for the bot's single-user pairing use case.
4.  **Phase 6 commit shape:** **One commit.** Item A provides the infrastructure that Item B exercises. Splitting them would result in Item A being "dead code" (or at least untested by the example) until Item B lands. A single cohesive Phase 6 commit is preferable.
5.  **Revisit `Client.create_pairing_qr` reservation?** **Yes.** Although the call `self._broker_storage.reserve(...)` is already present in `client.py`, confirming its behavior as a mandatory step in the new registry model is essential.

### 3. Missing / Refinements

*   **Error Definition:** Ensure `UnknownPairingIdError` is added to `sdk-python/syncler/broker_storage.py` and included in `__all__`.
*   **Broker Handler Placement:** The `is_reserved` check in `make_app` must happen after parsing the JSON body (to get the `pairing_id`) but **before** the expensive `decrypt_bootstrap_envelope` call. This provides a cheap early-exit for invalid IDs.
*   **InMemoryStorage Idempotency:** In `InMemoryBrokerStorage`, `is_reserved` should return `True` if the ID is in the "pending" set **or** if it has already been "completed" (exists in `_entries`). This ensures that idempotent `complete` calls or late polls still pass the reservation check.

### 4. Security & Footguns

*   **Process Boundary:** The `InMemoryBrokerStorage` footgun (state not shared between processes) should be clearly highlighted in the trading-bot README. If a user tries to run the `broker` subcommand and the `pair` subcommand as two separate CLI invocations, the pairing will fail because the reservation in the first process won't be seen by the second. The `pair` command should handle the entire lifecycle (reserve -> start thread -> wait).
*   **Thread Safety:** The `InMemoryBrokerStorage` already uses a `threading.Lock`, which is good. Ensure the new `reserved_ids` set is also accessed within that lock.

**Conclusion:** The plan is dual-GREEN. Please proceed with the implementation, starting with Item A.
