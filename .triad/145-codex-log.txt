# Consultation 145 — V3 #16 design (card.patch privacy)

V3 #14 closed at `ecf0c63` (impl 1-12 + post-work triad 144
FIX list). V3 #15 (SDK `subscribe()` sugar) shipped at
`5f82f4b`. V3 #16 is the next item: field-level live updates
on template-rendered cards.

Both 141 reviewers + the 144 verdict echoed that #16 needs its
own design triad BEFORE implementation because the privacy
contract is non-trivial. v0.1 dev posture preserved.

## What V3 #16 is for

Today the live-card path is whole-card upsert:
- Sender pushes a `cards.upsert` envelope containing a fresh
  `recipient_envelopes[]` set sealing the FULL payload for
  every active device. Device opens its envelope, decrypts,
  re-renders the template card.
- Cost: every sender update re-encrypts the whole payload
  per device.

#16 wants finer-grained: "update just the `subtitle` field on
this card without re-running JSONPath on the whole payload."
Use cases: live sports scores updating just one number;
typing-indicator updating just a status line; presence dot
changing color.

Wire pattern from `docs/live-channel.md` "#16 hook":

```json
{
  "type": "card.patch",
  "plugin_row_id": "...",
  "card_id": "...",
  "field": "$.subtitle",
  "value": "Updated"
}
```

But that wire shape has `value` in plaintext — both reviewers
flagged that as a CRITICAL design bug. The server would see
the field update, and the V2 E2EE contract evaporates.

## Proposed design — V3 #16a

### Encryption contract

A card.patch is a V2-shape envelope just like cards.upsert,
not a special "field-level" payload. The plaintext "value" is
inside the encrypted payload, NOT in the outer wire frame.

```json
{
  "type": "card.patch",
  "plugin_row_id": "...",
  "card_id": "...",
  "card_seq": 42,
  "recipient_envelopes": [
    { "device_id": "d1", "envelope": "<base64 HPKE-sealed>" },
    { "device_id": "d2", "envelope": "<base64 HPKE-sealed>" }
  ]
}
```

The HPKE-sealed payload is `{field_path, new_value}` — the
plaintext field name + value. Devices open their envelope,
extract the patch, apply to local card state.

This means: the server STILL sees `plugin_row_id` + `card_id`
+ `card_seq` (already public metadata in V2 today). The
server NEVER sees the field path or the new value.

### Renderer integration

The Android `TemplateCard` Composable currently re-runs
`resolveJsonPath` on every recomposition driven by a fresh
payload JSON. For #16, the patch path:

1. Device receives `card.patch` envelope on its live channel.
2. Decrypts → `{field_path, new_value}`.
3. Local card state holds the *current* payload JSON (from
   the original upsert + accumulated patches).
4. Apply the patch: replace the value at `$.subtitle` (etc.)
   in a deep-cloned payload JSON.
5. Re-render the card with the patched payload.

No new "patch mode" in the renderer — the renderer just sees
a new payload string. The patching is host-side state
management.

### Sequence numbers

card.patch needs ordering to apply correctly. If patches
arrive out of order (live channel is best-effort + ephemeral
per V3 #14), the device might apply a stale patch over a
newer one.

Options:
- (a) Require `card_seq` to be strictly monotonic; reject
  patches with `card_seq <= last_applied_seq`. Skip + fall
  back to inbox pull on gap.
- (b) Use the ordered lane (V3 #17 work) for card.patch
  delivery. Plays nicely with the durable Redis Streams.
- (c) Each patch carries `base_seq` (the upsert it patches
  against) + a small int incrementing per-patch. Reject if
  base_seq doesn't match the device's current card_seq.

(c) is what live-card upserts already do (sequence-monotone
CAS). Extending it to patches is natural.

### Wire shape

Server-side: cards router gains a `card_patch` action via
the same `/v1/cards/upsert` endpoint, dispatched by the
envelope's `wire_kind` field. Or a new
`/v1/cards/patch` endpoint — probably cleaner.

The patch wire frame ALSO rides the live channel for low
latency. But it also writes to a small server-side `card_patches`
table so a device that wasn't connected at patch time can
catch up via inbox pull (the live channel is ephemeral; the
inbox is authoritative).

### Privacy invariants

The patch's payload must:
1. Use the same V2 envelope shape (per-device HPKE + outer
   Ed25519 sender signature) as cards.upsert. Same key
   schedule, same recipient set rules.
2. NEVER leak the field path or value in the outer frame.
3. Respect the existing `min_plugin_version` + revocation
   gates.

### Renderer security implications

Field-level patches operate on a payload object the renderer
already trusts. A patch can't escalate beyond what the
manifest's `template.fields` already permits — the renderer
ignores unknown field names.

But: a malicious patch could OMIT a field present in the
original payload. The renderer falls back to "no value" for
that field. Acceptable? Or does the patch need to be
explicitly typed (`update` / `unset` / `replace`)?

## Concerns I want a second opinion on

1. **Wire-shape choice.** Reuse `cards.upsert` shape with a
   `wire_kind: "patch"` discriminator vs. dedicated
   `/v1/cards/patch` endpoint. Both reviewers' instinct?

2. **Sequence model.** Option (a) strict monotonic seq, (b)
   defer to V3 #17 ordered lane, (c) base_seq + patch_seq.
   I lean (c) — minimal new state on the device + maximally
   compatible with the ephemeral live channel.

3. **Cross-channel routing.** Patches ride the live channel
   AND get persisted server-side for inbox catch-up. Is the
   dual-write the right answer, or should patches be
   ephemeral-only (accept that disconnected devices miss
   field-level updates between full upserts)?

4. **Renderer state model.** Today's TemplateCard is stateless
   — it computes from `payloadJson` each render. #16 needs
   a stateful layer (current_payload accumulating patches).
   Where does that state live? `InboxViewModel`? A new
   `LiveCardStore`? Per-card Compose `remember { }`?

5. **Patch ops.** Only `replace` for V0.1, or also `unset`
   / `array_push`? The spec doesn't pin this — replace-only
   is the simplest contract.

6. **Sender SDK ergonomics.** Sender code currently calls
   `client.upsert_card(card_id, payload, ...)`. Adding
   `client.patch_card(card_id, field_path, new_value, ...)`
   is straightforward but requires the sender to know the
   field path syntax (`$.subtitle`). Acceptable, or pre-
   require the sender to pass a "patch ops" array?

7. **Patch validation.** Server validates the patch
   envelope shape (recipient set classifier, sender
   signature, sequence) but CANNOT validate the field path
   matches the manifest's template fields (patch payload is
   encrypted). The device-side renderer is the last gate.
   Is that acceptable, or should the patch's field name
   travel UNENCRYPTED so the server can reject malformed
   patches at publish time?

8. **Coexistence with whole-card upserts.** A sender can
   alternate: upsert → patch → patch → upsert → patch.
   The upsert resets `card_seq` to a new value; the device
   should reject patches whose `base_seq` predates the
   current `card_seq`. Trivial monotone-reject logic, but
   worth pinning.

9. **Renderer flicker on patch.** Compose recomposes when
   `payloadJson` changes; a patch causes a full TemplateCard
   re-render even though only one field changed. For V0.1
   that's probably fine; Compose's diffing is efficient. But
   if a stat_grid is being patched at high frequency, every
   patch re-renders all 8 tiles. Worth a granular
   `payloadFlow: Map<String, State<String?>>` model where
   each field path is its own observable, or premature
   optimization?

10. **Patch storage retention.** Server-side patch table:
    how long to retain? Whole-card live updates use 48h TTL
    today. Patches between two upserts can be collapsed
    (apply in order, fold into a new "virtual upsert"). Is
    a separate patch table worth maintaining, or just rely
    on V0.1 in-process buffer (lost on restart, devices
    catch up via the next sender upsert)?

11. **Triad scope.** This is the V3 #16 design pre-work
    triad. The implementation will mirror V3 #14's 13-step
    structure with its own post-work triad. Is the scope
    above tight enough to ship in 4-6 commits, or am I
    underestimating?

**Verdict: v2 design pass before implementation.** The core privacy correction is right: `field_path` and `new_value` must be inside the same V2 encrypted envelope model as `cards.upsert`. But the sequence/catch-up model needs tightening before code, because patch correctness depends on exactly what state a disconnected device can reconstruct.

1. **Wire-shape choice: DESIGN**

Use a dedicated `/v1/cards/patch` endpoint, but keep the inner envelope contract shared with `cards.upsert`.

Reason: semantically this is a different mutation type with different ordering and persistence rules. A `wire_kind: "patch"` discriminator inside `/v1/cards/upsert` risks blurring validation paths and making upsert-specific assumptions leak into patches.

Recommended outer shape:

```json
{
  "type": "card.patch",
  "plugin_row_id": "...",
  "card_id": "...",
  "base_seq": 42,
  "patch_seq": 7,
  "recipient_envelopes": [...]
}
```

The server validates only metadata, sender auth/signature, recipient set rules, version/revocation gates, and sequence metadata. It must not see `field_path` or `new_value`.

2. **Sequence model: DESIGN**

I would not use plain strict monotonic `card_seq` alone. It is too ambiguous once whole-card upserts and patches coexist.

Use:

- `card_seq`: version of the latest full-card upsert.
- `patch_seq`: monotonically increasing within that `card_seq`.
- encrypted payload contains the operation.
- outer frame contains `base_seq = card_seq`.

Device logic:

- If `base_seq < current_card_seq`: reject stale patch.
- If `base_seq > current_card_seq`: gap; trigger inbox/full-card catch-up.
- If `patch_seq <= last_patch_seq_for_base`: reject duplicate/stale patch.
- If `patch_seq > last_patch_seq_for_base + 1`: gap; trigger catch-up, do not apply.
- Else apply.

This is basically option `(c)`, but pin the names and rejection behavior.

3. **Cross-channel routing: DESIGN**

Dual-write is the right answer if patches are user-visible state rather than cosmetic hints.

Ephemeral-only patches create divergent cards for disconnected devices until the next full upsert. That is probably unacceptable for scores, status, presence labels, or anything the user may later inspect.

But do not persist patches indefinitely as an ever-growing log. Better model:

- live channel: low-latency delivery.
- inbox/catch-up: authoritative reconstruction.
- storage may retain either ordered patches or a collapsed latest encrypted card snapshot.

For V0.1, keeping a patch table with TTL matching existing card retention is acceptable, but the design should say catch-up must never require an unbounded patch chain.

4. **Renderer state model: FIX**

Do not put the accumulator in per-card Compose `remember { }`.

That makes patch state lifecycle too fragile: recomposition, navigation, process death, list recycling, and inbox refresh can all desync it.

Use a host-side store/view model layer. My recommendation:

- `LiveCardStore` owns current card state keyed by `(plugin_row_id, card_id)`.
- `InboxViewModel` observes/exposes rendered card state from that store.
- `TemplateCard` remains stateless and receives `payloadJson`.

This preserves the renderer boundary: Compose renders; host state applies upserts/patches.

5. **Patch ops: OK**

Replace-only for V0.1.

Explicitly define the encrypted patch payload as something like:

```json
{
  "op": "replace",
  "field_path": "$.subtitle",
  "new_value": "Updated"
}
```

Do not support implicit unset-by-omission. A patch should mutate exactly one field and leave everything else unchanged. If later you need `unset`, add it as a distinct op with explicit renderer/device semantics.

6. **Sender SDK ergonomics: OK**

`client.patch_card(card_id, field_path, new_value, ...)` is acceptable for V0.1.

I would still make the SDK API operation-shaped internally:

```python
client.patch_card(card_id, op="replace", field_path="$.subtitle", value="Updated")
```

or:

```python
client.patch_card(card_id, [{"op": "replace", "path": "$.subtitle", "value": "Updated"}])
```

For V0.1, allow exactly one replace op. That keeps the wire extensible without adding multi-op complexity now.

7. **Patch validation: OK**

Do not move the field path into plaintext just so the server can validate it.

That would violate the privacy goal. Server-side manifest validation is not worth leaking which field changed. Device-side validation is the correct final gate.

Required invariant: unknown or disallowed `field_path` must be ignored or rejected locally without mutating card state. Prefer reject-the-patch for malformed paths; ignore unknown template fields only if that is already renderer behavior.

8. **Coexistence with whole-card upserts: FIX**

Pin this explicitly.

Whole-card upsert should reset the patch chain:

- upsert with `card_seq = N` replaces current payload.
- device sets `current_card_seq = N`.
- device sets `last_patch_seq = 0`.
- patches with `base_seq < N` are rejected.
- patches with `base_seq > N` trigger catch-up.
- patches with `base_seq = N` apply only in contiguous `patch_seq` order.

Also decide whether `card_seq` is sender-generated, server-assigned, or server-CAS validated. For correctness, server must enforce monotonicity or CAS semantics on accepted mutations.

9. **Renderer flicker on patch: OK**

Full `TemplateCard` re-render from changed `payloadJson` is fine for V0.1.

Do not introduce `Map<String, State<...>>` yet. That complicates renderer APIs, state ownership, and test surface. If `stat_grid` patch rates become a real performance issue, optimize after measurement.

10. **Patch storage retention: DESIGN**

A separate patch table is acceptable, but only if the design defines compaction/catch-up semantics.

Minimum V0.1 acceptable contract:

- patches retained for the same TTL as the card/inbox state, currently 48h if that is the existing live-card TTL.
- catch-up returns either the latest full payload plus subsequent patches, or a fresh full upsert/snapshot.
- if required patches are missing, the device must not apply a partial chain; it should request/rely on a full upsert.

I would avoid an in-process-only buffer for #16. That makes correctness dependent on server uptime and undermines the inbox-authoritative story.

11. **Triad scope: DESIGN**

The implementation is probably bigger than 4-6 commits if done cleanly.

Likely slices:

1. Spec/wire docs and tests for privacy shape.
2. Server endpoint/router and validation.
3. Patch persistence/catch-up model.
4. SDK `patch_card`.
5. Android/device decrypt and sequence handling.
6. `LiveCardStore`/state accumulator.
7. Renderer integration tests.
8. Coexistence/gap/stale patch tests.
9. Post-work hardening/docs.

You can still ship in a compact series, but I would not assume this is just a small extension of `cards.upsert`.

**Additional Invariants**

- Outer frame must never contain `field`, `path`, `value`, op-specific plaintext, or a hash/digest of the field path unless explicitly analyzed for leakage.
- Server must validate recipient set equivalence to the current active device set using the same rules as `cards.upsert`.
- Patch payload must be authenticated as coming from the sender, not merely encrypted to recipients.
- Device must apply patches atomically: decrypt, validate sequence, validate op/path, deep-clone/mutate, then publish new payload state.
- Gap handling must fail closed: do not apply future patches while missing earlier ones.
- Whole-card upsert always wins over older patch chains.
- Unknown/malformed patch op must not partially mutate local payload.
- Catch-up must be deterministic: two devices receiving the same upsert plus same valid patch sequence must end with byte-equivalent logical payload.

**Go-List**

Proceed to implementation only after pinning:

- dedicated `/v1/cards/patch`;
- `card_seq/base_seq + patch_seq` semantics;
- durable or snapshot-backed catch-up;
- `LiveCardStore` as the accumulator owner;
- replace-only encrypted patch payload;
- explicit stale/gap/upsert-reset behavior.

With those pinned, the E2EE posture holds and the implementation has a clear correctness model.
