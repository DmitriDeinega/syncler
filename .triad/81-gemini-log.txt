### Reviewer Observations

#### Action Endpoint Scheme Rule (Item 2)
The transition to `new URL()` (SDK) and `urllib.parse.urlparse` (Server) is robust. 
- **Userinfo**: Properly rejected for both `http` and `https` schemes.
- **IPv4 Octets**: The new `isLanPrivateIpv4` (SDK) and `_is_lan_private_ipv4` (Server) helpers correctly validate all four octets against the 0..255 range.
- **LAN Coverage**: The 10.x, 172.16-31, 192.168.x, and 127.x ranges are correctly covered. The inclusion of `127.x` for loopback is appropriate for local development.
- **Edge Cases**: The use of standard URL parsers correctly handles ports, fragments, and queries. IPv6 literals in brackets (e.g., `[::1]`) are safely rejected for HTTP cleartext as they don't match the IPv4-only LAN regex.

#### Semver Parity (Item 1)
There is a minor asymmetry in the `semverPattern`:
- The SDK regex (`/^(0|[1-9]\d{0,5})\.../`) rejects leading zeros (e.g., `1.0.01`) and accepts build metadata (e.g., `1.0.0+build.1`).
- The server regex (`_SEMVER` in `services/plugins.py`) accepts leading zeros but **rejects** build metadata.
- **Recommendation**: This is a non-blocker as the SDK's 6-digit cap fulfills the primary goal of keeping integer parsers in agreement, and build metadata is rarely used in this context.

#### cardKeyPath and Plugin ID (Items 4 & 5)
The regex `^[a-zA-Z][a-zA-Z0-9-]*(\.[a-zA-Z][a-zA-Z0-9-]*)+$` is perfectly mirrored. The `cardKeyPath` change is a significant hardening, moving from a simple `startsWith("$")` to the same grammar used for template fields.
