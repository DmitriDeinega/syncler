1. **OK** — Explicit `Settings -> Security -> I lost a device` is the right v0.1 entry point. Do not add inactivity-driven banners yet. Inactivity is ambiguous and risks nudging users toward a destructive security action.

2. **NIT** — State machine is good. I would add one explicit preflight/guard state before `confirm_revoke`:

   `check_rotation_ready`

   Purpose: confirm this device can actually complete rotation before revoking the lost one, or at least warn if rotation prerequisites are unavailable. If rotation requires current encrypted state, account auth freshness, or master password availability, the user should know before the irreversible revoke step.

   Also rename `rotation_recommended` to something stronger in copy, even if the state name stays neutral. This is not just hygiene; after device loss, rotation is the security completion step.

3. **OK** — Revoke atomic, rotation resumable later is acceptable for v0.1. The banner should be tied to a concrete local marker like:

   `revoked_device_without_rotation_at`

   It should clear when rotation succeeds, when dismissed forever, or after the timeout. I would use a timeout measured in days, not app launches.

4. **OK** — Idempotent revoke plus stale-challenge handling is the right shape. The UX copy should avoid implying failure when the desired security state was already reached. For rotation 409s, distinguish:

   - already completed: “Master key was already rotated on another device”
   - currently in progress: “Rotation is already in progress on another device”
   - stale local challenge: “This rotation session expired; start again”

5. **NIT** — Follow-up CTA is acceptable for v0.1, but make it more concrete than “Re-pair senders.” The user may not understand which senders were affected. If the app can infer sender/plugin relationships associated with the revoked device, show a checklist or count. If it cannot, say “Review paired senders” and deep-link to the existing sender/pairing surface.

6. **OK** — Store `plugin_prefs` inside encrypted user-state. That matches the privacy model and gets cross-device sync without new server-visible state. Add a soft size budget to the spec now, even if enforcement waits: e.g. label caps, known enum values, no unbounded per-plugin history in this blob.

7. **FIX** — Schema is mostly right, but keying by `plugin_row_id` is fragile if row IDs are server-local lifecycle artifacts or can be recreated. Prefer a stable plugin identity if one exists, such as sender identity plus plugin manifest ID/version namespace.

   If `plugin_row_id` is truly stable across devices and account lifetime, document that invariant explicitly.

   Cadence enum is fine for v0.1. For batching, use wall-clock aligned windows for predictability and power efficiency, not rolling from each receipt. Rolling sounds intuitive until users get inconsistent notification times and every plugin maintains its own timer.

8. **OK** — Device-side intercept is correct. The server cannot inspect encrypted prefs, and OS DND is too coarse.

   One important distinction: quiet hours should suppress notifications, not data ingestion. The SSE/live hint can still trigger local sync if the app is active or the repository policy wants freshness. The notification adapter should decide whether to notify, queue a summary, or remain silent.

9. **OK** — Top-level `Plugins` tab is the cleaner model. Sender settings are trust/pairing controls; plugin settings are host policy. Mixing them will get confusing as soon as one sender exposes multiple plugins.

10. **OK** — Missing key defaults to empty dict. Add forward-compatible parsing rules: ignore unknown cadence values by falling back to default, preserve unknown fields when possible during encrypted-state writeback, and avoid deleting prefs for plugins that are temporarily unavailable.

**Bigger Picture**

The overall V4 split is sound. #18 should be treated as a security recovery wizard, not just a convenience wrapper. The main thing I would tighten is the revoke-before-rotate edge: either preflight rotation readiness before revoke, or make the copy explicit that revoke is immediate and rotation may need to be resumed.

For #19, encrypted user-state is the right storage location, but the identity key matters. I would not ship the schema until the plugin preference key is stable across reinstall, re-pair, server migration, and plugin row recreation scenarios. Everything else can evolve behind that.

Implementation order looks good. I would slightly adjust #19 tests to include cross-device merge/CAS behavior, because prefs living in encrypted user-state means stale writes are the main regression risk.
