**Findings**

- HIGH: Android `confirm()` does not compare the full preview identity. It checks `senderId`, `senderPublicKey`, and fingerprint, but not `senderName` or `senderNameHash`, even though `/preview` returns both and the persisted trust record stores response values. A server bug or race that changes only name/name_hash would pass the mismatch gate and persist unconfirmed identity data. See [PairingRepository.kt](D:/Projects/syncler/android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:68) and persistence at [PairingRepository.kt](D:/Projects/syncler/android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:80). Also, revoke-on-mismatch is best-effort and ignores unsuccessful responses at [PairingRepository.kt](D:/Projects/syncler/android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:73).

- HIGH: migration `0003` downgrade is not generally safe after the new behavior is used. Once re-pair-after-revoke creates multiple historical rows for the same `(user_id, sender_id)`, `downgrade()` drops the partial unique index and recreates the old full unique constraint, which can fail on valid M6.1 data. See [0003_pairing_partial_unique.py](D:/Projects/syncler/server/alembic/versions/0003_pairing_partial_unique.py:33). I would either document that downgrade requires no duplicate historical pairings, or clean/merge rows before recreating the hard unique constraint.

- MEDIUM: ORM metadata no longer matches the migrated production schema. [Pairing](D:/Projects/syncler/server/app/models.py:60) removed the hard unique constraint but does not declare the partial unique `Index`, while tests use `Base.metadata.create_all()` in [conftest.py](D:/Projects/syncler/server/tests/conftest.py:115). That means test/dev schemas created from metadata do not enforce double-active uniqueness, so the two-token same-user/same-sender race is only protected in Alembic-migrated DBs.

- MEDIUM: `/v1/pairing/preview` has no auth by design, and token brute force is negligible, but it is still an unauthenticated DB lookup endpoint with no IP rate limit. That is a DoS hygiene gap, not a token secrecy issue. See [pairing.py](D:/Projects/syncler/server/app/routers/pairing.py:114).

**Agreements With Fixes**

The core fixes are mostly correct: initiate now rate-limits by IP before auth and sender after signature verification; preview is non-consuming; single-token completion is race-safe via atomic `UPDATE ... RETURNING`; broker URLs now emit URL-safe no-padding base64; Android moved to two-phase preview/confirm; cancel no longer creates trust; JSON storage avoids delimiter corruption and base64-preserves binary fields; pairing is reachable from inbox; `expires_at` is plumbed into the DTO; and the pipeline rejects unpaired senders.

Caveats: standard base64 in a raw query URL still needs percent-encoding if it contains `+`; otherwise normal URL parsing can turn it into a space. Body `/complete` is fine. The pipeline is acceptable only as the stated stop-gap; it does not yet do layer-4 envelope verification despite the top comment implying it does.
