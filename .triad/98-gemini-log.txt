**GREEN** on the overall plan direction. Option A is the correct architectural choice for a root key rotation in an E2EE system. The clean break ensures cryptographic hygiene, and the UX friction of re-authentication is appropriate for the severity of the operation.

Here are the answers to your specific open questions:

1. **Option A vs B vs C:** Option A is definitely the right path. Options B and C introduce significant complexity (managing key lifecycles, dual-encryption states) and risk (retaining compromised keys, rollback vulnerabilities). A clean generational break is standard practice for root key compromise. Forcing a re-login on other devices is a security feature in this context.
2. **Phase scoping:** Split into **8a (spec/protocol)** and **8b (implementation)**. Cryptographic orchestrations require rigorous threat modeling before code is written. Nailing down the API contract, atomicity guarantees, and client-side state machine in a spec first will save you from gnarly E2EE race conditions later.
3. **Pairing.encrypted_state CAS:** Yes, `pairings` absolutely needs a `key_generation` or `state_version` column to participate in the CAS check. If a rotation happens concurrently with a sender pushing a pairing state update, a race condition could leave the pairing state encrypted under an unpredictable generation. The entire rotation payload (user state + all pairing states) must commit atomically against a specific generation.
4. **Per-sender HKDF-derived pairing keys:** Your realization is 100% correct. If you rotated the pairing keys, you would break the senders. The device decrypts the user-state blob with the old master key, extracts the existing pairing keys verbatim, and re-encrypts the blob with the new master key. The pairing keys are data, not derived live. This saves you from the nightmare of protocol-level re-pairing.
5. **Compromise rotation revokes sessions?:** Yes, the API should accept a `reason` parameter (e.g., `compromise`, `password_change`, `hygiene`). If `compromise`, the server MUST revoke all session tokens except the initiating one. Even for `password_change`, it is better UX to proactively revoke other sessions so the client immediately prompts for the new password, rather than failing opaquely with CAS or decryption errors in the background.

**Anything missing:**
*   **Offline Device Merge Conflict:** If Device B is offline, makes local state changes (e.g., dismisses a card), and then comes online *after* Device A has rotated the master key, Device B's sync will fail. Device B needs a graceful recovery path: detect the generation bump, prompt the user for the new password (if changed), fetch the new wrapped master key, decrypt the *new* server state, merge its local pending changes on top, and push a new CAS update.
*   **Transaction Payload Limits:** Atomically re-encrypting and uploading the `encrypted_user_state` blob plus *all* `pairings.encrypted_state` rows in a single HTTP request could get heavy for power users with many pairings. Ensure your server framework and database transaction limits are configured to handle a potentially multi-megabyte atomic rotation payload.

**Anything new (security, footgun, attacker model gap):**
*   **Downgrade / Rollback Attack:** A malicious server could theoretically serve an older `key_generation` and its corresponding older encrypted data to a client. Clients must persistently store the highest `key_generation` they have successfully synced locally and hard-fail if the server ever serves a generation lower than their high-water mark.
*   **Atomicity Footgun:** The server-side commit must be absolutely bulletproof. If the database transaction commits the new `encrypted_master_key` but fails midway through updating the `pairings` rows, the account is permanently bricked (keys and data desynced). Use strict, single-transaction atomic commits for the entire rotation request.
