## Disagreement 1: Migration

**Position:** Concede to a hybrid: **push existing local pairings on first unlocked launch**, but mark them as migrated and surface one review prompt.

**Smallest safeguard:** migrated pairings get `origin = migrated_local`, `migrated_at`, and a non-blocking “Review paired senders” notice. No mandatory acknowledgement gate.

**Minimum-correct implementation:**
- On first unlocked launch per device/account, merge local `PairedSenderStore` entries into M7.
- Use stable pairing fingerprint/id for idempotency.
- Add delete tombstones so later-migrating old devices cannot resurrect a pairing already removed post-M7.
- Mark migrated entries as `migrated_local`.
- Show one review entry point after migration; user can revoke/remove normally.

That preserves post-sync UX without pretending pre-sync deletes can be perfectly inferred.

## Disagreement 2: Mute Semantics

**Position:** Accept middle-ground: two separate settings, but keep UI minimal.

**Smallest implementation:**
- Synced setting: **Mute everywhere**.
- Local setting: **Mute on this device**.
- Effective mute = `mute_everywhere || mute_on_this_device`.

This handles both cases cleanly: spam follows the user when desired, bedside/tablet preferences stay local. It is not over-engineering because the domains are genuinely different.
