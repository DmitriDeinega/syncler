## v4 deltas vs v3

### Block 1: `reportError` BEFORE `scope.cancel()` (Codex 134 #1)

The init-timeout path:

```kotlin
// Init runs in a child job we can cancel without taking
// the parent scope (which owns the reporting machinery) down.
val initJob = pluginScope.async {
    val result = withTimeoutOrNull(INIT_TIMEOUT_MS) {
        plugin.onInit(context)
    }
    if (result == null) {
        // 1. Report first — we're still inside a healthy
        //    scope; HostSignals.reportError is synchronous
        //    so this lands before any cancellation.
        hostSignals.reportError("init_timeout", "")
        // 2. Now cancel the init Job specifically. The plugin
        //    scope stays alive so subsequent destroy() can
        //    still call webViewHost.destroy() and the
        //    coordinator can fire onPluginUnloaded.
        coroutineContext.cancel()
        return@async
    }
    result.fold(
        onSuccess = { hostSignals.reportReady() },
        onFailure = { hostSignals.reportError("init_failed", it.message ?: "") },
    )
}
```

`HostSignals.reportError` calls a synchronous lambda on the
parent coordinator (not suspendable), so the report lands
deterministically before cancellation cascades.

### Block 2: sandbox-side DEX size cap (Codex 134 #2)

In addition to the host-side cap (which validates the staged
bundle before opening the FD), the sandbox reads the
ParcelFileDescriptor with a hard cap:

```kotlin
const val DEX_MAX_BYTES = 4 * 1024 * 1024  // 4 MB

fun readDexBytes(fd: ParcelFileDescriptor): ByteBuffer {
    val stream = AutoCloseInputStream(fd)
    val out = ByteArrayOutputStream()
    val buf = ByteArray(8192)
    var total = 0
    while (true) {
        val n = stream.read(buf)
        if (n < 0) break
        total += n
        if (total > DEX_MAX_BYTES) {
            throw IllegalStateException(LoadFailureCodes.DEX_TOO_LARGE)
        }
        out.write(buf, 0, n)
    }
    return ByteBuffer.wrap(out.toByteArray())
}
```

`DEX_TOO_LARGE` joins the `LoadFailureCodes` set.

### Block 3: per-token ServiceConnection (Codex 134 #3)

Each native plugin's bind owns its own `ServiceConnection`
instance. No process-global "current token" state.

```kotlin
class NativePluginBinding(
    val sandboxToken: Int,
    val instanceName: String,  // = sandboxToken.toString()
    val onDeath: (deadToken: Int) -> Unit,
) : ServiceConnection {
    @Volatile var sandbox: IPluginSandbox? = null
    override fun onServiceConnected(name: ComponentName, binder: IBinder) {
        sandbox = IPluginSandbox.Stub.asInterface(binder)
    }
    override fun onServiceDisconnected(name: ComponentName) {
        sandbox = null
        onDeath(sandboxToken)
    }
}
```

`PluginSandboxConnection.bindIsolatedNative(token)` creates a
fresh `NativePluginBinding`; `unbindNative(token)` removes
it. The connection registry is `ConcurrentHashMap<Int,
NativePluginBinding>` keyed by token; no shared state across
plugins.

`SandboxRouter.handleSandboxDeath(deadToken)` becomes:

```kotlin
private fun handleNativeSandboxDeath(deadToken: Int) {
    val removed = handles.remove(deadToken) ?: return
    scope.launch {
        bridgeDispatcher.onPluginCrashed(deadToken, "process_died")
        // No global cleanup needed — the binding's own
        // ServiceConnection lifecycle is bounded to this token.
    }
}
```

JS `:plugin` death still calls the v2 `handleSandboxDeath` to
reap all JS tokens (they share one process).

### Block 4: ClassLoader / dependency policy (Codex 134 #4)

The native sandbox's `InMemoryDexClassLoader` is constructed
with `parentClassLoader = SynclerPluginRuntime::class.java.classLoader`.
The parent exposes ONLY:

**Allowed (parent-provided)**:
- `kotlin.*`, `kotlinx.coroutines.*`, `kotlinx.serialization.*`
  (whatever's bundled with the `:plugin-sdk-runtime` module).
- `app.syncler.plugin.runtime.*` — the public SDK surface
  (`SynclerPlugin`, `PluginContext`, request/response types).
- Standard Java SE that ART exposes: `java.util.*`,
  `java.time.*`, `java.io.*`, `java.nio.*`, etc.
- Android framework classes the isolated process can
  actually call (most are no-op'd by isolatedProcess
  restrictions, but they exist).

**Forbidden (host process internals)**:
- `app.syncler.android.pluginhost.*` — NOT on the `:plugin-sandbox`
  classpath, so plugins can't accidentally reach the host's
  `PluginBridge` / `SandboxRouter` / `AuditLogger` etc.
  Reflective access still fails at class-resolution.
- `app.syncler.core.network.*`, `app.syncler.core.crypto.*`,
  `app.syncler.core.storage.*` — same: not on the sandbox's
  classpath, not visible to plugins.

**Bundle-level rejection (host-side validation at publish time)**:

The published DEX must NOT declare classes in any of these
prefixes, OR `IllegalStateException(FORBIDDEN_PACKAGE_PREFIX)`
on load. This is anti-spoofing — a plugin shipping its own
`app.syncler.plugin.runtime.PluginContext` would shadow the
real one for callers within its own DEX.

```kotlin
val FORBIDDEN_PACKAGE_PREFIXES = setOf(
    "app.syncler.",
    "android.",
    "androidx.",
    "kotlin.",
    "kotlinx.",
    "java.",
    "javax.",
)
```

The host scans the DEX header for declared class names at
staging time. Cheap (dex header is structured); rejection
fires `forbidden_package_prefix` LoadFailureCode.

### Block 5: misc carry-overs from v3 (unchanged)

- `bindIsolatedService` for per-plugin process isolation.
- DEX over `ParcelFileDescriptor`; sandbox reads bytes into
  `InMemoryDexClassLoader`.
- `entry_class` + `native_sdk_abi` first-class
  `PluginLoadParcel` fields.
- entryClass binary-class-name regex.
- ABI mismatch fails before code load.
- Bounded per-plugin dispatcher.
- Bridge call attribution by host-allocated `sandboxToken`,
  never plugin-supplied identifiers.
- Trust model: isolated UID = real sandbox; ClassLoader =
  organizational only; documented.
- minSdk=29 for native plugins; JS plugins keep minSdk=26.

## Open question for v4

A. **Sandbox-side forbidden-prefix check too**: should we
   ALSO check at the sandbox (after the host's pre-flight
   rejection)? Defense in depth — if the host bug ships a
   forbidden-prefix DEX, the sandbox rejects. Recommend
   yes. The sandbox already has access to the dex bytes;
   one more scan is cheap.

B. **Reflection within the plugin's own DEX**: a plugin
   class can `Class.forName("kotlinx.coroutines.Dispatchers")`
   and call .Default. That's allowed — coroutine
   dispatchers are part of the parent SDK surface.
   Acceptable.

C. **What about plugins calling
   `kotlinx.coroutines.GlobalScope.launch { ... }`?**
   GlobalScope is global to the isolated process. Plugin's
   coroutines outlive its sandbox token's lifetime by
   design. We ASSUME the plugin won't do that, and the
   process-death-on-unload pattern (each native plugin
   gets its own process; process dies on unbind) makes
   leaked coroutines bounded anyway. Document but don't
   enforce.
