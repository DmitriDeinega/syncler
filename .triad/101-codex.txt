**Blocking issue**

§10.8 still has a concurrency hole around state mutation. The intro/delta says `pairing-create / pairing-delete / state-push` routes lock the user row, but the actual MUST only covers pairing create/delete:

> `POST /v1/pairing/complete`, `DELETE /v1/pairings/{id}`

It does not cover:

- `PUT /v1/state`
- `PUT /v1/pairings/{pairing_id}/state`

That matters because §10.8 separates “CAS verify” from “apply writes”. A concurrent pre-rotation state push can pass `key_generation_observed == 1`, then race with rotation’s CAS checks and writes. Depending on implementation order, this can produce either a lost update or a blob written under the old MK after rotation has logically begun.

Fix: make every `key_generation`-tagged state-mutating endpoint lock the user row before checking `key_generation_observed` and before mutating blob rows. That should be stated in §10.5 or §10.8 as a global MUST, not only under pairing-set mutation.

Suggested wording:

```text
Every endpoint that mutates a key_generation-tagged blob MUST
SELECT users.* FROM users WHERE id = :user_id FOR UPDATE before
checking key_generation_observed or applying CAS writes. This
includes PUT /v1/state, PUT /v1/pairings/{pairing_id}/state,
POST /v1/account/rotate-master-key, and any future state-mutating
endpoint.
```

Also tighten §10.8 steps 8-10 so CAS is implemented atomically or under row locks, e.g. `SELECT ... FOR UPDATE` on each blob row before CAS verification, or conditional `UPDATE ... WHERE state_version = :observed AND key_generation = :observed_generation` with affected-row checks.

**Non-blocking / YELLOW**

§10.5 is conceptually right, but currently only requires `key_generation_observed`; it does not require serialization with rotation. The missing lock belongs there too.

§10.8 step 3 verifies `users.auth_key_hash` before step 5 selects the user row. The spec should either move `SELECT users.* ... FOR UPDATE` before proof verification or explicitly say the auth hash is read from a locked user row. This is mostly clarity, but it avoids implementation drift.

**Anything new**

No new cryptographic objection to §10.7. The replay boundary is now honest: the proof is a TLS bearer credential, not challenge-bound. The downgrade-defense fix in §10.10 is also the right shape.

Current verdict: **RED**, narrowly, because the state-push locking requirement claimed in the v3 delta is not actually present in the normative text. Once that is added, I would move to GREEN.
