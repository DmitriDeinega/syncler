**Verdict: GREEN**

Phase 9a v2 effectively closes the architectural gaps identified in the previous round. The transition to a "Full Fanout" model is the correct pragmatic choice; it preserves signature integrity and implementation simplicity at a negligible bandwidth cost. The explicit trust model and cryptographic binding (HPKE `info`/`aad`) align with industry standards for this stage of development.

### 1. Analysis of v2 Improvements
*   **Signatures:** Sorting `recipient_envelopes` by `device_id` before signing ensures deterministic verification across all recipients. This elegantly bypasses the V1 Merkle complexity.
*   **Cryptographic Binding:** Including `payload_ciphertext_sha256` in the HPKE `info` block is a high-signal security improvement, preventing "Frankenstein" message attacks (reusing a valid CEK wrap with a different payload).
*   **Schema Safety:** Using `NULL` instead of a zero-sentinel prevents "silent failure" modes where messages might be encrypted to a non-existent key.
*   **Trust Model:** Documenting the "Trusted Directory" assumption is vital. It correctly identifies that while Phase 9 protects against passive observers and cross-device leaks, protection against an active server remains a future milestone (Attestation).

### 2. Missing Items / Refinements
*   **Standardized Error Codes:** The server should return a specific error (e.g., `422 UNPROCESSABLE_ENTITY` with a code like `RECIPIENT_KEY_MISSING`) if a publish is attempted while an active device lacks an X25519 key. This allows the client to trigger a "Wait for device sync" UI state.
*   **Retry Logic:** Senders should implement a single automatic retry if a publish fails due to a `STALE_RECIPIENT_SET` error (which the server should return if the `device_id`s in the request don't match the current active set).
*   **Sorting Specificity:** Define "lexicographic" as sorting by the standard string representation of the UUID to ensure consistent implementation across languages (Python vs. Kotlin/Swift).

### 3. Answers to Open Questions

**A. Sender's recipient-set knowledge / Refresh cadence:**
Refresh should be **event-driven + lazy**. Senders should refresh the device list upon receiving an SSE `devices.changed` event. Additionally, they should perform a "lazy refresh" if a publish attempt fails with a recipient-related error. Avoid refreshing on every publish to save battery/bandwidth.

**B. Recipient cap = 32:**
**32 is sufficient.** It comfortably covers power users (e.g., phone, tablet, laptop, desktop, and several dev/test instances). Keeping the cap low ensures the envelope array remains small (~2.5 KB), preventing bloated inbox responses.

**C. Live-card delete propagation:**
**Server-side soft-delete is sufficient.** A deletion is a metadata change, not a secret-bearing payload. As long as the `DELETE` request is authenticated by a valid device JWT, there is no need for an encryption envelope. The server simply stops serving the card in the projected inbox.

**D. SSE event shape:**
Keep it minimal to avoid leakage.
`{"type": "devices.changed", "user_id": "<uuid>"}`
This tells the client *something* changed regarding the account's device topology, prompting it to re-fetch the public directory from `/v1/devices`.
