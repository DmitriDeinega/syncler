I have completed the alignment audit of the M11 development phase and updated the `docs/integration-guide.md` to reflect the current platform state.

### Part 1: M11 Alignment Audit

The following table recaps the audit of the M11.0 through M11.7 commit ladder against the original agreements in consultations 35, 36, and reviews 37–44.

| Phase | Deliverable | Status | Justification |
| :--- | :--- | :--- | :--- |
| **M11.1** | `hostPreview` + Preview/Detail split | **GREEN** | Contract exactly matches the "cut" version (Title, Subtitle, Summary, SearchText). No drift into priority/category. |
| **M11.2** | Read state + M7 CAS sync | **GREEN** | Syncs via encrypted user-state blob. Triggered on detail-open only (confirmed as the intentional decision in Review 38). |
| **M11.3** | Bottom nav + Date buckets + Archive | **GREEN** | Nav (Inbox/Senders/Settings) and date buckets match spec. Archive honestly documented as "hide-from-active" for V1. |
| **M11.4** | Bundle-by-hash + Revocation reasons | **GREEN** | `/by-id` lookup ensures historical correctness. `compromised` reason triggers the requested hard E-stop in the UI. |
| **M11.5** | Host-side metadata search | **GREEN** | Covers Title, Subtitle, Summary, Sender, and `searchText` tokens. Case-insensitive substring match verified. |
| **M11.6** | Relative timestamps + Delete | **GREEN** | Cross-device delete (via CAS) and relative formatting implemented. Settings screen filters revoked devices. |
| **M11.7** | Multi-select + Group-by-sender | **GREEN** | Group-by-sender was added as an **optional toggle**, reconciling Consult 35's concerns with user feedback. |

**Pressure-Test Resolutions:**
1.  **Group-by-sender:** This was a planned drift to accommodate user feedback. By making it a toggle rather than the primary navigation, we preserved the time-sensitive chronological feed while allowing structured access.
2.  **Bundle-by-hash:** The V1 state is compatible. While disk persistence is deferred to V1.5, the implementation uses `/by-id` to ensure that messages always resolve to their original validated bundles, even after app restarts.
3.  **Delete vs Dismiss vs Archive:** The "trinary" is honored. Archive and Delete are synced UI actions; Dismiss remains a server-side endpoint for future sender-feedback loops.
