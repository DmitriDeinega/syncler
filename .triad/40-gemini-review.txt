In Review 40 (Phase 4), we see a solid foundation for bundle-by-hash stability and classified revocation, though the "V1 scope cuts" have left some significant functional gaps that make the new fields "dead weight" in the current build.

### 1. Bundle-by-hash: The "Restart Drift" Gap
The core goal of bundle-by-hash (Codex M11.3) was to prevent "history drift" when a plugin author publishes a breaking change. While you've added the `bundleHash` field and an in-memory `Map`, the current implementation still drifts on every app restart:
*   **The Problem:** On restart, `bundleByHash` is empty. `InboxRepository.refresh()` sees an old message, calls `fetchPluginOrNull`, which calls `/latest`. If the sender has published v2, the old message is assigned v2's bundle and hash.
*   **Verdict:** If V1.5 (disk persistence) is strictly required for this to work, then Phase 4 is technically "complete" as a data-layer update. However, to make this useful in V1, `SynclerApi.inbox` should return the `bundle_hash` the sender intended for that message (or at least the `bundle_hash` that was latest at the time of send). Without that, or local persistence, the "correctness gap" remains open across every process death.

### 2. Revocation Classification: The "Silent Failure" Gap
The server now records `revocation_reason`, but the device has **no way to discover it**.
*   **The Problem:** `get_latest_for_plugin` filters `revoked_at IS NULL`. When a plugin is revoked, the device hits `/latest`, receives a `404 Not Found`, and `InboxRepository` logs a warning and shows the generic "Plugin not loaded" fallback.
*   **The Impact:** The differentiated UX (Security Alert for `compromised` vs. silent for `superseded`) is impossible in the current build. The device can't distinguish between a network/404 error and a security revocation.
*   **Recommendation:** The `/latest` endpoint should return the revoked status and reason (e.g., `200 OK` with `{"revoked": true, "reason": "compromised", ...}` or a specific `403/451` error body) rather than a generic `404`.

### 3. Signature & Idempotency (Correctness)
*   **Shape Ambiguity:** Your `_revoke_envelope` correctly handles the optional reason. By omitting the key when `None`, you maintain a 2-field shape that matches M8.1 signers. `{"reason": null}` is correctly avoided.
*   **Idempotency:** The "promote reason" logic in `revoke_plugin_row` (L144-155) is correct. It allows a sender to escalate a `superseded` row to `compromised` later, which is a vital security requirement.
*   **Hash Mismatch vs. Replay:** The check `actualHash == latest.bundleHash` is robust. A "replay" of an old bundle at a new signed-URL is impossible due to pre-image resistance. A "forced downgrade" (making `/latest` return an old version) is only possible if the sender revokes all newer versions, which is a legitimate (if heavy-handed) administrative action.

### 4. Server Tests
The two new tests (`test_revoke_records_reason_when_provided` and `test_revoke_rejects_invalid_reason`) correctly cover the happy path and the Pydantic enum validation. I recommend adding a third test:
*   **`test_revoke_maintains_earliest_revoked_at`**: Verify that re-revoking to update the `reason` does *not* push the `revoked_at` timestamp forward (it should stay pinned to the first revocation).

### Summary of Pressure-Tests
*   **V1 Scope:** The "dead weight" status of `bundleHash` and `reason` is acceptable **only if** you are comfortable with "Silent Failure" UX for revocations in the V1 release.
*   **Hold Phase 5?** **No**, but I recommend a small fix-up to the `/latest` route (returning revocation info instead of 404) before we finalize the contract for the host-side search.

**Proceed to Phase 5 (host-side metadata search), but consider the `/latest` 404 behavior a "debt" item for the V1 polish phase.**
