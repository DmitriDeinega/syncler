=== REACTION ===
The corrected model is coherent, but it is a different and harder product: not “rich cards platform,” but “encrypted plugin runtime plus distribution trust layer.” The biggest thing that bothers me is that “platform has no opinion” conflicts with marketplace review: if we approve plugins, we own some runtime contract, permissions model, update policy, and abuse surface. I also think counting plugins outside the 80-hour budget is misleading if the lottery plugin is part of the acceptance test.

=== BATCH 3 QUESTIONS ===
1. Is the marketplace required for V1, or can V1 use manual plugin install by URL/file with a hardcoded “trusted developer mode”? Marketplace in V1 may consume half the budget by itself.

2. Who reviews plugins on day one: you manually, the triad, automated static checks, or no one until V1.5? “Marketplace review” is a real operating process, not just a screen.

3. Pick the Android stack now: native Kotlin WebView host, Capacitor/Ionic, React Native, or Flutter. Which one are you willing to debug when push, background execution, WebView storage, and crypto get weird?

4. What powers does a plugin get in V1: render UI, call declared network endpoints, store local data, receive camera/photo picker access, read files, use location, or run background code? Every “yes” is part of the permanent Plugin SDK contract.

5. Are plugin network calls restricted to declared domains/endpoints, or can installed plugin JS call any URL? If unrestricted, marketplace review becomes much more security-critical.

6. Describe the pairing ceremony exactly: does the phone scan a sender QR, does the sender scan a phone QR, does the user approve a one-time request, and where do E2E public keys and signing keys get exchanged?

7. How do plugin updates work: auto-update after marketplace approval, user manually approves every version, or pinned versions per sender? What happens if LotteryPlugin v1.1 changes the payload format while old messages are still on the phone?

8. Where does the platform server run for V1: AWS, Fly.io, Render, Railway, VPS, or your existing infra? Pick one based on push reliability, secrets management, logs, backups, and how quickly we can deploy.
