## Concerns I want a second opinion on

1. **WeakReference-to-Activity coordinator design.** Standard
   Android pattern but it has known gotchas around config
   change (Activity recreated â†’ weak ref nulled mid-await),
   process death (host process dies â†’ entire continuation map
   nuked), and re-entrancy (Activity launches a capability,
   gets paused, plugin sees no_foreground_activity). What's the
   right answer? Alternatives I considered:
   - (a) Use a foregroundService to mediate. Heavyweight, but
     survives Activity recreation.
   - (b) ActivityResultRegistry at the Application level (API
     30+). Lighter but still bound to whichever Activity
     last registered.
   - (c) Refuse capability calls when no Activity is
     foregrounded; let the plugin retry. Simplest; UX is
     "open the Syncler app, your plugin's request will run."

2. **Grant prompt rate-limit.** Spec says "at most one prompt
   every 60 seconds for the same (plugin, capability) combo."
   The window is per (device, plugin_row, capability); a denied
   prompt sets a cooldown. Is 60s right? Should the cooldown be
   per-Activity-foreground-session (i.e. zero-out when the user
   re-opens the app)? What about clear-once-on-foreground vs.
   leaky-bucket?

3. **OS permission + grant prompt sequencing.** Camera/location
   need BOTH a per-grant in-app prompt AND a runtime OS
   permission. Two prompts feels redundant â€” the in-app one
   could pre-explain why the OS prompt is about to fire. But:
   - Plugin already declared `camera` capability in manifest â†’
     user agreed at install/pair time.
   - Per-invocation in-app prompt explains "Plugin X wants to
     take a photo right now."
   - OS prompt is system-level "Allow Syncler to access camera?"
   The user might say yes to the in-app prompt and no to the
   OS prompt; we surface `os_denied`. Is the two-prompt sequence
   correct? Or should declared-capability + OS-grant be enough,
   skipping the per-invocation in-app prompt for capabilities
   the OS already gates?

4. **Photo Picker vs. SAF for gallery.** Photo Picker is API 33+
   but Google ships a backport in androidx for 26+ via
   `PickVisualMedia` ActivityResultContract. Should we use the
   backport unconditionally, or branch on SDK_INT? The backport
   is well-tested and means one code path.

5. **Result byte transfer size.** Capabilities return bytes
   directly via `bridgeCall` result JSON (base64-encoded). A
   2MB image base64-encodes to ~2.7MB. Binder transactions
   over 1MB are unreliable. Should we instead:
   - (a) Return a content:// URI the sandbox can `ContentResolver`-
     open. But the isolated UID can't read content URIs without
     explicit grant flags per Intent.
   - (b) Stage to a host-owned file, return its path + a
     short-lived host-mediated FD. Sandbox reads via the host's
     PluginContext.fileBytes(handle).
   - (c) Hard size cap (1 MB) on results; plugins handle
     compression themselves.
   What's the right shape?

6. **CameraBridge tempfile lifecycle.** ACTION_IMAGE_CAPTURE
   writes to a FileProvider URI we give it. We then read bytes
   and delete. If the host process is killed between capture
   and read, the tempfile leaks. Periodic cleanup of stale
   `plugin-camera-*` files in `cacheDir` is the obvious answer.
   Is there a more correct approach?

7. **Location capability granularity.** Spec lumps
   `ACCESS_COARSE_LOCATION` and `ACCESS_FINE_LOCATION` under
   one `location` capability. Should the manifest declare
   them separately (`location.fine` vs `location.coarse`)?
   Plugins that only need city-level shouldn't be triggering
   the fine-grained OS prompt.

8. **Audit log entry per invocation vs. per grant.** Spec says
   "every invocation logs (plugin_row_id, capability,
   sandbox_token, outcome)." That's potentially high-volume
   for a plugin that polls location every 30s. Should the
   audit be sample-based (1 in N) or aggregated (one row per
   plugin per hour)? Or is per-invocation correct and we
   manage volume via retention policy?

9. **Settings-sheet revoke UX.** Phase 12 step 11. Should the
   list be per-plugin (settings â†’ plugin X â†’ capabilities)
   or per-capability (settings â†’ camera â†’ which plugins)?
   Per-plugin is more discoverable for "what does this plugin
   have access to?"; per-capability is better for "who's using
   my location?"

10. **Manifest capability validation gap.** Currently the
    server's `PluginPublishRequest.capabilities` validates
    against a Literal["camera", "gallery", ...]. Phase 12 adds
    `location.fine` / `location.coarse` if we go with #7's
    split. Do we bump a manifest version? Add a new
    capability `location` that maps to fine-only for backwards
    compat? Reject mixed manifests?

## Per-grant prompt model

Capability grants live in a new Room table `plugin_capability_grants`:

```sql
CREATE TABLE plugin_capability_grants (
  plugin_row_id  TEXT NOT NULL,
  capability     TEXT NOT NULL,
  granted_at_ms  INTEGER NOT NULL,
  PRIMARY KEY (plugin_row_id, capability)
);
```

The bridge's grant check becomes:
1. Manifest declares the capability? (existing check)
2. Per-plugin grant row exists?
3. If not: show the per-grant prompt; insert row on accept.
4. OS permission state allows? (camera/location only)
5. If not: launch the runtime permission request.

Revoking a grant in the settings sheet deletes the row; next
invocation re-prompts.

## Activity-result plumbing

Camera/gallery/file all need an `Activity` to launch an Intent
and await its result. The plugin sandbox runs in `:plugin` (JS)
or `:nativePlugin` (DEX) â€š?" neither has an Activity. The host
process owns the foreground Activity, so the bridge call routes
to host code which:

1. Records a `pending_capability_call(call_id â€š+' continuation)`.
2. Calls into a singleton `CapabilityActivityCoordinator` in the
   host process that holds a weak reference to the currently
   foregrounded Activity.
3. If no Activity is foregrounded, returns
   `{"error":"no_foreground_activity"}` immediately. Plugin can
   retry when the user returns to the app.
4. Activity launches the intent via the ActivityResultLauncher
   API; the registered callback resolves the continuation.

The coordinator handles process death across the await: if the
host process dies mid-await, the sandbox sees `onPluginCrashed`
and the plugin coroutine is cancelled; no orphan continuations.

**Overall Verdict**

Do not greenlight the spec as-is. The permission model is mostly sound, but the Activity-result and byte-transfer parts need design changes before implementation starts. The highest-risk items are Binder payload size, continuation lifetime across Activity recreation, and location granularity.

**1. WeakReference Activity Coordinator: DESIGN**

The weak-reference singleton is acceptable only as a thin â€ścurrent UI ownerâ€ť pointer, not as the owner of the whole operation.

Recommendation:

- Keep `no_foreground_activity` for calls made while the app has no resumed Activity.
- Move pending operation ownership into a host-process scoped manager, not the Activity.
- Register Activity result launchers from the current Activity, but make operation state explicit: `call_id`, plugin identity, capability, launched state, timeout, continuation.
- Treat config change as recoverable until the OS result is delivered or timeout expires.
- Treat host process death as cancellation. That is fine for v0.1.

I would not use a foreground service for camera/gallery/file. It does not solve the UI launcher problem cleanly and creates the wrong mental model. Application-level `ActivityResultRegistry` is also not a real escape hatch; launch still depends on a lifecycle owner.

The spec should explicitly say: â€śActivity recreation must not cause an already-launched call to complete as `no_foreground_activity`.â€ť

**2. Grant Prompt Rate Limit: NIT**

Sixty seconds is defensible for v0.1, but define it as a denial cooldown, not a generic prompt cooldown.

Recommendation:

- Rate-limit only after denial, dismissal, or OS denial.
- Key by `(device, plugin_row_id, capability)`.
- Keep the cooldown wall-clock based, not per Activity session.
- Do not clear automatically on foreground; that lets a malicious plugin spam by causing navigation loops or relying on user reopen behavior.
- Return a distinct error like `capability_prompt_rate_limited` if the SDK can expose it cleanly; otherwise `capability_denied` is acceptable but less debuggable.

A leaky bucket is overkill right now. Add it later if plugins start abusing prompts across many capabilities.

**3. OS Permission + Grant Prompt Sequencing: OK, With One Change**

Two prompts are correct. They answer different questions:

- In-app prompt: â€śShould this plugin do this now?â€ť
- OS prompt: â€śShould Syncler as an app have this device permission?â€ť

Do not collapse per-invocation plugin consent into OS permission. Android grants permission to Syncler, not to a plugin.

Recommendation:

- For first invocation: show the in-app plugin prompt first, then OS permission if needed.
- If OS permission is already granted, still require the plugin grant.
- If the user accepts the in-app prompt but denies OS permission, return `os_denied`.
- Consider inserting the plugin grant only after the OS permission succeeds, or record a separate â€śplugin grant acceptedâ€ť state. Otherwise settings may show a plugin as granted even though the capability cannot work.

I prefer â€śinsert after full successâ€ť for v0.1 unless you specifically want plugin grant and OS permission state to be independently visible.

**4. Photo Picker vs SAF For Gallery: NIT**

Use the AndroidX `PickVisualMedia` / `PickMultipleVisualMedia` path as the primary API unconditionally where available.

Recommendation:

- Prefer the AndroidX Activity Result contract.
- Let AndroidX handle framework picker vs backport behavior.
- Keep `ACTION_OPEN_DOCUMENT` only as an explicit fallback if the contract reports unavailable or project dependencies make the backport unavailable.
- Document whether gallery supports single or multiple selection and enforce a max item count.

One code path is better here, but the spec should avoid saying â€śAPI 33+ / ACTION_OPEN_DOCUMENT fallbackâ€ť if the intended implementation is AndroidX picker across supported API levels.

**5. Result Byte Transfer Size: DESIGN**

Do not return media/file bytes directly through `bridgeCall` JSON. This is the biggest design issue.

Base64 over Binder is the wrong transport for camera, gallery, and file. A realistic image can exceed Binder limits, and JSON/base64 multiplies memory pressure in host, sandbox, and JS/native layers.

Recommendation:

- Return handles, not bytes, for large binary results.
- Host stages selected content into a host-owned cache file or blob table.
- Bridge response returns metadata plus opaque handles:
  - `handle`
  - `name`
  - `mime`
  - `sizeBytes`
  - maybe `expiresAtMs`
- Add host-mediated read APIs:
  - `platform.fileBytes(handle, offset?, length?)`
  - or native equivalent returning chunks / FD through the existing host bridge.
- Enforce max staged size and max lifetime.
- Delete handles after expiry, plugin unload, explicit close, or cache cleanup.

Option `(b)` is the right shape. Option `(a)` is fragile across isolated UIDs and content URI permission grants. Option `(c)` alone makes the feature too weak and pushes compression work to plugins that cannot safely access the original stream.

For v0.1, a simple handle plus chunked byte reads is enough. Do not try to pass raw file paths into isolated sandboxes unless you are certain the sandbox UID can read them and the path cannot be abused.

**6. Camera Tempfile Lifecycle: OK**

Tempfile plus FileProvider URI is normal. Stale-file cleanup is the correct answer.

Recommendation:

- Write camera temp files under a dedicated cache subdirectory, not the root cache dir.
- Use names scoped by call id: `plugin-camera-{callId}.jpg`.
- Delete on success, cancel, failure, timeout, and bridge cancellation.
- Run cleanup on app start and before launching a new camera call.
- Remove files older than a conservative TTL, for example 24 hours.

There is no need for something more complex unless you need durable recovery after process death, which v0.1 explicitly does not.

**7. Location Capability Granularity: FIX**

Split location. A single `location` capability is too coarse.

Recommendation:

- Add `location.coarse` and `location.fine`.
- `location.coarse` requests approximate location only.
- `location.fine` allows precise location request, subject to Androidâ€™s approximate/precise user choice.
- If a plugin declares only coarse, never request fine.
- If a plugin declares fine, it may also receive coarse if the OS grants approximate only; return accuracy and maybe a `precision` field.

The bridge API can remain `locationCurrent(opts)`, but `opts.accuracy` should be validated against manifest capability:

- `accuracy: "coarse"` requires `location.coarse` or `location.fine`.
- `accuracy: "fine"` requires `location.fine`.

This is worth changing before implementation because it affects manifest validation, grants table keys, prompts, settings, and audit semantics.

**8. Audit Per Invocation vs Per Grant: OK**

Per-invocation audit is correct for security-sensitive capability use. Do not sample.

Recommendation:

- Keep one row per invocation.
- Do not log coordinates or file names unless there is a deliberate privacy decision.
- Add retention policy and perhaps aggregation later for display.
- For noisy capabilities like location, enforce call-side throttling separately if needed.

Sampling undermines forensic value. Aggregation is fine for UI summaries, but the source audit table should stay event-based.

**9. Settings Revoke UX: NIT**

Start per-plugin, add per-capability later if needed.

Recommendation:

- Phase 12: settings â†’ plugin â†’ capabilities â†’ revoke.
- Include last-used timestamp and current granted/denied state if available.
- Longer term: add a privacy overview grouped by capability, especially location.

Per-plugin is the right v0.1 surface because grants are plugin-scoped and users are likely managing trust in a plugin.

**10. Manifest Capability Validation Gap: FIX**

Yes, bump or version the capability schema if you split location.

Recommendation:

- Add `location.coarse` and `location.fine` to the allowed capability literal set.
- Treat old `location` as legacy only if there are already published manifests using it.
- If this is not shipped yet, reject `location` and require explicit granularity.
- If backwards compatibility is needed, map old `location` to `location.fine` only with a migration note, because it is the more conservative interpretation from a privacy/permission standpoint.
- Reject manifests declaring both `location` and split capabilities.
- Store exact declared capability strings; avoid hidden expansion except at validation/migration boundaries.

If this is still pre-release, make the breaking change now.

**Missing Spec Items**

The spec needs a binary result transport section before implementation. Define handle lifecycle, max file size, max gallery item count, chunk size, expiry, cleanup, and cancellation behavior.

Add a continuation lifecycle state machine. At minimum: `created`, `waiting_for_grant`, `waiting_for_os_permission`, `launched`, `completed`, `cancelled`, `timed_out`. This will make Activity recreation and cancellation testable.

Add explicit timeout policy for Activity-result calls and location calls. Without timeout, pending continuations can leak until process death.

Add concurrency rules. Decide whether a plugin can have multiple simultaneous capability calls, and whether the host allows only one global Activity-result operation at a time. I would start with one active Activity-result capability operation globally, returning `capability_busy` for overlapping calls.

Add cancellation semantics. If the sandbox call is cancelled, the host should mark the pending call cancelled and ignore late Activity results.

Add prompt copy requirements at the policy level: plugin name, capability, immediate action, and whether Syncler/Android permission may be requested next.

Add OS permission degradation behavior for Android location approximate vs precise. `ACCESS_FINE_LOCATION` does not guarantee precise coordinates if the user chooses approximate.

Add MIME/type validation. For gallery/file/camera, do not trust returned MIME from providers alone. Record provider MIME, but sniff or constrain where practical.

Add SDK error taxonomy. Current errors should include at least: `capability_not_declared`, `capability_denied`, `capability_prompt_rate_limited`, `os_denied`, `no_foreground_activity`, `capability_busy`, `cancelled`, `timeout`, `result_too_large`, `io_error`.

**Greenlight Condition**

I would greenlight after these changes:

1. Replace JSON/base64 bytes with staged opaque handles.
2. Split location into `location.coarse` / `location.fine`.
3. Define Activity-result continuation lifecycle across config change, cancellation, timeout, and concurrency.
4. Clarify grant insertion timing around OS permission success.
5. Use AndroidX Photo Picker as the primary gallery path.

- Consider inserting the plugin grant only after the OS permission succeeds, or record a separate â€śplugin grant acceptedâ€ť state. Otherwise settings may show a plugin as granted even though the capability cannot work.

I prefer â€śinsert after full successâ€ť for v0.1 unless you specifically want plugin grant and OS permission state to be independently visible.

**4. Photo Picker vs SAF For Gallery: NIT**

Use the AndroidX `PickVisualMedia` / `PickMultipleVisualMedia` path as the primary API unconditionally where available.

Recommendation:

- Prefer the AndroidX Activity Result contract.
- Let AndroidX handle framework picker vs backport behavior.
- Keep `ACTION_OPEN_DOCUMENT` only as an explicit fallback if the contract reports unavailable or project dependencies make the backport unavailable.
- Document whether gallery supports single or multiple selection and enforce a max item count.

One code path is better here, but the spec should avoid saying â€śAPI 33+ / ACTION_OPEN_DOCUMENT fallbackâ€ť if the intended implementation is AndroidX picker across supported API levels.

**5. Result Byte Transfer Size: DESIGN**

Do not return media/file bytes directly through `bridgeCall` JSON. This is the biggest design issue.

Base64 over Binder is the wrong transport for camera, gallery, and file. A realistic image can exceed Binder limits, and JSON/base64 multiplies memory pressure in host, sandbox, and JS/native layers.

Recommendation:

- Return handles, not bytes, for large binary results.
- Host stages selected content into a host-owned cache file or blob table.
- Bridge response returns metadata plus opaque handles:
  - `handle`
  - `name`
  - `mime`
  - `sizeBytes`
  - maybe `expiresAtMs`
- Add host-mediated read APIs:
  - `platform.fileBytes(handle, offset?, length?)`
  - or native equivalent returning chunks / FD through the existing host bridge.
- Enforce max staged size and max lifetime.
- Delete handles after expiry, plugin unload, explicit close, or cache cleanup.

Option `(b)` is the right shape. Option `(a)` is fragile across isolated UIDs and content URI permission grants. Option `(c)` alone makes the feature too weak and pushes compression work to plugins that cannot safely access the original stream.

For v0.1, a simple handle plus chunked byte reads is enough. Do not try to pass raw file paths into isolated sandboxes unless you are certain the sandbox UID can read them and the path cannot be abused.

**6. Camera Tempfile Lifecycle: OK**

Tempfile plus FileProvider URI is normal. Stale-file cleanup is the correct answer.

Recommendation:

- Write camera temp files under a dedicated cache subdirectory, not the root cache dir.
- Use names scoped by call id: `plugin-camera-{callId}.jpg`.
- Delete on success, cancel, failure, timeout, and bridge cancellation.
- Run cleanup on app start and before launching a new camera call.
- Remove files older than a conservative TTL, for example 24 hours.

There is no need for something more complex unless you need durable recovery after process death, which v0.1 explicitly does not.

**7. Location Capability Granularity: FIX**

Split location. A single `location` capability is too coarse.

Recommendation:

- Add `location.coarse` and `location.fine`.
- `location.coarse` requests approximate location only.
- `location.fine` allows precise location request, subject to Androidâ€™s approximate/precise user choice.
- If a plugin declares only coarse, never request fine.
- If a plugin declares fine, it may also receive coarse if the OS grants approximate only; return accuracy and maybe a `precision` field.

The bridge API can remain `locationCurrent(opts)`, but `opts.accuracy` should be validated against manifest capability:

- `accuracy: "coarse"` requires `location.coarse` or `location.fine`.
- `accuracy: "fine"` requires `location.fine`.

This is worth changing before implementation because it affects manifest validation, grants table keys, prompts, settings, and audit semantics.

**8. Audit Per Invocation vs Per Grant: OK**

Per-invocation audit is correct for security-sensitive capability use. Do not sample.

Recommendation:

- Keep one row per invocation.
- Do not log coordinates or file names unless there is a deliberate privacy decision.
- Add retention policy and perhaps aggregation later for display.
- For noisy capabilities like location, enforce call-side throttling separately if needed.

Sampling undermines forensic value. Aggregation is fine for UI summaries, but the source audit table should stay event-based.

**9. Settings Revoke UX: NIT**

Start per-plugin, add per-capability later if needed.

Recommendation:

- Phase 12: settings â†’ plugin â†’ capabilities â†’ revoke.
- Include last-used timestamp and current granted/denied state if available.
- Longer term: add a privacy overview grouped by capability, especially location.

Per-plugin is the right v0.1 surface because grants are plugin-scoped and users are likely managing trust in a plugin.

**10. Manifest Capability Validation Gap: FIX**

Yes, bump or version the capability schema if you split location.

Recommendation:

- Add `location.coarse` and `location.fine` to the allowed capability literal set.
- Treat old `location` as legacy only if there are already published manifests using it.
- If this is not shipped yet, reject `location` and require explicit granularity.
- If backwards compatibility is needed, map old `location` to `location.fine` only with a migration note, because it is the more conservative interpretation from a privacy/permission standpoint.
- Reject manifests declaring both `location` and split capabilities.
- Store exact declared capability strings; avoid hidden expansion except at validation/migration boundaries.

If this is still pre-release, make the breaking change now.

**Missing Spec Items**

The spec needs a binary result transport section before implementation. Define handle lifecycle, max file size, max gallery item count, chunk size, expiry, cleanup, and cancellation behavior.

Add a continuation lifecycle state machine. At minimum: `created`, `waiting_for_grant`, `waiting_for_os_permission`, `launched`, `completed`, `cancelled`, `timed_out`. This will make Activity recreation and cancellation testable.

Add explicit timeout policy for Activity-result calls and location calls. Without timeout, pending continuations can leak until process death.

Add concurrency rules. Decide whether a plugin can have multiple simultaneous capability calls, and whether the host allows only one global Activity-result operation at a time. I would start with one active Activity-result capability operation globally, returning `capability_busy` for overlapping calls.

Add cancellation semantics. If the sandbox call is cancelled, the host should mark the pending call cancelled and ignore late Activity results.

Add prompt copy requirements at the policy level: plugin name, capability, immediate action, and whether Syncler/Android permission may be requested next.

Add OS permission degradation behavior for Android location approximate vs precise. `ACCESS_FINE_LOCATION` does not guarantee precise coordinates if the user chooses approximate.

Add MIME/type validation. For gallery/file/camera, do not trust returned MIME from providers alone. Record provider MIME, but sniff or constrain where practical.

Add SDK error taxonomy. Current errors should include at least: `capability_not_declared`, `capability_denied`, `capability_prompt_rate_limited`, `os_denied`, `no_foreground_activity`, `capability_busy`, `cancelled`, `timeout`, `result_too_large`, `io_error`.

**Greenlight Condition**

I would greenlight after these changes:

1. Replace JSON/base64 bytes with staged opaque handles.
2. Split location into `location.coarse` / `location.fine`.
3. Define Activity-result continuation lifecycle across config change, cancellation, timeout, and concurrency.
4. Clarify grant insertion timing around OS permission success.
5. Use AndroidX Photo Picker as the primary gallery path.
