## Specifically pressure-test

1. **Token uniqueness** — `pending_pairings.pairing_token` has UNIQUE constraint, but generate_pairing_token uses 32 random bytes via secrets.token_bytes — collision risk effectively zero, but is there any race in initiate where two concurrent requests might produce the same token? (None expected since INSERT will fail on UNIQUE conflict.)

2. **TTL handling** — `complete_pairing` checks `pending.expires_at <= now`. Timezone awareness same as M5.2 — is `expires_at` always stored timezone-aware? (Migration uses `TIMESTAMPTZ`, model uses `DateTime(timezone=True)` — should be fine.)

3. **Anti-spoofing layers** — fingerprint format is base32(SHA-256(pk)[0:8]) in 4-char hyphen groups. Is 64 bits of identity material enough? (Birthday bound ~2^32, fine for "this is the same sender I paired with" not "global hash collision resistance".)

4. **Initiate envelope** — sender signs `{sender_id, ttl_seconds, metadata}`. Could a different sender's signature be replayed against the same envelope? `get_active_sender(payload.sender_id)` returns the named sender's public key for verification, so signature must match that sender's private key. ✓

5. **Encrypted initial state** — Android sends a placeholder bytes "syncler-pairing-bootstrap-v1". Acceptable for V1 (M7 wires the real pairing-key-derivation handshake). Confirm this isn't a real security issue today.

6. **Race: pairing already exists** — `complete_pairing` raises `PairingAlreadyExistsError` if a non-revoked pairing exists for (user, sender). Re-pairing requires user to revoke first. UX-wise reasonable.

7. **PairedSenderStore.encodeRecord uses `|` as separator** — same pattern as messages.encrypted_body_pointer. Base64 alphabet doesn't include `|`. ✓

8. **/initiate broker_url** — built from `request.base_url`. Could a malicious proxy forge the base URL via X-Forwarded-Host? Yes if proxy isn't trusted; same caveat as the client-IP detection in rate_limit.py. Document but not a V1 fix.

9. **/pairing/complete is JWT-authed but accepts any pairing_token** — meaning any logged-in user could try every token and complete pairings they shouldn't have access to. Mitigation: token is 32 random bytes; brute-force infeasible in TTL window (5 min default, 15 min max).

10. **Android URL parsing** — `parseBrokerUrl` accepts only https://. Good. Token is extracted from `?token=` query param. Verified no path traversal or non-URL injection.

=== AGREEMENTS WITH CONCERNS ===
1. Agree. Token collision is cryptographically negligible. Only concern: a UNIQUE collision would currently surface as an unhandled DB error, but this is not a practical V1 race.

2. Agree for Postgres. `TIMESTAMPTZ` + `DateTime(timezone=True)` + `datetime.now(UTC)` are aligned.

3. Agree on 64 bits for human confirmation. Concern: the comment/test wording is wrong: 8 digest bytes base32-encode to 13 chars, not “6 groups of 4”.

4. Agree on signature binding. The public key is selected from `payload.sender_id`, so replaying another sender’s signature against this sender ID fails.

5. Mostly agree. The placeholder is not itself a confidentiality break because the server treats it as opaque, but M7 must not assume existing `pairings.encrypted_state` is real encrypted pairing material.

6. Disagree with the current behavior. The service allows re-pair after revoke in logic, but the DB unique constraint prevents it. See finding below.

7. Disagree. Base64 does not include `|`, but `senderName` is stored raw and sender names are not forbidden from containing `|`.

8. Agree. `request.base_url` inherits proxy trust assumptions. Document now; fix with trusted proxy/root URL config before production.

9. Agree. Token brute force is infeasible, but `/complete` should still get an auth/IP rate limit for hygiene.

10. Mostly agree. It requires `https://` and extracts `token`, but it accepts any HTTPS host/path and the server emits raw standard Base64 in a query parameter.

=== ADDITIONAL FINDINGS ===
- CRITICAL `/pairing/initiate` rate limiter is broken/spoofable
  evidence: `server/app/routers/pairing.py:66`, `server/app/middleware/rate_limit.py:144`
  proposal: use an IP pre-limit first, then after signature verification set `request.state.sender_id` and call `check_rate_limit(... RATE_LIMITS["pairing_initiate"])`.

- CRITICAL Android fingerprint confirmation is not gating pairing
  evidence: `android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingScreen.kt:65`, `android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:54`, `android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:72`, `android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingScreen.kt:143`
  proposal: add a non-consuming preview/inspect endpoint, show fingerprint first, then complete+persist only after OK. At minimum, dismiss must revoke server pairing and remove local state.

- HIGH Same pairing token can be completed by concurrent users
  evidence: `server/app/services/pairing.py:87`, `server/app/services/pairing.py:93`, `server/app/services/pairing.py:120`
  proposal: lock the pending row with `SELECT ... FOR UPDATE` or use an atomic `UPDATE ... WHERE consumed_at IS NULL RETURNING`, then test two concurrent completes.

- HIGH Re-pair after revoke is blocked by DB constraint
  evidence: `server/app/models.py:62`, `server/alembic/versions/0001_initial_schema.py:72`, `server/app/services/pairing.py:101`
  proposal: replace the full unique constraint with a partial unique index on active pairings: `(user_id, sender_id) WHERE revoked_at IS NULL`.

- HIGH `PairedSenderStore` delimiter encoding corrupts valid sender names
  evidence: `server/app/schemas.py:118`, `android/core/storage/src/main/kotlin/app/syncler/core/storage/PairedSenderStore.kt:100`, `android/core/storage/src/main/kotlin/app/syncler/core/storage/PairedSenderStore.kt:104`
  proposal: store JSON/Moshi, separate encrypted preference fields, or Base64-encode every variable text field.

- MAJOR Pairing UI is not reachable from the app
  evidence: `android/app/src/main/kotlin/app/syncler/android/MainActivity.kt:11`, `android/app/src/main/kotlin/app/syncler/android/MainActivity.kt:33`
  proposal: add navigation/state and an entry point from inbox/settings.

- MAJOR I4 runtime enforcement is not wired on Android
  evidence: `android/feature/pairing/src/main/kotlin/app/syncler/feature/pairing/PairingRepository.kt:20`, `android/app/src/main/kotlin/app/syncler/android/push/PushHostBindings.kt:26`
  proposal: message fetch/decrypt must look up `PairedSender` by `sender_id`, verify envelope signature against the locked public key, and reject unpaired/mismatched senders.

- MEDIUM Broker URL should not embed raw standard Base64
  evidence: `server/app/routers/pairing.py:88`
  proposal: percent-encode the token or use URL-safe Base64 consistently.

=== SHOW-STOPPERS FOR M7 ===
- Fix `/pairing/initiate` rate limiting so the endpoint works and cannot be bucket-spoofed.
- Make fingerprint confirmation a real pre-complete gate.
- Serialize token completion so one pending token creates at most one pairing.
- Fix the active-pairing uniqueness model before M7 builds re-pair/sync assumptions on it.
- Replace delimiter storage before relying on `PairedSenderStore` as the I4 trust anchor.

=== READY FOR M7? ===
No. The protocol shape is close, but M6 is not yet a safe foundation for I4. The biggest issue is that Android creates and stores trust before the user confirms the fingerprint, and the server does not enforce single-consumer completion under concurrency.
