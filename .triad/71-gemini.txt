### Per-topic verdict

1.  **Resolution of Disagreements: GREEN**
    *   **Separate X25519 key**: Fully support this. Avoiding Ed25519 $\to$ X25519 conversion removes a significant source of implementation friction in Python and Android, while providing cleaner audit trails and independent key rotation.
    *   **Spec before code**: Gating 5a-2 on 5a-1 (spec/vectors) is the correct high-integrity path.
    *   **Broker URL & Polling**: The resolutions (signed `initiate`, echoed `preview`, and 1s/120s jittered polling) are pragmatic and secure.

2.  **HPKE-style construction: GREEN**
    *   The HKDF derivation using `eph_pub || sender.bootstrap_pub` as salt effectively binds the key to the specific session context.
    *   The info string `"syncler-v1-bootstrap-aead"` provides necessary domain separation.
    *   The use of AES-256-GCM with a 12-byte random nonce is standard and robust.

3.  **AAD field set: GREEN**
    *   The field set (`protocol_version`, `pairing_id`, `sender_id`, `broker_url`, `bootstrap_key_id`, `exp`) covers all necessary binding points to prevent replay and confusion attacks.
    *   Including `broker_url` in the AAD ensures that even if a Syncler server misroutes an envelope, the intended sender will detect the mismatch.

4.  **Broker CAS semantics: GREEN**
    *   The "First POST wins / 200 idempotent / 409 conflict" model is the correct way to handle the bootstrap race condition while preventing simple replay attacks.

5.  **`exp` window: GREEN**
    *   The 60s TTL from creation, combined with a 5-minute skew tolerance at the broker, is a reasonable balance for mobile device clock drift.

6.  **Test vector requirements: GREEN**
    *   The commitment to include HKDF, AEAD, and Ed25519 signature vectors in 5a-1 is sufficient for ensuring interoperability.

7.  **5d batch sizing: GREEN**
