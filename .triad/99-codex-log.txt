## 10. Master-Key Rotation (V1.5)

The user's 32-byte master key (`§2`) is the root secret for all
per-user symmetric crypto: it AES-GCM-encrypts the synced user-
state blob (`encrypted_user_state.encrypted_blob`) and every
per-pairing opaque blob (`pairings.encrypted_state`). Rotation
replaces the master key while preserving access to all
historical state.

### 10.1 Rotation modes

Every rotation request carries a `reason` field. The reason
controls what's regenerated and what side effects fire:

- `password_rewrap` — user changed their password. Derive a new
  `master_key_wrap_key` from the new password + the existing
  `auth_salt`, re-wrap the **unchanged** master key under the
  new wrap key. No data re-encryption. No session revocation.
  No `key_generation` bump.
- `root_hygiene_rotation` — periodic hygiene rotation. Generate
  a new 32-byte master key (`MK'`). Re-encrypt all dependent
  state (user-state blob + every pairing's `encrypted_state`)
  under `MK'`. Bump `key_generation`. Force other devices to
  re-pull (sessions stay valid but next sync prompts a re-fetch).
- `root_compromise_rotation` — master key may have been exposed.
  Same data flow as `root_hygiene_rotation` PLUS: revoke all
  device sessions except the initiating one; the initiating
  session is forced to re-derive credentials too on next
  authenticated request.

### 10.2 What rotation does NOT protect against

Master-key rotation re-encrypts data that was stored encrypted
under the old key. It does NOT revoke:

- Sender-held pairing keys. The senders received their stable
  pairing keys at bootstrap and continue to encrypt messages
  with them. A compromised pairing key needs a separate
  re-pair protocol (V2 follow-up).
- Messages already delivered and decrypted on devices.
- Data exfiltrated from an unlocked device.
- Server sessions not explicitly revoked. `password_rewrap`
  does not revoke; `root_hygiene_rotation` does not revoke;
  only `root_compromise_rotation` revokes.
- Cached state on devices that don't sync after rotation.

The Android UX MUST distinguish "rotate account encryption key"
from "revoke paired senders" from "sign out other devices".

### 10.3 Schema

Three additions over the pre-Phase-8 schema:

```sql
ALTER TABLE users
    ADD COLUMN key_generation INTEGER NOT NULL DEFAULT 1;

ALTER TABLE encrypted_user_state
    ADD COLUMN key_generation INTEGER NOT NULL DEFAULT 1;

ALTER TABLE pairings
    ADD COLUMN state_version INTEGER NOT NULL DEFAULT 1;
ALTER TABLE pairings
    ADD COLUMN key_generation INTEGER NOT NULL DEFAULT 1;
```

`users.key_generation` is the user's current generation. Every
encrypted blob is tagged with the generation it was encrypted
under. Clients persist the highest generation they've ever seen
locally and **MUST hard-fail if the server returns any encrypted
blob whose `key_generation` is lower than that high-water mark**
(downgrade-attack defense; Gemini 98 footgun).

`pairings.state_version` is an optimistic concurrency counter
on the per-pairing encrypted_state. Without it, concurrent
sender-side state pushes could race the rotation transaction
(Codex 98 RED).

### 10.4 Wire format

```http
POST /v1/account/rotate-master-key
Content-Type: application/json
Authorization: Bearer <session_token>

{
  "reason": "root_hygiene_rotation" | "root_compromise_rotation" | "password_rewrap",
  "current_password_proof": "<base64 32-byte auth_key derived from CURRENT password>",
  "new_auth_key_hash": "<base64 32 bytes; password_rewrap + root_compromise_rotation only>",
  "new_auth_salt": "<base64 16 bytes; password_rewrap + root_compromise_rotation only>",
  "new_encrypted_master_key": "<base64 AES-GCM blob wrapping the new (or unchanged) master key>",
  "new_encrypted_user_state": {
    "encrypted_blob": "<base64 new blob under new master key>",
    "state_version_observed": <int — must equal the server's current state_version>
  },
  "pairings": [
    {
      "pairing_id": "<uuid>",
      "state_version_observed": <int>,
      "new_encrypted_state": "<base64 new blob under new master key>"
    },
    ...
  ]
}
```

`current_password_proof` is the auth_key derived from the
CURRENT password's Argon2id output — the same 32 bytes the
server already has hashed in `users.auth_key_hash`. The server
verifies this constant-time. Without it, an attacker with a
hijacked session could rotate the master key without re-proving
ownership (Codex 98 missing item).

`pairings[]` MUST contain EVERY pairing row currently associated
with the user, even pairings whose `encrypted_state` is empty
or unchanged in content (a re-encryption produces a new
ciphertext under the new key even for identical plaintext).
Missing pairings → 400.

### 10.5 Server-side processing

The rotation handler runs as a single Postgres transaction:

1. Verify `current_password_proof` against
   `users.auth_key_hash`. Fail with 401 if mismatch.
2. Apply rate limit: at most N rotations per user per day (e.g.
   3), to bound damage from a session-hijacked rotation
   request.
3. Lock the user row `SELECT ... FOR UPDATE` to serialize
   concurrent rotations.
4. Check `users.key_generation` is still the same one the
   client saw (the request implicitly observed it via the
   pairing/state CAS values, but a fresh `key_generation_observed`
   field is cleaner — add to wire format).
5. CAS-check `encrypted_user_state.state_version ==
   state_version_observed`. Fail with 409 (return current state)
   if not.
6. CAS-check EACH pairing's `state_version`. Fail with 409
   (return the list of mismatched pairings) if any not.
7. Compute `new_generation`:
   - `password_rewrap`: same generation (no data change).
   - `root_*`: `users.key_generation + 1`.
8. Write the new `users.encrypted_master_key`,
   `users.auth_key_hash` (if changed), `users.auth_salt` (if
   changed), `users.key_generation = new_generation`.
9. Write the new `encrypted_user_state.encrypted_blob`, bump
   `state_version`, set `key_generation = new_generation`.
10. For each pairing in the request: write new
    `encrypted_state`, bump `state_version`, set
    `key_generation = new_generation`.
11. For `root_compromise_rotation`: also delete all
    `device_sessions` rows except the one in `request.headers.Authorization`.
12. Commit. Emit a single audit log entry capturing
    `(user_id, reason, old_generation, new_generation,
    initiating_session_id, initiating_device_id, ip,
    user_agent, paired_count_at_rotation)`. NO secret material
    in the log.
13. Respond `200` with `{key_generation: <new>, state_version: <new>}`.

If any step fails before commit, the entire transaction rolls
back — no partial-commit "bricked account" scenario (Gemini 98
atomicity footgun).

### 10.6 Client-side flow

#### 10.6.1 Initiating device

1. User invokes "Rotate master key" from Settings (or "Change
   password" — different flow path leading to `password_rewrap`).
2. App prompts for current password. Derive Argon2id locally
   to recover `current_master_key_wrap_key` (to unwrap the
   current master key) AND `current_auth_key` (= `current_password_proof`).
3. For password change: prompt for new password, derive new
   Argon2id outputs.
4. For `root_*`: generate new master key with CSPRNG.
5. Fetch latest server state via `GET /v1/state` and every
   `GET /v1/pairings/{id}/state` (collect `state_version` for
   each). Record the observed `key_generation` from the user
   row (via `GET /v1/account` or similar).
6. Decrypt each blob with the CURRENT master key.
7. Re-encrypt each blob with the NEW master key (or the
   unchanged master key, for `password_rewrap`).
8. Re-wrap the new (or unchanged) master key with the new
   wrap key.
9. POST the rotation request.
10. On 200: update local state (new master key locked in
    Session, new `key_generation` persisted as high-water mark,
    new state cached locally).
11. On 409 (CAS): refetch the affected blob(s), re-decrypt with
    current master key, re-encrypt with new master key, retry.
    NO partial commit — every retry rebuilds the entire payload.

#### 10.6.2 Other device coming online

1. Periodic sync polls `GET /v1/state`. Response carries
   `key_generation`.
2. If server's `key_generation > device's persisted high-water
   mark`: a rotation has happened. Show "Account encryption key
   rotated — please re-enter your password" banner.
3. User enters password. Device derives Argon2id with the
   `auth_salt` returned by `/pre-login` (which may have been
   changed by a `password_rewrap` or `root_compromise_rotation`).
4. Login as normal, fetch the new wrapped master key, unwrap.
5. Refetch all state blobs (they're now encrypted under the
   new master key). Persist `key_generation` as the new
   high-water mark.

#### 10.6.3 Offline merge

If the device made local changes while offline that haven't
been pushed, the device's CAS push will 409 against the
post-rotation state. The device:

1. Discards its old encrypted local blob (it was under the
   superseded master key).
2. Decrypts the new server blob with the new master key.
3. Replays the local pending changes on top of the new
   decrypted state.
4. Re-encrypts with the new master key, pushes via fresh CAS.

If a local change conflicts with a server-side change that's
been merged into the rotated blob (e.g. both dismissed the same
message), the existing per-field merge rules in the user-state
CAS protocol apply unchanged.

### 10.7 Downgrade-attack defense

Every device persists `highest_key_generation_seen` in local
non-encrypted storage. Every response from a server endpoint
that returns a `key_generation`-tagged blob is checked:

```
if server_response.key_generation < highest_key_generation_seen:
    hard_fail("server returned downgraded generation; possible attack")
```

This catches a malicious or compromised server attempting to
serve pre-rotation data to an already-rotated client (Gemini 98
footgun).

### 10.8 Test vectors

The following byte-equivalent fixtures MUST pass on every
implementation:

```text

old_wrap_nonce:      000000000000000000000001

new_wrap_nonce:      000000000000000000000002

# Plaintext user state JSON (canonical: sort_keys=True, separators=(",",":")):

The AAD shape for `encrypted_user_state` is currently unspecified
in the existing crypto-spec — Phase 8a locks it as:

```json
{"key_generation": <int>, "state_version": <int>, "user_id": "<uuid>"}
```

Canonical JSON: `sort_keys=True`, `ensure_ascii=True`,
`separators=(",", ":")`. Same canonicalization as §4 message AAD.

`pairings.encrypted_state` AAD:

```json
{"key_generation": <int>, "pairing_id": "<uuid>", "state_version": <int>, "user_id": "<uuid>"}
```

The bindings into AAD give the rotation transaction explicit
cryptographic coupling between the blob and its generation
counter — a server attempting to attach an old `encrypted_blob`
to a new `key_generation` row would fail AEAD tag verification
on the client.

## Open questions

1. **AAD shape for user-state + pairing-state.** Currently
   unspecified in the existing spec. The above proposal binds
   `key_generation` + `state_version` + `user_id` (and
   `pairing_id` where applicable) into AAD so the server can't
   silently rebind a blob to a different generation. Are these
   the right fields? Anything missing?

2. **`key_generation_observed` in the rotation request.**
   Currently the wire format I drafted relies on the per-blob
   `state_version_observed` for CAS, but doesn't have a
   user-level `key_generation_observed`. Codex 98 / Gemini 98
   both want strict CAS-against-rotation. Should the request
   carry it explicitly? I lean yes; flag if you disagree.

3. **Recent-auth window for `current_password_proof`.** Should
   we accept the password proof up to N minutes after the last
   successful login, OR force the user to re-type the current
   password at rotation time? I lean: force re-type (a session
   hijacker shouldn't be able to rotate without the live
   password).

4. **What about cards' encrypted_payload?** Live cards encrypt
   their payload with the per-sender pairing key, not the
   master key. Since rotation preserves pairing keys verbatim
   (audit insight), cards survive rotation unchanged. Confirm.

5. **Backup / recovery banner.** Should the spec say the
   client UI MUST show a "back up your password — no recovery
   if lost" warning before a password-rewrap or root rotation?
   I lean yes (spec-level requirement, not just product UX).

6. **Mixed-client behavior.** What does an older app version
   (running pre-Phase-8 schema, doesn't know about
   `key_generation`) do when it sees a server response carrying
   it? The pre-Phase-8 app would ignore the unknown field, try
   to decrypt with its cached master key, and either succeed
   (if the user is still at gen 1) or fail with AEAD tag error
   (if generation > 1). Acceptable? Or should servers refuse
   pre-Phase-8 clients explicitly once the user has rotated?

**Verdict: RED**

Do not commit verbatim. The shape is good, but there are several spec-level contradictions that will produce incompatible implementations.

**Open Questions**

1. **AAD fields:** Mostly right. Include `user_id`, `key_generation`, `state_version`, and `pairing_id` for pairing state. Also specify exact UUID string format and whether AAD uses pre-update or post-update `state_version` / `key_generation`. It should bind to the row values stored with that ciphertext, meaning the new post-rotation values.

2. **`key_generation_observed`:** Yes, explicit field required. Do not rely on blob CAS values to imply user-level generation. Add it to the wire format and make the server compare it under the user row lock.

3. **Recent-auth window:** Force re-type at rotation time. A cached recent-auth bit weakens the core protection against session hijack. If you later want a UX shortcut, make it a separate explicit threat-model decision.

4. **Cards’ `encrypted_payload`:** Confirmed, assuming cards are encrypted only under sender/pairing keys and not transitively under the master key. They should survive unchanged. The spec should state cards are excluded from the rotation transaction.

5. **Backup/recovery banner:** Yes. Add a client UX MUST before password change or root rotation: warn that losing the password/master unwrap path makes encrypted account data unrecoverable.

6. **Mixed clients:** I would require server-side refusal for pre-Phase-8 clients once `users.key_generation > 1`, if the server can identify client capability. Silent AEAD failure is technically safe but poor and may cause confusing destructive recovery flows. Minimum: endpoints returning generation-tagged blobs should include a clear “client upgrade required” error for clients lacking key-generation support.

**Blocking Issues**

`password_rewrap` is internally inconsistent. §10.1 says unchanged master key, no data re-encryption, no generation bump. But the wire format requires `new_encrypted_user_state` and `pairings`, and server steps 9-10 rewrite blobs and bump `state_version`. Pick one. I recommend: `password_rewrap` only updates `encrypted_master_key` and auth credentials, does not send/rewrite user-state or pairing-state, and does not bump blob `state_version`.

Salt handling contradicts itself. §10.1 says password rewrap uses the existing `auth_salt`; the wire format says `new_auth_salt` is required for `password_rewrap`; §10.6.2 says `auth_salt` may change after `password_rewrap`. Decide whether password change rotates salt. I recommend rotating salt on password change, but then fix §10.1.

`root_compromise_rotation` says the initiating session is forced to re-derive credentials on next authenticated request, but server processing only deletes other sessions. You need a concrete mechanism: session credential epoch, `requires_reauth` flag, session version mismatch, or revoke the initiating session after response.

`root_compromise_rotation` requiring `new_auth_key_hash` / `new_auth_salt` is ambiguous. Is root compromise also a password change? If yes, say it forces password reset and new password derivation. If no, those fields should not be required.

`current_password_proof` is dangerously underspecified. Sending the raw `auth_key` makes it a password-equivalent credential. Specify TLS-only, never logged, constant-time hash verification, and preferably use a server nonce/challenge bound to session/user/reason so a captured proof cannot be replayed as a future rotation proof.

The response shape is too small. `{key_generation, state_version}` only covers user-state, not per-pairing versions. Either return all updated pairing versions or require clients to refetch after success.

**Other Spec Fixes Needed**

Define whether `state_version_observed` for `encrypted_user_state` exists in schema already; §10.3 only adds `key_generation`, but later relies on `encrypted_user_state.state_version`.

Define the exact 409 payloads. “Return current state” could leak encrypted blobs, versions, pairing IDs, or all of the above. Implementers need exact fields.

Define missing-pairing behavior under concurrent pairing creation/deletion. If a pairing is added after client fetch but before rotation, missing pairings → 400 or 409? This should be 409 conflict, not malformed request.

Move audit logging semantics either inside the transaction with durable insert before commit, or explicitly after commit with failure handling. Current “Commit. Emit” means rotation can succeed with no audit log.

Replace “at most N” with a real configurable policy name/default. Specs should not contain unresolved placeholders.

Test vectors are not yet actionable because Argon2id parameters, AES-GCM blob encoding format, UUID fixture values, AAD versions, and tag/ciphertext concatenation order are not locked.

Overall: the design is close, but this is not ready for verbatim commit. Fix the contradictions around `password_rewrap`, salt rotation, root-compromise session handling, and explicit generation CAS first.
