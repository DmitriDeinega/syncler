YELLOW on the overall direction.

Option A is the right baseline, but the current plan needs one important correction: do **not** treat per-pairing message keys as derivable runtime material if senders already hold stable pairing keys. Master-key rotation should rotate the root that encrypts local/synced user-owned state, not silently change sender-facing shared secrets.

**1. Option A vs B vs C**

Option A is the right choice.

Option B adds a lot of protocol surface for a weak benefit. “All devices acknowledged” is not well-defined unless you already have durable device identity, device inventory, and retirement semantics. That is exactly what #4 is supposed to introduce, so doing it now risks building a temporary quasi-device model.

Option C is unsuitable for compromise rotation because it preserves historical exposure under the compromised key. It is only acceptable for performance-oriented key rollover, not for this stated threat model.

I would frame Option A as:

- `password change`: rewrap only, no root/data re-encryption.
- `hygiene rotation`: new master key, full re-encryption, other sessions may continue or be forced depending on product choice.
- `compromise rotation`: new master key, full re-encryption, revoke other sessions.

**2. Phase scoping**

Split into 8a / 8b.

8a should lock the protocol, invariants, schema shape, failure behavior, and UX/security semantics before code. This is too easy to get subtly wrong if implementation starts while the sender/pairing-key semantics are still ambiguous.

Suggested split:

- `8a`: crypto spec, server protocol spec, migration plan, rotation transaction contract, conflict semantics, Android UX states.
- `8b`: server migration/routes/service, Android implementation, tests.

I would not include Python SDK changes unless the SDK has direct authenticated-user state operations. Sender SDKs should remain unaffected if pairing keys are stable.

**3. Pairing `encrypted_state` CAS**

Yes, add a version column.

Without `pairings.state_version` or equivalent optimistic concurrency, rotation can overwrite an active pairing-state update. The fact that `encrypted_state` is opaque makes this worse: the server cannot merge or reason about conflicts.

Minimum recommendation:

- Add `pairings.state_version integer not null default 1`.
- Updates to `pairings.encrypted_state` must use compare-and-swap.
- Rotation request includes the observed `state_version` for each touched pairing row.
- If any pairing row changed, the whole rotation fails with conflict and the client must refetch/retry.

Also consider a coarse user-level lock or transaction-scoped “rotation in progress” guard if there are multiple routes mutating encrypted user-owned blobs. CAS is still needed, but a rotation endpoint should not race piecemeal with normal sync writes.

**4. Per-sender HKDF pairing keys**

Your correction is right, assuming the audit is accurate: master-key rotation should **not** regenerate sender-facing pairing keys.

The important distinction is:

- The master key encrypts the synced user-state blob.
- The user-state blob contains stable per-pairing keys as data.
- Senders already possess those stable pairing keys from bootstrap.
- Therefore rotation re-encrypts the blob that stores the pairing keys, but does not change the pairing key values.

If current code actually derives the live message key as:

```text
HKDF(master_key, sender_id, ...)
```

on every decrypt/encrypt path, then rotation would break senders unless you introduce sender key rotation. But your later clarification says the HKDF-derived value is generated/stored at first pairing and then treated as the stable pairing key. That should be verified in code/spec and made explicit.

I would update the terminology to avoid future mistakes:

- Avoid saying “per-pairing AES keys are derived per-sender from master key” unless they are derived every time.
- Say: “Initial pairing key material may be generated/derived client-side, then stored as stable pairing secret inside encrypted user state and shared with sender.”
- Rotation MUST preserve pairing key bytes.

For compromise rotation, this has a security implication: if the master key was compromised and the encrypted user-state blob was available, then historical and future sender pairing keys may also be compromised. Re-encrypting them under a new master key does not revoke sender-channel compromise. That needs a separate “rotate pairing keys / re-pair sender” story.

**5. Session revocation**

Yes for compromise rotation.

I would make rotation mode explicit in the API, not inferred:

```text
password_rewrap
root_hygiene_rotation
root_compromise_rotation
```

Policy:

- `password_rewrap`: no session revocation required, though password-auth sessions may need normal credential refresh.
- `root_hygiene_rotation`: probably force re-auth/key refresh on other devices, but not necessarily revoke all sessions.
- `root_compromise_rotation`: revoke all sessions except initiating session, invalidate cached wrapped-master-key material, and require password login again elsewhere.

If the system cannot distinguish these modes safely in UX, bias toward stronger invalidation for “Rotate master key” and keep “Change password” separate.

**Anything Missing**

The biggest missing item is rollback/failure semantics.

The rotation transaction must be all-or-nothing across:

- `users.encrypted_master_key`
- `users.key_generation`
- `encrypted_user_state`
- every touched `pairings.encrypted_state`
- session revocation metadata, if compromise mode

You also need define what happens if the client decrypts all state, builds the rotation payload, and then CAS fails. Answer should be: no partial commit; client refetches latest state and retries from scratch.

Other missing pieces:

- Audit logging: record rotation type, timestamp, initiating session/device/user agent, without logging secrets.
- Rate limiting / recent-password confirmation: rotation should require fresh password verification or equivalent recent auth.
- Backup/recovery warning: a failed or forgotten new password after rewrap can permanently lose access.
- Test vectors: old MK decrypts old blob, new MK decrypts new blob, old MK fails after rotation.
- Mixed-client behavior: older app versions should fail closed with a clear “update required / re-login required” state.
- Rotation authorization: possession of a valid session alone may be insufficient for compromise-sensitive rotation; require password/current credential.

**New Security Footguns**

The main footgun is overclaiming compromise recovery. Root rotation protects data that was only exposed as encrypted blobs under the old master key. It does **not** revoke:

- sender-held pairing keys,
- messages already delivered/decrypted,
- data exfiltrated from an unlocked device,
- server sessions not explicitly revoked,
- old local device caches.

So the UX should distinguish:

- “Rotate account encryption key” from
- “Revoke paired senders / re-pair senders” from
- “Sign out other devices.”

For #4 readiness, Option A gives you the needed primitive: decrypt all user-owned encrypted state, re-encrypt under a new root, commit atomically with generation/CAS. Just keep sender-facing pairing secrets stable during this phase, and document that sender-key revocation is a separate protocol.
