=== ADDITIONAL FINDINGS ===
- **CRITICAL** **Sandbox Escape via Reflection**:
  evidence: `PluginLoader.kt` uses `webView.addJavascriptInterface(bridge, ...)`. This is a known dangerous API. A malicious plugin can execute `window.__syncler_native__.getClass().forName("...")` to get a handle to arbitrary Java classes on the classpath and invoke their methods, completely bypassing the `PluginBridge`'s capability checks and escaping the sandbox.
  proposal: Replace the Javascript Interface with a `WebMessageListener` and `postMessage` communication channel. This is the modern, recommended, and much safer way to bridge between a WebView and native code, as it does not expose any native objects to the JS context.

- **HIGH** **In-Memory Nonce Registry Allows Replay Attacks**:
  evidence: `server/app/crypto/nonce.py` and `docs/crypto-spec.md`. The `NonceRegistry` is an in-memory LRU cache. If the server restarts or runs multiple workers, there is no shared state. An attacker can trivially replay a message to a different worker or after a restart.
  proposal: Implement the V1.5 plan mentioned in the spec immediately. Use a database or Redis-backed store for the nonce registry to ensure it is shared across all server processes and persists across restarts.

- **HIGH** **Rate Limiter Race Condition on SQLite**:
  evidence: `server/app/middleware/rate_limit.py`. The `_increment_and_read_sqlite` function performs a non-atomic `SELECT` followed by an `UPDATE` or `INSERT`. This is a textbook Time-of-Check-to-Time-of-Use (TOCTOU) race condition that can allow an attacker to bypass rate limits under concurrent requests.
  proposal: Wrap the SELECT/UPDATE/INSERT logic in a transaction with `SERIALIZABLE` isolation level. While SQLite's support for concurrency is limited, this will prevent simultaneous writes to the same row. Better yet, use the Postgres-style `INSERT ... ON CONFLICT` if the schema can be adapted (e.g., by making the composite key the primary key).

- **HIGH** **Lack of Plugin Key Revocation Mechanism**:
  evidence: `PluginLoader.kt`. Plugin trust is anchored solely on the sender's public key. If a plugin developer's private signing key is compromised, an attacker can sign and distribute malicious updates to all users of that plugin, and there is no mechanism to revoke the compromised key.
  proposal: This is a significant architectural risk. A mechanism for key revocation is needed. This could be a server-delivered list of revoked public keys that clients fetch periodically. For now, this must be documented as a critical risk of the current design.

- **MEDIUM** **Crypto-Spec Drift in Android AAD Implementation**:
  evidence: `android/core/crypto/src/main/kotlin/app/syncler/core/crypto/Aad.kt` and `server/app/crypto/aead.py`. The spec and Python implementation use sorted keys for canonical AAD JSON. The Kotlin implementation uses a hardcoded `linkedMapOf`. While the order currently matches alphabetical, this is fragile and will break if new fields are added out of order, causing all message decryption to fail.
  proposal: Modify the Kotlin `toCanonicalJsonBytes` function to explicitly sort the map keys before serialization to strictly adhere to the spec.

- **MEDIUM** **Android Crypto Implementation Untested Against Spec Vectors**:
  evidence: `docs/crypto-spec.md` mentions the test vectors are asserted by `server/tests/test_crypto.py`. I can find no corresponding evidence of a test suite in the Android codebase that consumes and verifies these canonical vectors.
  proposal: Create a JUnit test suite in `android/core/crypto` that parses the test vectors from `crypto-spec.md` (or a machine-readable version of it) and runs them against the Kotlin crypto implementations (`KeyDerivation`, `Signing`, `Aad`, etc.).

- **LOW** **Insecure WebView URL Loading**:
  evidence: `PluginWebViewClient.kt`'s `shouldOverrideUrlLoading` returns `true`, which blocks navigation. However, it would be more secure and explicit to have the WebView call `loadUrl` on an external browser via an `Intent` for any navigation attempts not targeting the base URL, rather than silently swallowing them. This is a minor hardening improvement.
  proposal: In `onPageStarted` or `shouldOverrideUrlLoading`, if the URL is not the initial one, instead of just `stopLoading()`, create and fire an `Intent(Intent.ACTION_VIEW, uri)` to open the link in an external, trusted browser.
