### Answers to Open Questions

1.  **Bootstrap key columns**: Inline + nullable on the `senders` table is the correct choice for V1.5. This keeps the schema simple while fulfilling the requirement for a single current bootstrap identity.
2.  **Broker URL persistence**: Retaining `broker_url` on the `pairings` row for audit is acceptable. It is non-sensitive metadata that aids in troubleshooting pairing failures.
3.  **BrokerServer placement**: Shipping the helper in the SDK is acceptable. However, to keep the base SDK lightweight, recommend making the `fastapi` dependency an optional "extra" (e.g., `pip install syncler[broker]`).
4.  **`wait_for_pairing` semantics**: Synchronous polling is preferred for consistency with the existing `Client` methods.
5.  **Android post-confirm UX**: Consolidating the loading and failure states within the existing `PairingPreview` screen is the cleaner UX choice, avoiding unnecessary fragment transitions.

---

### Anything missing?
*   **Rate Limiting**: Ensure the new `/v1/senders/me/bootstrap-key` endpoint is integrated into the server's rate-limiting middleware to prevent rotation spam.
*   **Broker Timeout**: The Android `OkHttpClient` for the broker POST should use a specific, shorter timeout (e.g., 10s) to ensure the manual fallback is presented promptly if the sender's broker is unresponsive.
