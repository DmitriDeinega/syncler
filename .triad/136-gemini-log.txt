Here is the Phase 11b mid-track review.

### 1. `bundleFd` lifecycle on the success path
**FIX**
The host must explicitly close its `bundleFd` immediately after `loadPlugin` returns successfully. In Android Binder IPC, passing a `ParcelFileDescriptor` duplicates the underlying FD in the target process. The sending process retains its own open FD. Relying on GC to close the host's FD can easily lead to FD exhaustion under moderate plugin churn before the finalizer runs.

### 2. `bundleFd` ownership on the failure path
**OK**
Your current order (`runCatching { bundleFd.close() }` followed by `unbindForToken`) is correct. Because the sandbox receives its own independent FD pointing to the same file description, the host closing its FD will not corrupt or interrupt a mid-flight `read()` in the sandbox.

### 3. `InMemoryDexClassLoader` parent = SDK runtime classloader
**FIX**
Using `SynclerPlugin::class.java.classLoader` gives the isolated process visibility into the *entire* host APK classpath (including internal networking libraries, view models, and third-party dependencies). While the sandbox lacks OS permissions, this violates the principle of least privilege, allows accidental linkage against host internals, and risks exposing in-memory host constants. 
*Recommendation:* Implement a strict delegating class loader as the parent. It should forward requests for `app.syncler.plugin.runtime.*` (and standard `java.*`/`android.*` library classes) to the host's class loader, and explicitly throw `ClassNotFoundException` for everything else.

### 4. `BridgePluginContext` JSON encode/decode
**FIX**
The `escape()` function ignores control characters (`< 0x20`). If a plugin sends a string containing `\b`, `\f`, or `\u0000`, it generates malformed JSON. When this crosses the bridge, the host's robust JSON parser (e.g., Moshi) will throw a parse exception, which could crash the host if not caught, or at best blindly fail the IPC call. You must add an `else if (c < ' ')` branch to emit `\uXXXX` unicode escapes.

### 5. Per-token process death attribution
**OK**
There is no race here. `handleSandboxDeath` explicitly filters out native handles via `!it.value.isNative`. Native handles are strictly managed by their own per-token `onServiceDisconnected` callbacks, ensuring cleanly partitioned death handling.

### 6. `bindIsolatedService` executor choice
**OK** (with a **NIT**)
A direct executor (`{ r -> r.run() }`) means the callbacks run synchronously on the Binder IPC thread. This is safe because `NativePluginSandboxConnection` only performs fast map manipulations and launches a coroutine. 
*NIT:* For strict consistency with the JS `bindService` path—and to avoid any subtle thread-safety assumptions if `SandboxRouter` is modified later—consider using a single-thread executor or `Dispatchers.Main.asExecutor()`.

### 7. Native sandbox concurrent-load semantics
**NIT**
Since `bindIsolatedService` with `instanceName=token` guarantees a pristine 1:1 process-to-plugin mapping, the sandbox receiving two loads is strictly a host invariant violation. Returning `concurrent_load_in_progress` is overly graceful. Change it to `error("Invariant violated: multiple loads sent to isolated process")` to crash the sandbox and fail-fast.

### 8. Forbidden-prefix scan defense-in-depth
**OK**
Keep it. Repeating the scan in the isolated sandbox acts as the ultimate enforcer. It completely mitigates Time-Of-Check-To-Time-Of-Use (TOCTOU) vulnerabilities between the host's scan and the `bundleFd` IPC transfer, and protects against future host refactors accidentally skipping the check. The CPU cost is negligible.

### 9. DEX size cap enforcement timing
**NIT**
Reading `DEX_MAX_BYTES + 1` before throwing is safe from OOM but slightly inefficient since it buffers the overage. You could increment an accumulator in the 64KB read loop and throw `DEX_TOO_LARGE` immediately upon crossing the threshold, avoiding the final heap allocation and array copy. 

### 10. API gate placement
**OK**
Because the host synchronously rejects API < 29 before attempting IPC or spawning the sandbox process, the sandbox will never emit this error. Keeping `native_only_api_29` purely in the host domain and out of the `NativeLoadFailureCodes` AIDL contract is logically sound.

### Additional Issues Spotted

**FIX — JSON Decoding Fragility in `BridgePluginContext`**
While hand-rolling keeps the trusted surface small, the decoders are too brittle:
1. `decodeStringList`: It splits the array body using `body.split(',')`. If the host returns a list of strings that *contain* commas (e.g., file paths or user-defined prefixes), this will blindly shred the string into invalid fragments.
2. `pickString`: It stops at the first `"` it finds (`json.indexOf('"', from)`). If the host sends a string containing an escaped quote (e.g., `"error": "User said \"no\""`), `pickString` will terminate early and return garbage. 
*Recommendation:* You must fortify these minimal parsers to respect `\"` escapes and properly tokenize JSON arrays, or bundle a micro JSON-reader.

**NIT — Redundant Coroutine Cancellation in `RealNativePluginHost.kt`**
Inside `initJob = scope.launch { ... }`, when `withTimeoutOrNull` returns `null`, you call `initJob?.cancel()`. This is redundant. `withTimeoutOrNull` already cancels its inner block and waits for it to finish. Cancelling `initJob` just tells the coroutine to cancel itself precisely as it's already completing the timeout block. If the intent is to kill the whole sandbox upon initialization timeout, you should be calling `scope.cancel()` instead.
