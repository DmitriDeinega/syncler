Verdict: **YELLOW**

The IPC scaffolding is architecturally sound and the state machine logic in `PluginTokenCoordinator` is robust. However, there are significant risks regarding the reference counting of the sandbox connection during crash/error scenarios that should be addressed before moving to the disruptive Step 5.

### 1. Reference Count Leak on Process Death (CRITICAL)
In `SandboxRouter.handleSandboxDeath()`, you clear the `handles` map but do **not** call `connection.release()` for the dead handles.
*   **Impact:** Every time the `:plugin` process dies, the `refCount` in `PluginSandboxConnection` is permanently leaked by the number of active plugins at the time of death. 
*   **Fix:** `handleSandboxDeath` should iterate over the handles and call `connection.release()` for each (or reset the `refCount` if it's authoritative that the process is dead).

### 2. The "Load/Crash" Race Condition
There is a window in `SandboxRouter.loadPlugin()` between `sandbox.loadPlugin(...)` returning and `handles[parcel.sandboxToken] = handle` being executed.
*   **Risk:** If the sandbox crashes in this millisecond, `handleSandboxDeath` won't find the token in `handles` (it’s not there yet). When `loadPlugin` resumes, it puts the handle into the map, but the "death" event has already passed.
*   **Result:** You have a stale handle in the map, and the `acquire()` from `loadPlugin` is never balanced by a `release()`, because the process is already gone and won't fire `onPluginUnloaded`.
*   **Recommendation:** Synchronize the `handles` map access or use a state-gate to ensure `loadPlugin` can't "miss" a crash event.

### 3. Double-Release Risk on Load Failure
In `PluginSandboxService.loadPlugin`, if `startLoad` throws, the sandbox calls `coordinator.unload()`, which fires `onPluginUnloaded` to the host.
*   **Risk:** The host's `loadPlugin` also catches the exception and calls `connection.release()`. If the oneway `onPluginUnloaded` arrives at the host's stub, it might trigger a second `release()`.
*   **Recommendation:** The sandbox should avoid firing `onPluginUnloaded` if the synchronous `loadPlugin` call is going to throw an exception anyway. Let the host's `catch` block be the sole owner of cleanup for sync failures.

### 4. Mutex Contention & Dead Code
In `PluginSandboxConnection.acquire()`, the `mutex.withLock` wraps the entire method, including the `await()` on `connectInflight`.
*   **Performance:** This serializes all `acquire()` calls, even when the sandbox is already connected. While safe, a `Mutex` is heavy for a fast-path null-check.
*   **Redundancy:** Because the mutex is held during suspension, `connectInflight` is effectively dead code—no other coroutine can enter the block to see it while the first one is binding.
*   **Recommendation:** Consider using an `AtomicReference` for the `sandbox` check outside the mutex to allow concurrent fast-path acquisitions.

### 5. Minor Observations
*   **String Length:** `PluginLoadParcel` uses `String.length` for the 64KB cap. Since Binder limits are byte-based, a string with many multi-byte characters could theoretically still be problematic, though 64K characters is likely safe enough for manifest JSON.
*   **Hilt:** Lack of Hilt in `:feature:plugin-host` is fine for now, but ensure the `Application` class correctly manages the singleton lifecycle of `PluginSandboxConnection` to avoid multiple uncoordinated bind/unbind cycles.

**Proceed to Step 5 only after fixing the reference leak in `handleSandboxDeath`.**
