# Consultation 61 — Triad Diagnosis & Repair Plan

The triad (Gemini, Claude, and Codex) has audited the working tree and the reported build failure. We confirm the tree is in a "split-phase" state where Phase 3c (MuteStore) relies on a schema change (Phase 3c/V5) that hasn't been applied to `EncryptedUserState`, while Phase 3b (Live Cards) has partially overwritten the Phase 3a (Templates) logic in `InboxRepository`.

## 1. Diagnosis

### Build Break (RED)
- `MuteStore.kt` references `mutedSenders` on `EncryptedUserState`, which does not exist.
- `StateMerger.kt` lacks the merge logic for this new field.

### Incoherence (YELLOW)
- `InboxRepository.kt` references `decryptLiveCard` and `decryptEventMessage` in its `refresh()` loop, but these helpers are missing from the file.
- `PluginPublishRequest` in `server/app/schemas.py` allows `card_type: "live"` without enforcing `card_key_path` is non-null.
- The `InboxItem` data class in Android has been updated with `cardKey`, `sequenceNumber`, etc., but the decryption logic doesn't yet populate them correctly for live cards.

---

## 2. Recommended Repair Sequence

We recommend proceeding with a **unified Phase 3 landing**. Since the user has already layered 3a, 3b, and 3c in the working tree and rejected stashing, the safest path is to make this combined state coherent and verified.

### Step 1: Unblock Android Build (Storage)
- **Extend `EncryptedUserState.kt`**: Add `mutedSenders: List<String>`, bump to `SCHEMA_V5`, and update JSON serialization/deserialization.
- **Update `StateMerger.kt`**: Implement union merge for `mutedSenders`.

### Step 2: Restore Inbox Coherence (Feature)
- **Update `InboxRepository.kt`**: 
    - Implement `decryptEventMessage` (renamed from `decryptWithAnyKey`).
    - Implement `decryptLiveCard` (populates `cardKey`, `sequenceNumber`, `type="live"`).
    - Update `refresh()` and `upsertCard()` to use these helpers.

### Step 3: Server-side Safety
- **Update `server/app/schemas.py`**: Add a cross-field validator to `PluginPublishRequest` ensuring `card_type == "live"` requires a `card_key_path`.

## 3. Triad Finding Details

### [Finding 1] `EncryptedUserState` Schema Bump
- **Status:** **RED** (Blocking)
- **Finding:** `MuteStore` needs `mutedSenders`. 
- **Recommendation:** Add as `List<String>`. Bump to `SCHEMA_V5`. `fromJson` should map `V0-V4` to `V5` and provide an empty list for the missing field.

### [Finding 2] `InboxRepository` Decryption Helpers
- **Status:** **YELLOW** (Incoherent)
- **Finding:** The `refresh()` logic was updated for Phase 3b but the helper methods were not added.
- **Recommendation:** Implement `decryptLiveCard`. It must use the `card_key_path` from the plugin manifest to extract the stable key, or use a server-provided `cardKey` if available in the DTO. (The plan says `card_key_path` is the source of truth).

### [Finding 3] Live Card Validation
- **Status:** **YELLOW** (Security/Integrity)
- **Finding:** Senders could publish a "live" plugin without a `card_key_path`, which would cause client-side extraction to fail.
- **Recommendation:** Add `model_validator` in `PluginPublishRequest`.
