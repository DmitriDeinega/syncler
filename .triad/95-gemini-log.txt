**1. Per-area:**
GREEN. The schema, composite PK `ON CONFLICT DO NOTHING` design, and removal of the in-memory cache represent a massive architectural upgrade. Tying the nonce registration directly to the SQL transaction guarantees consistency and prevents lock contention across workers.

**2. Answers to open questions:**

**1. TTL cleanup mechanism:** Option **B** (Periodic background task with advisory lock). Running a simple `asyncio.sleep` loop via a FastAPI startup event that grabs `pg_try_advisory_lock` ensures cleanup happens reliably without forcing random HTTP requests to absorb the latency spike of a bulk `DELETE` (which Option A risks). If you strongly prefer A for simplicity, execute the delete via FastAPI's `BackgroundTasks` so the client isn't blocked.

**2. Cards.py replay protection:** **Scope IN**. Live cards enforce a 48h `MAX_TTL`. Since the machinery is being built, dropping `record_nonce_or_reject` into `cards.py` is trivial and plugs a theoretical attack vector (resurrecting deleted/tombstoned cards by replaying old sequence numbers).

**3. TTL value:** Tie to **`MAX_RETENTION` (30 days)**. Both `store_message` and `store_card` aggressively enforce `expires_at <= now`. If you delete a nonce from the DB after 30 days (`seen_at < NOW() - INTERVAL '30 days'`), replaying that envelope will inherently fail because the new `now` will exceed the payload's `expires_at`. This makes a 30-day TTL mathematically safe for both domains.

**4. Keep in-memory NonceRegistry as L1 cache?** **Delete it completely**. L1 caches masking DB state introduce invalidation footguns. If a DB transaction rolls back, your in-memory cache would incorrectly lock out the client from a legitimate retry. The Postgres index on `(sender_id, nonce)` will easily serve this look-up fast enough.

**5. Postgres multi-worker concurrency semantics:** You are entirely covered. `ON CONFLICT DO NOTHING` resolves race conditions correctly even if the prior inserting transaction hasn't committed yet (the second transaction will block in READ COMMITTED until the first finishes, then evaluate the conflict).

**3. Anything missing:**
Ensure you update `cards.py`'s exception handling to also catch your new `NonceReplayError` (or equivalent HTTP 409) if scoping cards IN.

**4. Anything new (security, footgun):**
- **Transactional Atomicity (Positive):** The legacy in-memory `mark()` happened *after* `store_message`. The new `record_nonce_or_reject` happens *before*. Because they share the same SQLAlchemy `AsyncSession`, if `store_message` or `store_card` raises a validation error or throws a 500, the HTTP framework's rollback will *also rollback the nonce insert*. This correctly permits the sender to retry a failed send, while definitively rejecting replays of successful sends. Just ensure you don't prematurely `db.commit()` inside the service layer.
