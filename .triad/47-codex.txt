**Reviewer A**

1. **Migration:** Push existing local pairings up on first unlocked run after schema bump. Waiting strands old pairings. Use CAS merge by `pairingId`, one-shot migrated flag, and skip locals whose server pairing is already revoked.
2. **Tombstone GC:** No current precedent: `deletedMessages` grows forever. Keep pairing tombstones at least 180 days, matching server revoked-pairing retention, and only GC when the server pairing is also revoked/expired.
3. **Pairing CAS failure:** Do not roll back after server `completePairing` succeeds. Persist locally, mark state dirty, retry later. But fully offline QR confirm cannot work because the server token must be consumed online.
4. **Re-pair/rotation:** Docs must say rotation means revoke old pairing, re-pair, and update the sender to use the new key. Also tombstone old active pairings for the same `senderId`; `bySenderId()` otherwise becomes ambiguous.
5. **Threat hole:** Current device revocation does not actually stop `/v1/state` or `/v1/messages/inbox`. State is user-token only; inbox checks ownership, not `revoked_at`. Fix this before relying on the lost-device story.
6. **Mute per device:** Ship separately. It is UX-related, not security-critical, and synced pairing is already enough migration surface.
7. **Other:** `UserStateRepository.mutateLocal` is private and lives in `feature:inbox`; `PairedSenderStore` is in `core:storage`. Move/expose a small state mutation interface or relocate ownership cleanly.

Recommendation: synced pairing is the right direction, but do not ship as written until device revocation is made real and same-sender conflict/migration semantics are explicit.

**Reviewer B**

1. **Migration:** Same answer: import local pairings immediately. Add tests for two devices importing different local pairings concurrently.
2. **Tombstone GC:** Do not copy `deletedMessages`; it has no GC. Pairing tombstones need a deliberate TTL because they prevent removed keys from being resurrected.
3. **CAS failure:** Local-first is fine, but fix the existing CAS dirty-flag bug first: if conflict retry fails, `push()` can still clear dirty today.
4. **Sender rotation:** V1.5 docs must be blunt: re-pairing only rotates future messages after the sender updates its stored key.
5. **Threat hole:** Lost online device can still pull new state with a valid JWT unless routes are device-bound and reject revoked devices. This is the blocker.
6. **Mute per device:** Defer.
7. **Other:** `PairedSenderStore` should observe session lock/logout or be per-user partitioned; otherwise local sender keys can outlive the account context.

Overall: ship the V1.5 shape only after two adjustments: enforce active-device authorization on state/inbox/sensitive routes, and add first-run import plus same-`senderId` active conflict rules. Then implement the synced `pairedSenders` projection local-first with dirty retry.
