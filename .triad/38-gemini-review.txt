The Phase 2 implementation of read-state sync introduces critical concurrency and multi-user safety issues in `UserStateRepository.kt`. While the schema migration and UI visual treatment are sound, the core synchronization logic risks data loss and cross-user contamination.

### 1. Schema & Merge Semantics

*   **Forward Migration:** `EncryptedUserState.fromJson` correctly handles V0/V1 blobs by defaulting `readMessages` and `archivedMessages` to empty lists. This is non-destructive for existing V1 fields.
*   **Forward Compatibility Risk:** As noted in the prompt, a V2 client encountering a V3 blob (from a future client) will parse only known fields and drop V3-specific data on the next `toJson -> push` cycle. While we "own all clients," preserving unknown fields in `EncryptedUserState` (e.g., via a `Map<String, Any?>` for unrecognized JSON keys) would be a more robust "lesson learned" from M7.
*   **Merge Determinism:** `StateMerger` uses `a.readAt >= b.readAt` where `a` is local and `b` is remote. This correctly implements "local wins on equality," providing a deterministic tie-breaker for identical timestamps.

### 2. UserStateRepository: Concurrency & Atomicity (CRITICAL)

The current implementation has several race conditions that will lead to lost updates in production:

*   **Non-Atomic Local Mutation:** `markRead` (and `markArchived`) reads `_state.value`, copies it, and writes back. If two messages are marked read simultaneously, one will overwrite the other, losing the first update.
    *   *Fix:* Use `_state.update { ... }` (available on `MutableStateFlow`) or gate mutations with `syncMutex`.
*   **Pull vs. MarkRead Race:** `pull()` fetches remote state and merges it with the *current* `_state.value`. If a user marks a message read *during* the network call, `pull()`'s final `applyLocal()` will overwrite that new local mark with the stale merge result.
    *   *Fix:* The read-merge-write cycle in `pull` must be atomic relative to local mutations.
*   **Concurrent Pushes:** While `push()` is gated by `syncMutex`, multiple `markRead` calls will queue multiple `push()` operations. If the first push succeeds, the second push will re-encrypt and re-push the exact same state (redundant but safe). If the first fails with 409, the conflict handler runs, but the second queued push will still trigger.

### 3. Multi-User Safety & Logout (CRITICAL)

`UserStateRepository` is a `@Singleton` and does not clear its state on logout.
*   **Cross-User Contamination:** If User A logs out and User B logs in on the same device, `UserStateRepository` still holds User A's `readMessages` in memory (and on disk in `EncryptedSharedPreferences`).
*   **Data Corruption:** When User B triggers a `pull()`, the repository merges User B's remote state with User A's stale local state. User B now "inherits" User A's read marks and installed plugins. On the next `push()`, this corrupted state is written to User B's server account.
*   **Privacy Leak:** Even before a `pull`, the UI will show User A's read marks to User B.
    *   *Fix:* `UserStateRepository` must expose a `clear()` method called by `AuthRepository.logout()`, or be scoped to the session.

### 4. UI & Visuals

*   **Layout Stability:** Preserving the 8dp dot slot when read is the correct choice to prevent horizontal jitter when the user marks a row read.
*   **Trigger Logic:** Marking read on "detail open" is consistent with the conservative "scanning is not reading" philosophy.
*   **Logout/Login Lifecycle:** The `initialValue` in `InboxViewModel.readMessageIds` correctly pulls from the repository's current state, but as noted above, that state might be stale/stolen from a previous user if the app wasn't process-killed between sessions.

### 5. Sync Correctness

*   **CAS Policy:** One retry in `handleConflictAndRetry` is a pragmatic V1 policy. If a second 409 occurs, bailing and waiting for the next foreground/poll tick prevents infinite loops.
*   **Encryption Format:** The use of `Aead.encrypt` (nonce || ciphertext+tag) matches the project's standard for message-body encryption and is appropriate for the opaque server blob.

### Recommendations

1.  **Immediate Fix:** Add a `clear()` method to `UserStateRepository` that resets `_state` and clears `SharedPreferences`. Call it in `AuthRepository.logout()`.
2.  **Atomicity:** Refactor `markRead`, `markArchived`, and `pull` to use a consistent locking strategy (or `update {}` for the Flow) to ensure the read-modify-write cycle is atomic.
3.  **Future-Proofing:** Consider updating `EncryptedUserState` to retain unknown JSON fields to prevent data loss when mixing client versions.
