=== ARCHITECTURE CHECK ===
The zero-knowledge multi-device model is sound, but the key hierarchy needs to be explicit: password-derived key unlocks an encrypted user master key; sender pairings and message keys hang off that, not directly off the password. Broker pairing fits multi-device if pairing attaches to the user account/master public identity, then syncs encrypted sender state to every device. `platform.showNotification(...)` creates duplicate notification and action risks across devices; V1 needs account-level message IDs and sender-side idempotency so two phones cannot accidentally submit the same action twice. Plugin version pinning still works, but every device must be able to fetch/cache the exact plugin version required to render or handle a message.

=== READY FOR PLANNING? ===
yes. The first planning artifact should be the key hierarchy and sync model, because it now determines auth, pairing, plugin install sync, push fan-out, and recovery behavior.
