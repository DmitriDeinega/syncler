This specification (V1.5) is a high-integrity resolution of the critical issues identified in Consultation 99. The lockstep contract in §10.4 and the global requirement for `key_generation_observed` in §10.5 provide the necessary structural defenses against the AAD-prediction and stale-state-overwrite attacks.

### 1. Per-Section Review
*   **§10.1 - §10.3:** **GREEN**. The rotation matrix is now unambiguous, revokation semantics for compromise are appropriately "scorched earth," and the schema correctly accounts for the necessary versioning.
*   **§10.4 - §10.5:** **GREEN**. These are the most critical sections. The "observed + 1" lockstep solves the AAD prediction problem elegantly. The mandatory mismatch-check on all state-mutating endpoints ensures system-wide consistency.
*   **§10.6 - §10.8:** **GREEN**. The nonce-based challenge flow (§10.7) properly secures the `current_password_proof`. The 13-step atomic transaction logic in §10.8 covers all concurrency edge cases, specifically the pairing-count check within the `FOR UPDATE` lock.
*   **§10.9 - §10.13:** **GREEN**. The AAD shapes are comprehensive. The mixed-client handling via `426 Upgrade Required` is the correct path for account-level security upgrades.

### 2. Answers to Open Questions
1.  **AAD for `users.encrypted_master_key`:** I recommend including `key_generation` in this AAD as well. While the rationale in §10.9 suggests a chicken-and-egg problem, the client actually receives the `key_generation` integer unencrypted in the user profile response alongside the encrypted blob. Including it in the AAD prevents a server from serving a downgraded MK blob from a previous generation.
2.  **Phase-detection header:** **GREEN**. The `X-Syncler-Client-Min-Phase` header is significantly more reliable and easier to test than User-Agent parsing.
3.  **5-minute challenge TTL:** **GREEN**. This is a well-balanced window for Argon2id overhead and user latency while remaining tight enough for security.
4.  **3 rotations per 24h:** **GREEN**. This is a sensible rate-limit that prevents brute-force credential stuffing/rotation loops while allowing for human error.
5.  **Argon2id output fixtures:** **GREEN**. Locking these at the first implementation is acceptable, provided the implementation is verified against a standard library.

### 3. Missing/Wrong
*   **§10.6 Wire Format:** The `POST /v1/account/rotate-master-key/challenge` endpoint mentioned in §10.7 should be explicitly listed in the wire format summary for completeness.

### 4. New Findings
*   **Revocation timing:** In §10.8 Step 12, ensure the session deletion includes a mechanism to notify the current session to clear local caches, though a `401 Unauthorized` on the next call is the standard and sufficient backup.

**Verdict: Dual-GREEN.** I approve the commitment of this spec as Phase 8a. The shift to a deterministic lockstep for AAD prediction is a major security win for the project.
