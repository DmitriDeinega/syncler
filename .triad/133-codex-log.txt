# Consultation 133 — Phase 11a design v2 (post-132 RED)

Triad 132: Codex RED. Gemini not yet returned (will run in
parallel). Codex's verdict: ClassLoader isolation in shared
UID is not a sandbox; the V1 design lets native plugin code
bypass every capability gate via direct `java.net.*` / file
APIs / reflection / dynamic dex / `System.load(path)`. The
real fix is `android:isolatedProcess="true"`.

v2 incorporates Codex 132's structural fixes.

## v2 deltas vs v1

### Block 1: real Android sandbox via `isolatedProcess`

The new native runtime gets its own subprocess marked
`android:isolatedProcess="true"`, distinct from Phase 10's
`:plugin` (which keeps WebView for JS plugins).

```xml
<service
    android:name=".PluginNativeSandboxService"
    android:process=":nativePlugin"
    android:isolatedProcess="true"
    android:exported="false" />
```

What `isolatedProcess` gives us (Android docs):

- Runs under a SEPARATE UID synthesized per process.
- NO access to host app's data dir (`/data/data/<pkg>`).
- NO access to most `Context`-backed system services.
- NO INTERNET permission (even if host has it).
- NO ContentProvider, telephony, location, etc.
- Can only communicate via Binder back to the host.

This is exactly the sandbox we want for untrusted JVM code.
The plugin can call `java.net.URL.openConnection()` all it
wants — the OS refuses the network call because the
synthesized UID has no INTERNET grant.

The capability dispatch (`platform.network.fetch`, etc.)
flows over the existing AIDL `IPluginHostCallback.bridgeCall`
back to the host process, where `PluginBridge` runs the
actual privileged work and returns results via
`IPluginSandbox.deliverBridgeResult`. The host process is
the only place that holds INTERNET / storage / etc. grants.

This is identical to Phase 10's JS bridge model, but with
the OS doing real sandboxing instead of WebView's hardening.

### Block 2: DEX (not AAR) as the bundle format

Per Codex 132 #1: AAR isn't directly loadable at runtime;
it's a build-time format. V2 ships **DEX-only** for V1.5:

- Bundle is `classes.dex` (Dalvik/ART executable) wrapped
  in a JAR or plain bytes.
- No resources, no native libs, no manifest merging.
- Pure Kotlin/Java code; assets/resources stay host-side
  (plugin gets typed `PluginContext` instead).

Future Phase 11+ could add an AAR pipeline that
extract+dex+sanitize, but V1.5 doesn't need it.

Bundle hash: SHA-256 of the raw DEX bytes (same shape as JS
bundle).

### Block 3: `entry_class` in `PluginLoadParcel`

Per Codex 132 (and the Phase 10 invariant): host parses the
manifest ONCE and projects normalized fields into the
parcel. `entry_class` joins as a first-class field:

```kotlin
data class PluginLoadParcel(
    ...,
    val renderer: String,          // "script" | "native_kotlin"
    val bundleFilePath: String,
    val bundleHashHex: String,
    val entryClass: String,        // NEW — required when renderer == "native_kotlin"
    val nativeSdkAbi: Int,         // NEW — declared SDK ABI version (see Block 7)
    ...,
) : Parcelable
```

The host validates: `entryClass` MUST be present iff
`renderer == "native_kotlin"`; reverse-DNS-shape (same
regex as `pluginId`); fully qualified.

### Block 4: bounded per-plugin dispatcher

Per Codex 132 open question #4. Each native plugin gets its
own coroutine scope on a bounded dispatcher (max 4
concurrent coroutines, blocking IO offloaded to a separate
small pool):

```kotlin
class PluginExecutor(pluginId: String) {
    val cpu = ThreadPoolExecutor(
        corePoolSize = 1,
        maximumPoolSize = 4,
        keepAliveTime = 30L,
        unit = TimeUnit.SECONDS,
        workQueue = LinkedBlockingQueue(64),
    ).asCoroutineDispatcher()

    val scope = CoroutineScope(SupervisorJob() + cpu)

    fun shutdown() { scope.cancel(); cpu.close() }
}
```

The isolated process itself has a small main looper; we
deliberately don't use it for plugin coroutines.

### Block 5: `onInit` load timeout

Per Codex 132 missing #8: bound `onInit`'s wall-clock to
10s (default; configurable per plugin). If it doesn't
return Success in 10s, the sandbox calls `reportError("init_timeout")`
and unloads.

### Block 6: explicit "no dynamic dex" guard

The plugin's `ClassLoader` is constructed once at load with
the staged DEX path. Plugins CAN technically still call
`DexClassLoader(...)` themselves to load runtime DEX from
disk — but inside an isolated process with no INTERNET and
no host-app-data access, the only filesystem locations they
can write to are their own ephemeral private dir. Even if
they download more dex into that dir, the OS still refuses
the network call. So dynamic dex is "as harmful as
hand-written native code" — bounded by the same isolated
UID's restrictions.

Documented in the threat model rather than enforced. The
sandbox's strength is the OS-level UID isolation, not class
loader gating.

### Block 7: SDK ABI versioning

Per Codex 132 #6: plugins declare `nativeSdkAbi: Int` in the
manifest. The runtime SDK ships a `NATIVE_SDK_ABI =
<int>` constant. Mismatch → load fails with
`UnsupportedSdkAbiError`. Bump the int on every
breaking-change release.

V1.5 SDK ABI = 1.

### Block 8: native plugin API surface

```kotlin
interface SynclerPlugin {
    suspend fun onInit(ctx: PluginContext): Result<Unit>
    suspend fun onInbox(ctx: PluginContext, event: InboxEvent): Result<Unit> =
        Result.success(Unit)
    suspend fun onAction(ctx: PluginContext, event: ActionEvent): Result<Unit> =
        Result.success(Unit)
}

interface PluginContext {
    val pluginId: String
    val grantedCapabilities: Set<String>

    // Same surface as the JS platform.* API.
    suspend fun networkFetch(req: NetworkRequest): NetworkResponse
    suspend fun storage(): PluginStorage
    suspend fun showNotification(n: Notification): Result<Unit>
    suspend fun cameraCapture(opts: CameraOptions): Result<CameraResult>
    suspend fun galleryPick(opts: GalleryOptions): Result<GalleryResult>
    suspend fun filePick(opts: FileOptions): Result<FileResult>
    suspend fun locationCurrent(opts: LocationOptions): Result<LocationResult>
    suspend fun messageRespond(actionId: String, payload: ByteArray): Result<Unit>
}
```

All `PluginContext` calls marshal as `bridgeCall(method,
argsJson, callbackId)` over AIDL to the host — same path as
JS. Audit log, rate limit, capability check all reuse Phase
10's machinery.

Direct `java.net.*` / `System.loadLibrary` / etc. fail at
the OS because the isolated UID lacks those grants. The
runtime's job is making the typed Kotlin path *easier* than
those alternatives, not preventing them.

## Trust model (explicit)

V2 is honest about what the sandbox protects:

- **Confidentiality of host app data**: protected. Isolated UID
  has no access to `/data/data/<host_pkg>` or any host
  process state.
- **Direct network access**: blocked. Isolated UID has no
  INTERNET permission.
- **Other capabilities** (camera, location, etc.): blocked
  at the OS level. Plugin has only the grants of its
  synthesized UID, which is minimal.
- **Reflection inside the plugin's own process**: plugin can
  reflect on its own classes and the SDK classes. NOT a
  security issue — there's nothing privileged in the SDK
  surface; it's all proxies back over Binder.
- **Cross-plugin interference**: all native plugins share
  the same `:nativePlugin` process (each gets its own
  `sandboxToken` + ClassLoader + coroutine scope, but heap
  is shared). A buggy native plugin can OOM siblings. V2
  acceptable (same as Phase 10's `:plugin` shared model).
  V2+ could shard across multiple isolated processes.
- **CPU / memory exhaustion**: bounded per-plugin
  dispatcher mitigates CPU starvation. OOM is a real risk;
  Android will kill the whole `:nativePlugin` process if
  it goes over budget — `SandboxRouter.handleSandboxDeath`
  already handles that.

## Open questions for v2

A. **WebView in isolatedProcess**: Phase 10's `:plugin` is
   NOT isolated. Could we mark `:plugin` itself isolated
   too and get JS plugin sandboxing as a bonus? Android
   docs suggest WebView in isolatedProcess works since
   API 28, with caveats (no Service Worker, limited
   storage). For V1.5 leave `:plugin` non-isolated;
   document the asymmetry as a known gap (JS plugins
   trusted, native plugins sandboxed). Future phase
   could isolate `:plugin` too.

B. **`PluginBridge.show*` callbacks from isolated process**:
   `platform.showNotification` and `messageRespond`
   eventually need the host to invoke Android
   `NotificationManager` etc. Those calls happen in the
   HOST process (already do today via `PluginBridge`),
   not in `:nativePlugin`. AIDL is the bridge. Confirmed
   feasible.

C. **Process death + re-bind**: `SandboxRouter`'s
   `handleSandboxDeath` already handles the death of the
   shared `:plugin` process. With TWO sandbox processes,
   the router needs per-process death tracking. Minor
   refactor.

D. **Plugin SDK runtime distribution**: the `:plugin-sdk-runtime`
   module is on the `:nativePlugin` classpath (loaded by
   the system class loader, parent-first). Plugin AAR
   does NOT bundle the SDK. SDK version is fixed at
   build time of the host app; plugins declare
   `nativeSdkAbi` to detect mismatch. Confirm.

E. **Audit log: how do we attribute direct java.net.* calls
   that get OS-blocked**: we don't. The plugin gets a
   `SecurityException` and the OS rejection is invisible
   to our audit. The audit log only sees the audited
   capability path. Acceptable — direct calls failing
   means the plugin author wrote unportable code anyway.

F. **Tracking missing items 8 from Codex 132**: I'm
   addressing isolatedProcess (the big one), DEX-only,
   ABI versioning, `entry_class` in AIDL, dispatcher
   bounds, load timeout, threat-model carve-outs.
   Anything else?

1. **Verdict: YELLOW**

`isolatedProcess` is the right structural primitive, but v2 still has two security/design blockers to resolve before GREEN:

- `bundleFilePath` will not work as described if the DEX is staged in host private app storage. The isolated UID should not be able to read `/data/data/<host_pkg>/...`. Pass the DEX over Binder as a `ParcelFileDescriptor`, copy it inside the isolated process, or use `InMemoryDexClassLoader` from verified bytes.
- Sharing one `:nativePlugin` process across multiple untrusted native plugins is stronger than v1 against host compromise, but it is not per-plugin isolation. A malicious plugin can reflect over the shared process heap and potentially reach other plugins’ contexts, classloaders, callback objects, tokens, or dispatch state. That can become a cross-plugin capability bypass, not just an OOM risk.

For GREEN, I’d require either one isolated process per native plugin/trust domain, or an explicit V1.5 constraint that only one native plugin is loaded per isolated process at a time.

2. **`isolatedProcess`**

Yes, it is the right Android primitive for host-data and direct-permission isolation. The main missing nuance is that it also breaks ordinary file-path sharing from the host. Treat paths from the host as advisory metadata only; actual code bytes need an FD or in-memory load path.

Also, do not rely on classloader isolation inside the same isolated process for security. It is organizational isolation only.

3. **DEX-only**

DEX-only is acceptable for V1.5 and preferable to AAR if the goal is a narrow, auditable launch. But make the SDK/tooling story consistent: the Kotlin template should produce the accepted runtime artifact, not “AAR scaffold” unless the build task ultimately emits a sanitized DEX/JAR bundle.

I would not add AAR/resource/native-lib policy yet. Keep V1.5 pure code, no resources, no native libs, no manifest merging.

4. **Other Missing Items**

- Validate `entryClass` with a Java/Kotlin binary class-name regex, not exactly the plugin-id reverse-DNS regex. Class segments often start uppercase; nested classes may need an explicit decision on `$`.
- ABI mismatch should fail before binding/loading plugin code.
- `onInit` timeout should cancel the plugin scope/job, not merely report error.
- Bridge calls must be attributed by host-side connection/session identity, not by plugin-supplied `pluginId`.
- Document that direct blocked OS calls will not appear in the audit log; that part is fine.
- Consider a hard payload/result size limit on Binder bridge calls.

Net: the big v1 flaw is fixed. The remaining serious issue is shared-process cross-plugin trust.
