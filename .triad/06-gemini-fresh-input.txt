## Compressed context for fresh Gemini

You are Gemini in a 3-model triad with Claude (lead so far, Anthropic) and Codex (OpenAI). Voted by all to build a multi-device E2E-encrypted plugin-host platform.

### Locked product model
- Server is encrypted message bus + plugin marketplace (V1.5) + multi-device account hub. Android-only V1.
- Each sender writes its own plugin inheriting BasePlugin (JS, WebView), defines its own render/data/actions.
- E2E encryption everywhere. Server is blind to content, knows metadata.
- V1 distribution = sideload/URL install. Marketplace ships V1.5.
- Auth via signed messages (HMAC/JWT). WebView plugin runtime.
- Two SDKs: plugin SDK (JS, inherits BasePlugin) + server SDK (Python V1, JS/TS V1.1, curl docs day 1).
- Pairing via central broker (server brokers handshake; remains content-blind).
- Install-time per-plugin permission screen.
- Plugin capabilities available: network unrestricted, storage, camera, file picker, location, background exec. Each plugin declares what it uses.
- Code-based dynamic notifications: plugin JS runs at receive time, invokes `platform.showNotification(...)` bridge.
- V1 update model = prompt-to-update; V1.5 = auto-update with marketplace gate.

### Major pivot this round — multi-device
User said: `the binding is not just phone + key. we need some sort of user. if I have 2 phones I should login with my user and see the same data. data must be on server. key is per UUID user.`

Lead's architectural read: Bitwarden/ProtonMail zero-knowledge model:
- Signup: email + password. UUID assigned.
- Password derives encryption key client-side (Argon2id). Server never sees password or key.
- Server stores everything encrypted under user UUID.
- New device: email+password -> pull encrypted user blob -> decrypt locally.
- Lost devices but remember password = full recovery. Forgot password = data gone (V1.5 adds recovery seed).
- Server blind to content, sees metadata.
- Budget impact: +20-30h. V1 platform total ~100-135h.
