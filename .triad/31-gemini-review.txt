## Verify

1.  **Server CAS race: the atomic UPDATE WHERE version=expected returns no rows on conflict; we then re-read and raise. Is there any window where two concurrent PUTs could both think they succeeded?**
    -   **Verdict: Safe.** The implementation in `upsert_state_cas` is robust. It re-reads the state from the database after the `UPDATE` is committed and verifies that its change was the one that stuck. If another process won the race, `refreshed.state_version` will not equal `new_version`, and a `StateConflictError` is correctly raised. This prevents a split-brain scenario.

2.  **StateMerger.userScopedStorage remote-wins: appropriate V1 simplification, or does this lose data dangerously?**
    -   **Verdict: Dangerous data loss.** The "remote-wins" strategy for `userScopedStorage` will discard local changes if the remote state has also changed. For example, if two offline devices both modify different keys within the same plugin's storage, the second device to sync will lose its changes entirely for that plugin's scope. This is a significant data integrity issue, even for a V1. The planned V1.5 fix (per-key timestamps) is necessary.

3.  **EncryptedUserState.fromJson handles missing fields (.optJSONArray/.optJSONObject) — schema-version-0 (older blob) parses safely?**
    -   **Verdict: Safe.** The use of `optJSONArray` and `optJSONObject` combined with `.orEmpty()` default values in `EncryptedUserState.fromJson` ensures forward compatibility. An older state blob lacking newer fields will parse correctly into a model with empty lists/maps for the missing fields, preventing crashes and data corruption during schema evolution.

4.  **State endpoint returns base64-encoded blob; tests pass blob as `_b64(bytes)`. Confirm no encoding mismatch.**
    -   **Verdict: Correct.** The server-side code in `routers/state.py` consistently uses base64 encoding for `encrypted_blob` on both `GET` responses and in the `409 Conflict` body. The corresponding `decode_base64` on the `PUT` request ensures symmetry. The contract is clear and the test coverage reflects this.

5.  **PUT empty blob (length 0) — currently `minimum=1` in schema. Is 0-length a valid state? Probably no, since users always have at least the schema_version field, but worth confirming.**
    -   **Verdict: Correct.** The `minimum=1` constraint on the decoded `new_encrypted_blob` in `StatePutRequest` is a valid safeguard. The minimal valid `EncryptedUserState` serializes to a non-empty JSON string (e.g., `{"schema_version":1,...}`), which, when encrypted, will produce a non-empty blob. This prevents storing ambiguous or invalid empty states.

6.  **Server 409 returns the conflict body inline in `detail`. FastAPI's HTTPException stringifies dict detail. Is the client-side parsing path well-defined?**
    -   **Verdict: Well-defined.** FastAPI does *not* stringify a Pydantic model or dict passed to `detail` in an `HTTPException`; it serializes it to a proper JSON object within the error response. The response will be `{"detail": {"current_state_version": ...}}`. The presence of `StateConflictBodyDto` on the Android client shows there is a clear, type-safe path for parsing this JSON error body with Retrofit's `response.errorBody()`.
