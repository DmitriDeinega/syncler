**Pressure Tests**

`isUnlocked` propagation looks consistent: `stateIn(..., initialValue = session.sessionState.value.isUnlocked)` matches the current session value, and StateFlow replay means downstream collectors get the latest value. Warm start with only stored token is still locked because `masterKey == null`; after `authenticate`, true is visible.

`phase1MigrationDoneAt` merge semantics are green: once either side is non-null, merge never returns null; earlier timestamp wins deterministically.

`sawMissingPairing` unknown-spam concern: server send does require an active pairing before storing messages, so true unknown-sender spam should not land. New yellow: messages sent before a later pairing revoke can still be returned by inbox, while the client has tombstoned the pairing and filters it out, so cursor hold can stall until expiry.

First-launch timing is not green because migration can run immediately on `isUnlocked == true` before the initial `pull()` has loaded the current user’s remote state.

**Overall**

Do **not** green-light Phase 1 yet. Blocker is #2: migration still lacks a reliable current-user ownership check for legacy records. I would fix that before Phase 2.
